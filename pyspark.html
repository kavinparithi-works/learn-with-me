Variables & Data Types

Master Python's building blocks: how data is stored, typed, and transformed in memory.

🟢 Beginner ⏱ 20 min 🐍 Python
Step 1 - Concept Introduction

What Are Variables?

A variable is a named label that points to a value stored in memory. Python is dynamically typed - you do not declare types explicitly. Python infers the type from the value you assign.

Core data types: int (whole numbers), float (decimals), str (text), bool (True/False), None (absence of value).

Step 2 - Why It Exists

Why Variables Matter in Data Engineering

Every PySpark job uses variables to hold DataFrame references, configuration values, schema definitions, and transformation results. Understanding Python's type system prevents silent bugs where integers are treated as strings or floats lose precision during CSV ingestion.

Think of computer memory as a giant whiteboard with labeled boxes. When you write age = 25, Python draws a box labeled age and writes 25 inside it. Assigning age = 26 erases the old value and writes the new one - the label stays, the contents change.

Step 7 - Common Mistakes

Mistake - Mutable Default Argument
Bug
def add_item(item, lst=[]):
    lst.append(item)
    return lst

print(add_item(1))  # [1]
print(add_item(2))  # [1, 2] NOT [2]!

The default list is created once and reused across all calls.

Fix
Fixed
def add_item(item, lst=None):
    if lst is None:
        lst = []
    lst.append(item)
    return lst

Use None as default, create a fresh list inside the function body.

Mistake - Variable Shadowing
Bug
list = [1, 2, 3]  # shadows built-in!
print(list("abc"))  # TypeError!

Naming a variable list, str, id, or type overwrites the built-in.

Fix
Fixed
numbers = [1, 2, 3]  # descriptive name
print(list("abc"))   # ['a','b','c']

Always use descriptive names that do not clash with Python built-ins.

Step 9 - Interview Questions

What is the difference between == and is in Python?▼
== checks value equality. is checks object identity (same memory address). For small integers Python caches objects, so a = 256; b = 256; a is b returns True, but with larger numbers it may return False. Always use is only when checking for None.
What does type() return for None?▼
type(None) returns <class 'NoneType'>. None is a singleton - there is exactly one None object in any Python process. The correct way to check for None is x is None, not x == None.
Why are mutable default arguments a common Python gotcha?▼
Default argument values are evaluated once at function definition time, not at each call. A mutable default like [] is shared across all calls. The fix is to use None as the default and create the mutable object inside the function body.

Step 10 - Knowledge Check

Quiz
What does int("3.14") return in Python?
int() cannot convert a string with a decimal point directly. You must first convert to float: int(float("3.14")) gives 3.

Conditions

Control program flow using if/elif/else, comparison operators, logical operators, and ternary expressions.

🟢 Beginner ⏱ 18 min 🐍 Python
Step 1 - Concept Introduction

Conditional Logic

Conditions allow a program to make decisions. Python uses if, elif (else-if), and else to branch execution. The condition must evaluate to a truthy or falsy value.

Python's falsy values: False, 0, 0.0, "", [], {}, set(), None. Everything else is truthy.

Step 2 - Why It Exists

Why Conditions Are Essential

In data pipelines, conditions decide routing: which transformation to apply, whether to raise an error, whether to skip a record, or which environment config to load. Every PySpark job is wrapped in conditional environment checks.

Think of a railway signal box. When a train arrives, the operator checks the destination and flips switches to route it down the correct track. Python conditions work the same way: based on the value of data, execution follows a different path.

Step 9 - Interview Questions

What values are falsy in Python?▼
Falsy values: False, 0, 0.0, 0j, "", b"", [], (), {}, set(), None. Everything else is truthy. Empty containers are falsy, enabling if my_list: style guards.
How does Python's ternary expression differ from Java's?▼
Java uses condition ? true_val : false_val. Python uses true_val if condition else false_val. Python's syntax reads like natural English, but the true branch appears first which surprises developers coming from Java or C.
What is short-circuit evaluation and why does it matter?▼
In A and B, Python skips B if A is falsy. In A or B, it skips B if A is truthy. This is critical for safety: if conn is not None and conn.is_active(): avoids calling is_active() on None. It also enables the idiom value = input_val or default_val.

An empty list is falsy, so or evaluates and returns the right operand. The result is "default". This short-circuit behavior is widely used for providing fallback values.

Loops

Iterate over data with for loops, while loops, enumerate, zip, and list comprehensions.

🟢 Beginner ⏱ 22 min 🐍 Python
Step 1 - Concept Introduction

Iteration in Python

Loops let you repeat a block of code. Python has two loop types: for (iterate over a sequence) and while (repeat while a condition is true). Python also supports list comprehensions for concise single-line transformations.

Key built-ins: range() generates number sequences, enumerate() adds an index, zip() pairs two iterables together.

Step 2 - Why It Exists

Why Loops Power Data Pipelines

Data engineering constantly iterates: over file lists to load each one, over schema fields to build validation checks, over partition keys to trigger downstream jobs. Understanding which loop construct is idiomatic Python will make your code readable to your teammates.

Imagine a conveyor belt in a factory. Each item comes down the belt and a worker performs the same operation on it before passing it along. A Python for loop is the conveyor belt: each element in the collection passes through your code block exactly once.

Step 7 - Common Mistakes

Mistake - Modifying List While Iterating
Bug
items = [1, 2, 3, 4, 5]
for item in items:
    if item % 2 == 0:
        items.remove(item)  # Bug! skips items
print(items)  # [1, 3, 5] but skip occurs

Modifying a list while iterating over it causes items to be skipped.

Fix - Iterate Over a Copy
Fixed
items = [1, 2, 3, 4, 5]
# Use list comprehension instead
items = [x for x in items if x % 2 != 0]
print(items)  # [1, 3, 5]

Use a list comprehension to filter into a new list instead of mutating in-place.

Mistake - Accidental Infinite Loop
Bug
i = 0
while i < 10:
    print(i)
    # forgot: i += 1 - runs forever!

Forgetting to increment the counter in a while loop causes an infinite loop.

Fix
Fixed
i = 0
while i < 10:
    print(i)
    i += 1  # always increment!

Prefer for loops when you know the iteration count. Use while only when the termination condition is dynamic.

Step 9 - Interview Questions

What is the difference between break and continue?▼
break immediately exits the entire loop. continue skips the rest of the current iteration and moves to the next one. Both affect only the innermost loop when nested loops are involved. break is used for early exit; continue for skipping specific conditions.
When would you use a while loop instead of a for loop?▼
Use while when the number of iterations is not known in advance, for example: polling an API until a job is done, reading lines from a file until EOF, or retrying an operation until it succeeds. Use for when you have a fixed collection or range to iterate.
How does a list comprehension differ from a for loop in performance?▼
List comprehensions are typically 10-30% faster than equivalent for loops because they are optimized at the bytecode level. However, for very complex logic or multi-statement bodies, a regular for loop is more readable and should be preferred over unreadable one-liners.

Functions

Define reusable logic with def, master *args/**kwargs, closures, lambdas, and Python's LEGB scope rules.

🟡 Intermediate ⏱ 25 min 🐍 Python
Step 1 - Concept Introduction

Functions and Scope

A function is a named, reusable block of code. Python functions are defined with def, can accept parameters (including default values, *args, and **kwargs), and optionally return a value.

Scope rules (LEGB): Python resolves names in this order: Local, Enclosing, Global, Built-in. This matters when debugging why a variable inside a function has an unexpected value.

Step 2 - Why It Exists

Why Functions Are the Core of Clean Pipelines

Every transformation in a production PySpark pipeline is wrapped in a Python function. Functions let you name your logic, test it in isolation, and reuse it across multiple jobs. Without functions, data pipelines become monolithic scripts that are impossible to maintain.

A function is like a vending machine. You put in specific inputs (coins + selection code), and you reliably get the same output every time. The internal mechanism is hidden - you do not need to know how it works, just what inputs to provide and what output to expect.

Step 7 - Common Mistakes

Mistake - Not Returning a Value
Bug
def double(x):
    result = x * 2
    # forgot return!

y = double(5)
print(y)  # None, not 10

A function without a return statement returns None implicitly.

Fix
Fixed
def double(x):
    result = x * 2
    return result  # explicit return

print(y)  # 10

Mistake - Scope Confusion with Global
Bug

def increment():
    counter += 1  # UnboundLocalError!
increment()

Python treats counter as local once you assign to it, but it was not locally defined first.

Fix - Pass as Parameter Instead
Fixed
def increment(counter):
    return counter + 1  # pure function

counter = 0
counter = increment(counter)  # 1

Prefer passing variables as parameters rather than relying on global state.

Step 9 - Interview Questions

Explain Python's LEGB scope rule.▼
Python resolves variable names in this order: Local (inside current function), Enclosing (enclosing function for nested functions/closures), Global (module level), Built-in (Python built-ins like len, print). The first match wins.
What is the difference between *args and **kwargs?▼
*args collects extra positional arguments into a tuple. **kwargs collects extra keyword arguments into a dict. Both allow writing flexible functions that accept a variable number of inputs. Common use: def wrapper(*args, **kwargs): return original(*args, **kwargs).
What is a closure and when would you use one?▼
A closure is an inner function that captures variables from its enclosing scope. Use closures to create factory functions that produce customized functions. Example: make_validator(rules) returns a validator function pre-loaded with specific validation rules, avoiding repeated parameter passing.

Python's Four Collection Types

list: ordered, mutable, allows duplicates - [1, 2, 3]

tuple: ordered, immutable, allows duplicates - (1, 2, 3)

set: unordered, mutable, NO duplicates - {1, 2, 3}

Step 2 - Why It Exists

Why Collections Drive Data Engineering

Data engineering is fundamentally about transforming collections of data. Lists hold column names, tuples hold schema field definitions, sets perform deduplication, and dicts hold configuration mappings. Every PySpark schema is built from nested Python collections.

Imagine four types of storage containers: a list is a filing cabinet - ordered, items can be added/removed. A tuple is a sealed envelope - contents fixed forever. A set is a bag of unique stamps - no duplicates. A dict is a labeled binder - each section has a name and contents.

Step 7 - Common Mistakes

Mistake - Mutating a Tuple (immutable)
Bug
coords = (10, 20)
coords[0] = 99  # TypeError!

Tuples are immutable. Attempting to modify them raises TypeError.

Fix - Use a List or Create New Tuple
Fixed
coords = [10, 20]   # use list if mutable
coords[0] = 99    # works fine: [99, 20]

Use lists when you need mutability; use tuples for fixed records like schema field definitions.

Mistake - Using a List for Lookups
Slow
bad_ids = [1, 2, 3, ..., 10000]  # O(n) lookup
if some_id in bad_ids:  # scans entire list
    ...

List membership testing is O(n) - for large collections this is very slow.

Fix - Use a Set for Membership Tests
Fast

Distributed Computing

Understand clusters, master and worker nodes, horizontal scaling, and data partitioning across machines.

🟢 Beginner ⏱ 20 min 🌩️ Spark
Step 1 - Concept Introduction

How Distributed Computing Works

A cluster is a group of machines (nodes) that work together as one system. There is one master node (also called the Driver in Spark) that coordinates the work, and multiple worker nodes (Executors) that do the actual computation.

Data partitioning is the key insight: instead of one machine processing 1TB, you split (partition) it into 200 chunks of 5GB and each of 40 worker nodes processes 5 chunks in parallel. Total time decreases by up to 40x.

Step 2 - Why It Exists

Horizontal vs Vertical Scaling

Vertical scaling: buy a bigger single machine (more RAM, more CPUs). This hits physical limits quickly and is expensive. Doubling RAM may cost 10x more.

Horizontal scaling: add more commodity machines to the cluster. Cheap, flexible, and effectively limitless. Spark is designed from the ground up for horizontal scaling.

Think of a restaurant kitchen. A single chef can cook 20 meals per hour. When 500 guests arrive, you do not replace the chef with a superhuman - you hire 50 chefs who all cook in parallel. The master chef (Driver) coordinates who cooks what, and every worker chef (Executor) processes their assigned orders simultaneously.

The Spark Execution Model

Driver: Your Python/Scala program. It translates user code into a DAG execution plan and coordinates the cluster.

DAG Execution & Lazy Evaluation

Understand how Spark builds a Directed Acyclic Graph, what lazy evaluation means, and how Catalyst optimizes your queries.

🟡 Intermediate ⏱ 24 min 🌩️ Spark
Step 1 - Concept Introduction

What is a DAG?

A Directed Acyclic Graph (DAG) is a graph where edges have a direction and there are no circular paths. In Spark, each transformation adds a node to the DAG. The graph flows forward from source data to final output - never backwards (acyclic).

Lazy evaluation: When you call filter() or select(), Spark does NOT execute immediately. It records the transformation in the DAG. Only when you call an action (show, count, write) does Spark evaluate the entire DAG at once. This lets Spark optimize the full plan before execution.

Step 2 - Why It Exists

The Power of Lazy Evaluation

Lazy evaluation enables query optimization. Spark's Catalyst optimizer can look at your entire chain of transformations and reorder, merge, or eliminate steps for better performance. For example, it pushes filters as early as possible to reduce data volume before expensive joins.

Without lazy evaluation, each transformation would execute immediately and Spark could not optimize the full pipeline. Eager execution is fine for debugging but inefficient for production workloads.

Lazy evaluation is like a restaurant waiter who takes your entire meal order before going to the kitchen. They collect: appetizer, main course, dessert, and specific modifications. Only then does the kitchen start cooking - and the chef can optimize: bake the bread while grilling the steak instead of doing them one by one. If you ordered the appetizer and the kitchen started immediately, the chef could not optimize.

Step 9 - Interview Questions

What is lazy evaluation and why does Spark use it?▼
Lazy evaluation means transformations are not executed when called - they are recorded as a plan. Spark uses it because it allows the Catalyst optimizer to see the entire query before executing, enabling optimizations like predicate pushdown, constant folding, and stage merging that would be impossible if each transformation executed eagerly.
What is predicate pushdown and why is it important?▼
Predicate pushdown moves filter conditions as early in the DAG as possible - ideally all the way down to the data source (Parquet, Delta Lake). If you filter by year=2024 after reading 10 years of data, predicate pushdown pushes that filter to the file scan level, so only 2024 files are read. This can reduce I/O by 10x or more.
What is the difference between the logical plan and physical plan?▼
The logical plan represents what you want to compute (user's intent, unresolved, then analyzed). The physical plan represents how Spark will actually execute it, choosing specific algorithms (hash join vs sort-merge join), deciding which partitions to scan, and optimizing operator order. explain() shows both.

Step 1 - Concept Introduction

What is a Spark DataFrame?

A Spark DataFrame is a distributed collection of data organized into named columns. Unlike pandas, it does not load data into memory on one machine - it represents a distributed dataset that spans across all executors in the cluster.

The entry point to PySpark is the SparkSession. All DataFrame operations, SQL queries, and data reads go through the SparkSession object.

Step 2 - Why It Exists

DataFrames added named columns, SQL-like operations, schema enforcement, and crucially: the Catalyst optimizer. DataFrames are 10-100x faster than equivalent RDD operations because Catalyst generates optimized JVM bytecode.

A Spark DataFrame is like a giant spreadsheet stored across 100 computers, with each computer holding some rows. You write operations against the whole spreadsheet, and Spark figures out how to distribute those operations across all 100 computers simultaneously - without you having to think about which rows are on which machine.

What is the difference between a DataFrame and an RDD?▼
An RDD is a low-level, untyped collection of objects with no schema. A DataFrame has named columns, schema enforcement, and SQL-like operations. ⚠️
Transformations Are LAZY
Calling select(), filter(), withColumn() does NOT execute anything. It just adds nodes to the DAG. Execution only happens when you call an action (show, count, write). This is fundamental to how Spark works.
Step 1 - Concept Introduction

Core Transformation Methods

select() chooses which columns to keep. filter()/where() reduces rows. withColumn() adds or replaces a column. withColumnRenamed() renames. drop() removes columns. cast() changes type. alias() renames in expressions.

Understanding each method's semantics prevents common bugs like accidentally modifying an existing column or performing a filter that returns no rows.

Think of transformations as assembly line stations. Raw material (source data) enters at one end. Each station (transformation) modifies the product in some way - trim it, shape it, label it. Each station hands off to the next. No work happens until the entire line is running (an action is called).

What is the difference between filter() and where()?▼
They are identical. where() is an alias for filter() provided for SQL familiarity. Both accept either a Column expression (col("age") > 18) or a SQL string ("age > 18"). There is no performance difference. Use whichever reads more naturally in context.
How does withColumn behave if the column already exists?▼
If the column name already exists in the DataFrame, withColumn() replaces it with the new expression. This is useful for in-place updates like type casting: df.withColumn("age", col("age").cast("int")). Be careful: this can unintentionally overwrite columns if you mistype a column name.
Why must Spark column conditions use & and | instead of and/or?▼
Python's and/or operators perform short-circuit evaluation on Python objects. A Spark Column object is always truthy in Python, so col("a") == 1 and col("b") == 2 returns just col("b") == 2. The & and | operators are overloaded on the Column class to produce the correct Spark AND/OR column expressions.

Step 1 - Concept Introduction

Join Types in PySpark

The join() method combines two DataFrames on a key. The how parameter specifies the join type.

inner: Only rows with matching keys in BOTH DataFrames. left: All rows from left, matched rows from right (null if no match). right: All rows from right. full (outer): All rows from both. anti: Rows from left that have NO match in right. semi: Rows from left that HAVE a match in right (but no right columns). cross: Every combination of rows.

Step 2 - Why It Exists

Joins are the Core of Data Engineering

Data engineering is mostly about joining tables: orders to customers, events to sessions, transactions to products. Understanding when to use each join type and how to avoid performance pitfalls (shuffles, skew) is what separates junior from senior data engineers.

Think of an INNER JOIN as a guest list check: only guests whose name appears on BOTH the reservation system AND the check-in list get in. A LEFT JOIN is like keeping all reservations but noting which ones showed up. An ANTI JOIN finds no-shows: reservations with no matching check-in.

What is a broadcast join and when should you use it?▼
A broadcast join ships a small DataFrame to all executor nodes, so each executor can join locally without shuffling the large DataFrame over the network. Use it when one side is small (typically under 10MB, configurable). It eliminates the shuffle stage, often giving 10-100x speedup. Spark auto-broadcasts if the table is smaller than spark.sql.autoBroadcastJoinThreshold.
What is the difference between left_semi and inner join?▼
Both return only rows that have a match. The difference is the output columns: an inner join returns columns from BOTH tables. A left_semi join returns only columns from the LEFT table. Semi joins are useful for filtering without polluting the result with right-side columns and without risking row multiplication from one-to-many relationships.
How does a sort-merge join work in Spark?▼
Sort-merge join: (1) shuffle both DataFrames so rows with the same key end up on the same executor, (2) sort each partition by the join key, (3) merge-scan through both sorted lists simultaneously. It is the default join strategy for large-large joins. It requires a shuffle (expensive) but is memory-efficient. Broadcast join avoids the shuffle entirely for small tables.

Window Functions

Compute rankings, running totals, lag/lead comparisons, and moving averages with Window partitioning.

🔴 Advanced ⏱ 35 min 🌩️ PySpark
Step 1 - Concept Introduction

What Are Window Functions?

A window function computes a value for each row based on a "window" of related rows - without collapsing the result into a single summary row like groupBy does.

You define the window with Window.partitionBy() (groups rows like groupBy) and Window.orderBy() (determines row order within the group). Then you apply functions like row_number(), rank(), lag(), sum().over().

Step 2 - Why It Exists

What Window Functions Enable

Without window functions, many analytical queries require complex self-joins or subqueries. Window functions make these patterns concise: rank employees by salary within each department, calculate running totals, find the previous day's value, assign sequential IDs within a group.

Imagine you have a spreadsheet of sales data by day. A window function is like a sliding calculator that moves over your rows. At each row, it looks at the surrounding rows (the window) and computes something - "what was yesterday's value?", "what rank is this row among its peers?", "what is the cumulative total up to this point?" - then writes the result back to that row without collapsing the dataset.

df = df.withColumn("next_amount", lead("amount", 1).over(win_time))

    Window.unboundedPreceding, Window.currentRow)
df = df.withColumn("running_total", sum("amount").over(win_cumul))

# Moving average (7-day window)
win_7d = Window.partitionBy("product").orderBy("date").rowsBetween(-6, 0)

Step 9 - Interview Questions

What is the difference between rank() and dense_rank()?▼
Both assign ranks, but they differ when there are ties. rank() gives tied rows the same rank and skips the next rank (1, 1, 3, 4). dense_rank() gives tied rows the same rank but does not skip (1, 1, 2, 3). row_number() assigns unique sequential numbers regardless of ties (1, 2, 3, 4).
How would you deduplicate records keeping only the most recent version?▼
Use row_number() with a window partitioned by the record ID and ordered by updated_at descending. Then filter for row_number == 1 to keep only the latest version of each record. This is the standard deduplication pattern in SCD Type 1 and delta lake merge scenarios.
What does rowsBetween vs rangeBetween mean in a window?▼
rowsBetween(start, end) specifies the frame in terms of physical row offsets (-1 = one row before, 0 = current row). rangeBetween(start, end) specifies frame in terms of value range relative to the current value (useful for date-based windows). For most cases, rowsBetween is more predictable.

Step 1 - Concept Introduction

Spark's Complex Data Types

ArrayType: A column that holds a list of values (like tags: ["Python", "Spark", "SQL"]). Functions: explode(), array_contains(), array_size(), flatten().

Think of a Spark row as a Russian nesting doll (matryoshka). The outer doll is the main row. Open it and you find a struct column with sub-fields. Open that and you find an array of items. Spark's complex type functions let you reach inside these nested structures without unwrapping everything manually.

Partitions

Control parallelism with repartition and coalesce, understand optimal partition sizing, and write partitioned files.

🔴 Advanced ⏱ 28 min ⚙️ Optimization
Step 1 - Concept Introduction

What Are Partitions?

A partition is a chunk of your DataFrame stored on one executor. Spark processes partitions in parallel - one task per partition per available core. More partitions = more parallelism. Too many small partitions = too much scheduling overhead. Too few large partitions = some executors sit idle.

Optimal partition size: around 128MB. This matches HDFS/S3 block size and gives executors enough work without excessive memory pressure. With 1TB of data, target ~8,000 partitions of 128MB each.

Step 2 - Why It Exists

Why Partition Tuning Matters

The default shuffle partition count is 200 (spark.sql.shuffle.partitions). For small datasets, 200 is too many (most partitions empty). For large datasets (TBs), 200 is too few (each partition too large). Tuning this single setting often gives the biggest performance improvement in a Spark job.

Imagine dividing a 1,000-page book among workers. If you give each worker one page, you have 1,000 workers but each finishes in seconds with no meaningful work. If you give one worker all 1,000 pages, only one worker works while the rest are idle. The sweet spot is about 10-20 pages per worker - enough work to be efficient, but still parallelized.

Module 5.2 - Optimization 🔀

Shuffles

Understand what triggers shuffles, their true cost in I/O and network, and wide vs narrow transformations.

🔴 Advanced ⏱ 26 min ⚙️ Optimization
🚨
Shuffles Are the #1 Performance Killer
A shuffle requires ALL executors to write their data to disk, then ALL executors to read data over the network from every other executor. For 100GB of data, this can be 200GB of disk writes plus 200GB of network reads - causing minutes of extra latency.
Step 1 - Concept Introduction

What Triggers a Shuffle?

A shuffle redistributes data across partitions so rows with the same key end up on the same executor. It is triggered by: groupBy(), join() (non-broadcast), repartition(), distinct(), sort(), cogroup().

Narrow transformations (no shuffle): filter(), select(), withColumn(), map(), coalesce(). Each input partition produces exactly one output partition with no data movement between executors.

Wide transformations (cause shuffle): each output partition depends on data from multiple input partitions.

Step 2 - Why It Exists

Why Shuffles Are Unavoidable (But Minimizable)

Some operations inherently require grouping data: you cannot compute "total sales per region" without bringing all "US" rows to the same place. Shuffles are necessary but expensive. The goal is to minimize shuffle data volume by: filtering early, projecting only needed columns, using broadcast joins for small tables.

Imagine a post office with 100 workers sorting mail. A narrow transformation is like each worker sorting their own pile independently. A shuffle is like re-sorting all mail by destination city: every worker must package up all their mail, and then each worker receives packages from all 100 other workers. The re-sorting process requires massive coordination and movement.

What operations trigger a shuffle in Spark?▼
Shuffles are triggered by: groupBy/agg, join (non-broadcast), repartition, distinct, sort/orderBy, cogroup, intersection, subtract. You can verify by looking for "Exchange" in the explain() output. Each Exchange node represents one shuffle stage boundary.
What is shuffle spill and how do you address it?▼
Shuffle spill occurs when a shuffle stage's data exceeds executor memory. Spark serializes the excess data to disk (spill to disk). This is much slower than in-memory operations. To address: increase executor memory, reduce partition size (more partitions means less data per partition), filter data before the shuffle-triggering operation.
How does bucketing help reduce shuffle overhead?▼
Bucketing pre-partitions data by a column when writing (like hash partitioning on disk). When two bucketed tables with the same bucket count and key are joined, Spark skips the shuffle entirely - the data is already co-located. This is a write-once, read-many optimization for frequently joined large tables.

filter() is a narrow transformation - it processes each partition independently without moving data between executors. groupBy().agg() and distinct() are wide transformations that require data from multiple partitions to be co-located on the same executor.

Caching & Persistence

Use cache() and persist() to eliminate redundant recomputation, choose the right StorageLevel, and manage memory.

🔴 Advanced ⏱ 24 min ⚙️ Optimization
Step 1 - Concept Introduction

Caching and Persistence

cache() stores a DataFrame in memory after its first computation. Subsequent actions read from the cached copy instead of recomputing from source. It is equivalent to persist(StorageLevel.MEMORY_AND_DISK).

Storage Levels: MEMORY_ONLY - fast, but drops partitions when memory is full. MEMORY_AND_DISK - spills to disk if memory full (default for cache()). DISK_ONLY - always on disk, slowest but never OOM. MEMORY_ONLY_SER - serialized (smaller footprint, slower to read).

Step 2 - Why It Exists

When to Cache

Cache when a DataFrame is: (1) used in multiple actions, (2) the result of expensive transformations (complex joins, aggregations), and (3) fits in cluster memory. Do NOT cache when: the DataFrame is used only once, it does not fit in memory (would cause eviction), or the source is already fast (Parquet with predicate pushdown).

Caching is like making a photocopy of a document you need to reference repeatedly. Without caching, each time you need the document you drive to the archive (recompute from source). With caching, you keep a copy on your desk. The first time still requires the drive, but every subsequent use takes seconds instead of hours.

Step 9 - Interview Questions

What is the difference between cache() and persist()?▼
cache() is shorthand for persist(StorageLevel.MEMORY_AND_DISK). persist() lets you specify the storage level explicitly. Both are lazy - they register the DataFrame for caching but do not actually store data until an action runs. Use persist() when you need control over memory vs disk tradeoffs.
When should you NOT cache a DataFrame?▼
Do not cache when: (1) the DataFrame is used only once - caching adds overhead with no benefit, (2) the source data is already fast to read (e.g., Parquet with aggressive predicate pushdown may be faster than cache), (3) the DataFrame is too large for executor memory - it will be evicted immediately, (4) you are at the end of a pipeline and writing results to storage.
What is the difference between cache() and checkpoint()?▼
cache() stores data in executor memory/disk but retains the full lineage graph. checkpoint() saves data to HDFS/S3 and truncates the lineage graph. Use checkpoint() for very long transformation chains (50+ steps) where the lineage itself becomes expensive to track and can cause StackOverflow errors during re-planning.

Catalyst Optimizer

Understand how Spark turns your Python code into optimized execution plans through four transformation stages.

🔴 Advanced ⏱ 26 min ⚙️ Optimization
Step 1 - Concept Introduction

The Four-Stage Catalyst Pipeline

Stage 1 - Unresolved Logical Plan: Your Python code is parsed into an AST. Column names are not yet validated.

Stage 2 - Analyzed Logical Plan: Column names and types are resolved against the catalog. Errors detected here.

Stage 3 - Optimized Logical Plan: Rules applied: predicate pushdown, constant folding, column pruning, null propagation.

Stage 4 - Physical Plan: Cost-based optimizer selects algorithms (broadcast vs sort-merge join). Tungsten code generation produces JVM bytecode.

Step 2 - Why It Exists

How Catalyst Makes PySpark Competitive with Scala

Python is slow. But Catalyst converts your Python DataFrame code into optimized JVM bytecode that runs at near-Java speed. This is why PySpark DataFrames are often faster than hand-written Scala code using RDDs - the optimizer does what a human would not bother to do manually.

Think of Catalyst as a professional editor reviewing your writing. You hand in a draft (logical plan). The editor removes redundant sentences (column pruning), moves the thesis earlier (predicate pushdown), and rewrites complex passages in clearer language (constant folding). The final published version (physical plan) is far more efficient than your first draft, even though it conveys the same meaning.

Step 9 - Interview Questions

What is predicate pushdown and how does Catalyst achieve it?▼
Predicate pushdown moves filter conditions to the earliest possible point - ideally into the data source scan. For Parquet files, Spark pushes filters into the file reader itself, so only matching row groups are read from disk. Catalyst detects filter conditions in the logical plan and rewrites the physical plan to execute them at scan time, dramatically reducing I/O.
Why are Python UDFs slow and what are the alternatives?▼
Python UDFs (registered with udf()) require row-by-row serialization: each row is serialized from JVM to Python, processed in Python, then serialized back to JVM. This has two costs: Python's slower execution and per-row serialization overhead. Alternatives: (1) built-in spark functions (best), (2) Pandas UDFs / vectorized UDFs via @pandas_udf (batch processing, much faster), (3) Scala UDFs (JVM native, no serialization).
What is AQE and what three problems does it solve?▼
Adaptive Query Execution (Spark 3.0+) optimizes at runtime using actual statistics instead of estimates. It solves: (1) Too many small partitions after a shuffle: AQE coalesces them into larger partitions. (2) Join strategy choice: if one side shrinks after filtering, AQE switches to broadcast join. (3) Data skew in joins: AQE detects and splits skewed partitions into smaller chunks.

Step 10 - Knowledge Check

Quiz
What does an "Exchange" node in explain() output indicate?
An Exchange node in the physical plan represents a shuffle - data being redistributed across partitions/executors. Minimizing Exchange nodes is the key to reducing shuffle cost. BroadcastExchange is a broadcast join (good - no full shuffle). Regular Exchange is a sort-merge or hash shuffle (expensive).
