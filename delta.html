5.1 Partitions

A partition = one chunk of data = one task for one executor. Understanding partitions is the foundation of Spark performance tuning.

Default Partitions

spark.sql.shuffle.partitions defaults to 200 after any shuffle operation.

Optimal Size

Target 128 MB per partition. Too small = overhead; too large = OOM risk.

repartition()

Full shuffle. Can increase or decrease partitions. Even distribution guaranteed.

coalesce()

No shuffle. Can only decrease partitions. Merges on same executor.

Python — Partitions
# Check current partitions
print(df.rdd.getNumPartitions())  # e.g., 200

# repartition - full shuffle, evenly distributes
df_repartitioned = df.repartition(8)  # increase for better parallelism
df_by_dept = df.repartition(4, "department")  # partition by column

# coalesce - no shuffle, just merge partitions
df_coalesced = df.coalesce(1)  # reduce to 1 file for output (use carefully!)

# Rule of thumb
# - After filter: repartition (fewer rows, too many partitions)
# - Before write: coalesce (avoid many small files)
# - After join/groupBy: already repartitioned, check count

# Set default shuffle partitions (tune per job size)
spark.conf.set("spark.sql.shuffle.partitions", "50")  # for small datasets
# Default 200 good for 100GB+, reduce for smaller jobs

5.2 Shuffles

Shuffle = data moving across the network between executors. The most expensive Spark operation — minimize it wherever possible.

Shuffle Triggers

groupBy, orderBy, distinct, join (non-broadcast), repartition all cause shuffles.

Broadcast Join

For small tables (<10MB default). Sends small table to all executors, no shuffle needed.

Pre-Partitioning

Repartition both DataFrames by join key before joining to avoid the join shuffle.

Bucketing

Pre-sort and bucket on disk by key. Eliminates shuffle for repeated joins on same key.

Python — Reduce Shuffle Cost
# What triggers a shuffle
df.groupBy("department").count()    # SHUFFLE
df.orderBy("salary")                # SHUFFLE
df.distinct()                       # SHUFFLE
df.join(other_df, "id")             # SHUFFLE (unless broadcast)

# Reducing shuffle cost

# 1. Broadcast small tables (< 10MB default, configurable)
from pyspark.sql.functions import broadcast
df.join(broadcast(small_df), "dept_id")

# 2. Pre-partition data before join
df1 = df1.repartition("customer_id")
df2 = df2.repartition("customer_id")
df1.join(df2, "customer_id")  # no shuffle needed

# 3. Increase broadcast threshold
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "50m")  # 50MB

# 4. Use bucketing for repeated joins on same key
df.write.bucketBy(32, "customer_id").sortBy("customer_id").saveAsTable("bucketed_customers")

5.3 Caching

Cache keeps a DataFrame in memory (or memory+disk) so repeated use avoids expensive recomputation from scratch.

MEMORY_ONLY

Fastest access. Stored as JVM objects. If OOM, partitions are recomputed, not spilled.

MEMORY_AND_DISK

Default for cache(). Spills to disk if memory full. Reliable but slower for disk portions.

DISK_ONLY

Always writes to disk. Slowest but always fits. Good for data larger than memory.

MEMORY_AND_DISK_2

Replicated across 2 nodes. Fault-tolerant — no recompute on executor failure.

Python — Caching Strategies
from pyspark import StorageLevel

# cache() = shortcut for MEMORY_AND_DISK
df_employees = spark.read.parquet("/data/employees/")
df_employees.cache()

# First action triggers caching
df_employees.count()  # data loaded + cached

# Subsequent actions use cache
df_employees.filter(col("dept") == "Engineering").show()   # fast
df_employees.groupBy("dept").count().show()                # fast

# persist() with explicit storage level
df.persist(StorageLevel.MEMORY_ONLY)         # fastest, may spill if OOM
df.persist(StorageLevel.MEMORY_AND_DISK)     # cache() default
df.persist(StorageLevel.DISK_ONLY)           # slowest, but always fits
df.persist(StorageLevel.MEMORY_AND_DISK_2)  # replicated (fault tolerant)

# When NOT to cache
# - Single use (cache overhead not worth it)
# - Larger than available memory (causes eviction churn)
# - Infrequent access

# Always unpersist when done to free memory
df_employees.unpersist()

5.4 Catalyst Optimizer

Catalyst is Spark's automatic query optimizer. It transforms your logical plan into an efficient physical execution plan through 4 phases — without you doing anything.

Animation: Plan Transformation Pipeline
Phase 1
Parsed Logical Plan
Raw SQL / DataFrame API resolved to unvalidated tree
→
Phase 2
Analyzed Logical Plan
Column names resolved, schema validated, types checked
→
Phase 3
Optimized Logical Plan
Predicate pushdown, column pruning, constant folding applied
→
Phase 4
Physical Plan + Code Gen
Optimal join strategy selected, Tungsten JVM bytecode generated

Predicate Pushdown

Filters moved as close to data source as possible. Parquet skips entire row groups.

Column Pruning

Only reads columns actually used in the query. Parquet's columnar format makes this free.

Constant Folding

Expressions like 1+1 evaluated once at compile time, not per row at runtime.

Tungsten Engine

Generates optimized JVM bytecode. Operates on binary data directly, avoids serialization.

Python — Inspect Execution Plans
# See the execution plan
df.filter(col("salary") > 80000).select("name", "salary").explain()

# Verbose plan with all 4 stages
df.filter(col("salary") > 80000).select("name", "salary").explain(True)

# Predicate pushdown - Catalyst pushes filter to data source
# You write:
spark.read.parquet("/data/employees/") \
     .filter(col("salary") > 80000)    # filter AFTER read

# Catalyst rewrites as:
# Read parquet WITH filter applied at file level (skip entire row groups)
# Result: reads far less data from disk

# Column pruning - Catalyst only reads needed columns from Parquet
df.select("name", "salary")  # Parquet only reads 2 columns, not all 20

6.1 Delta Lake Fundamentals

Delta Lake adds ACID transactions, schema enforcement, and time travel to Parquet files on your data lake — no separate database engine required.

💡 Without Delta: Parquet has no transactions, no schema enforcement, no time travel, and no concurrent write protection. Delta's _delta_log/ transaction log solves all of these.

Atomicity

Writes fully succeed or fully fail. No partial writes that corrupt the table.

Consistency

Schema enforced on every write. Constraints respected. Table always valid.

Isolation

Concurrent writes use optimistic concurrency — don't corrupt each other.

Durability

Committed data persists through failures via the transaction log on durable storage.

Python — Delta Lake Basics
# Write as Delta
df.write.format("delta").mode("overwrite").save("/delta/employees/")

# Read Delta
df = spark.read.format("delta").load("/delta/employees/")

# Delta table history
spark.sql("DESCRIBE HISTORY delta.`/delta/employees/`").show()

# Time travel - query previous version
df_v1 = spark.read.format("delta") \
              .option("versionAsOf", 1) \
              .load("/delta/employees/")

# Time travel by timestamp
df_yesterday = spark.read.format("delta") \
                    .option("timestampAsOf", "2024-01-02") \
                    .load("/delta/employees/")

# Schema enforcement (Delta rejects bad data)
# This will FAIL if extra column not in schema:
bad_df.write.format("delta").mode("append").save("/delta/employees/")
# DeltaAnalysisException: A schema mismatch detected

# Schema evolution (opt-in to allow new columns)
new_df.write.format("delta").mode("append") \
      .option("mergeSchema", "true") \
      .save("/delta/employees/")

6.2 MERGE / Upsert

MERGE is the most important Delta operation. Update if a record exists, insert if it doesn't, delete if flagged — all atomically in one pass.

whenMatchedUpdate

Record exists in both source and target. Update target fields from source values.

whenNotMatchedInsert

Record in source but NOT in target. Insert as a new row into the Delta table.

whenMatchedDelete

Record exists in both but marked for deletion. Removes row from target table.

Use Cases

Incremental loads, CDC processing, SCD Type 1, daily fact table updates.

Python — MERGE (PySpark + SQL)
# MERGE in PySpark (Delta)
from delta.tables import DeltaTable

target = DeltaTable.forPath(spark, "/delta/employees/")
source_df = spark.read.parquet("/staging/employees_updates/")

target.alias("target").merge(
    source_df.alias("source"),
    "target.id = source.id"
).whenMatchedUpdate(set={
    "name":       "source.name",
    "salary":     "source.salary",
    "department": "source.department",
    "updated_at": "source.updated_at"
}).whenNotMatchedInsert(values={
    "id":         "source.id",
    "name":       "source.name",
    "salary":     "source.salary",
    "department": "source.department",
    "created_at": "source.created_at",
    "updated_at": "source.updated_at"
}).whenMatchedDelete(
    condition="source.is_deleted = true"
).execute()

# SQL MERGE syntax
spark.sql("""
    MERGE INTO employees AS target
    USING employees_updates AS source
    ON target.id = source.id
    WHEN MATCHED THEN
        UPDATE SET target.salary = source.salary, target.department = source.department
    WHEN NOT MATCHED THEN
        INSERT (id, name, salary, department) VALUES (source.id, source.name, source.salary, source.department)
    WHEN MATCHED AND source.is_deleted = true THEN
        DELETE
""")

6.3 Delta Optimization

OPTIMIZE, VACUUM, and Z-ORDER keep Delta tables fast and storage-efficient. The small files problem is the #1 performance killer for streaming/incremental pipelines.

⚠️ Small Files Problem: Every small append creates a new Parquet file. After 1000 daily loads, you have 1000 files for a simple scan. Each file requires a separate I/O operation and metadata read. OPTIMIZE fixes this.

OPTIMIZE

Compacts small Parquet files into larger ones (default 1GB target). Improves scan speed dramatically.

Z-ORDER

Co-locates related data within files using a space-filling curve. Enable file skipping on filter columns.

VACUUM

Deletes old Parquet files no longer in the transaction log. Default 7-day retention for time travel.

Auto Optimize

Databricks feature: optimizeWrite and autoCompact automatically manage file sizes on write.

Python — OPTIMIZE, Z-ORDER, VACUUM
# OPTIMIZE: compacts small files into larger ones
spark.sql("OPTIMIZE delta.`/delta/employees/`")

# Z-ORDER: co-locate related data within files for faster filter queries
# Best for high-cardinality filter columns
spark.sql("OPTIMIZE delta.`/delta/employees/` ZORDER BY (department, hire_date)")
# Now queries filtering by department read far fewer files

# VACUUM: remove old data files no longer referenced by transaction log
# Default retention = 7 days (for time travel)
spark.sql("VACUUM delta.`/delta/employees/`")                    # 7 day default
spark.sql("VACUUM delta.`/delta/employees/` RETAIN 168 HOURS") # explicit 7 days

# WARNING: reducing retention too low breaks time travel
# spark.sql("VACUUM delta.`/delta/employees/` RETAIN 0 HOURS")  # DANGEROUS

# Auto Optimize in Databricks (set on table creation or ALTER)
spark.sql("""
    ALTER TABLE employees
    SET TBLPROPERTIES (
        'delta.autoOptimize.optimizeWrite' = 'true',
        'delta.autoOptimize.autoCompact' = 'true'
    )
""")

Interview Questions

12 real interview questions covering Spark optimization and Delta Lake — from entry-level to senior engineer difficulty.

What is the difference between repartition() and coalesce()?
Easy▼
repartition(n) performs a full shuffle and can increase or decrease partition count with even distribution. coalesce(n) avoids shuffle by merging partitions on the same executor — only decreases count and may be uneven. Use coalesce when reducing partitions (e.g., before write), use repartition when increasing or when you need even distribution for joins.
What operations cause shuffles in Spark?
Easy▼
groupBy, orderBy, distinct, join (non-broadcast), repartition. Shuffles move data across the network between executors — the most expensive operation in Spark because it involves serialization, network transfer, and deserialization. filter, select, withColumn, and coalesce do NOT cause shuffles.
What is the difference between cache() and persist()?
Medium▼
cache() is shorthand for persist(StorageLevel.MEMORY_AND_DISK). persist() lets you specify the storage level explicitly: MEMORY_ONLY (fastest, recomputes on OOM), MEMORY_AND_DISK (default), DISK_ONLY (always fits, slowest), MEMORY_AND_DISK_2 (replicated for fault tolerance). Use persist() when you need fine control over the memory vs. disk tradeoff.
Explain Catalyst Optimizer's 4 phases.
Medium▼
1) Analysis — resolves column names, validates schema, checks types. 2) Logical Optimization — applies rule-based optimizations: predicate pushdown, column pruning, constant folding. 3) Physical Planning — generates multiple physical plans, chooses the best based on cost model (e.g., decides broadcast vs. sort-merge join). 4) Code Generation — Tungsten engine generates optimized JVM bytecode, operates on binary data to minimize serialization overhead.
What are Delta Lake's ACID properties?
Medium▼
Atomicity — writes fully succeed or fail, no partial writes that corrupt the table. Consistency — schema enforced on every write, constraints respected, table always in valid state. Isolation — concurrent writers use optimistic concurrency control via the transaction log, don't corrupt each other. Durability — committed data persists through failures because the transaction log is stored on durable object storage.
How do you query a previous version of a Delta table?
Easy▼
Time travel using versionAsOf or timestampAsOf option. Example:
spark.read.format("delta").option("versionAsOf", 1).load(path)
or
spark.read.format("delta").option("timestampAsOf", "2024-01-02").load(path)
Only works if the underlying data files haven't been removed by VACUUM.
When would you use MERGE vs simple overwrite?
Hard▼
Use MERGE for incremental loads, CDC processing, SCD Type 1 (update existing records), or when you need to handle inserts + updates + deletes atomically in one pass. Use overwrite for full refreshes, initial loads, or when the source represents the complete current dataset. MERGE is slower (requires scanning target to match) but preserves existing records not present in the source batch.
What is the difference between OPTIMIZE and VACUUM?
Medium▼
OPTIMIZE compacts many small Parquet files into fewer larger files (default 1GB target), improving read performance by reducing I/O overhead. It does NOT delete any data. VACUUM removes old Parquet files that are no longer referenced by the transaction log, freeing actual storage. VACUUM has a retention period (default 7 days) to preserve files needed for time travel. Run OPTIMIZE for performance, VACUUM for storage costs.
Explain Z-ORDER and when to use it.
Hard▼
Z-ORDER co-locates related data within Parquet files using a space-filling curve algorithm. When you filter on Z-ORDERed columns, Delta's data-skipping statistics let it skip entire files that don't match. Best for high-cardinality columns frequently used in WHERE clauses (department, date, customer_id). Run as: OPTIMIZE table ZORDER BY (col1, col2). Not beneficial for low-cardinality columns (e.g., boolean, status with 3 values) — partitioning is better for those.
What causes the small files problem and how do you fix it?
Medium▼
Cause: Frequent small appends (streaming micro-batches, hourly CDC loads) each create new small Parquet files. After 1000 loads, a simple scan reads 1000 files — each requiring separate I/O + metadata overhead. Fix: Run OPTIMIZE regularly (daily or post-load), enable delta.autoOptimize.autoCompact = true in Databricks, use coalesce before writes, batch small writes into larger ones, or enable delta.autoOptimize.optimizeWrite for automatic bin-packing.
How do you configure the broadcast join threshold?
Easy▼
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "50m") sets the threshold to 50MB. Default is 10MB. Tables smaller than the threshold are automatically broadcast to all executors, eliminating shuffle. You can also force a broadcast explicitly: df.join(broadcast(small_df), "key"). Set to -1 to disable auto-broadcast entirely.
How do you choose a partitioning strategy for a large Delta table?
Hard▼
Partition by low-cardinality, frequently filtered columns (date, region, department). Avoid high-cardinality columns (id, timestamp) — creates too many small partition directories. Target 1GB+ per partition directory for efficient reads. Use Z-ORDER within partitions for secondary filter columns (e.g., partition by date, Z-ORDER by customer_id for fast customer lookups within a date range). For time-series data: partition by year/month, not full date if data is sparse per day.
