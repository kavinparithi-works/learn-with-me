What is the difference between WHERE and HAVING?
Easy
v
WHERE filters rows before aggregation. HAVING filters groups after aggregation.

WHERE filters individual rows and cannot reference aggregate functions like COUNT() or SUM().
HAVING filters the result of GROUP BY and can reference aggregates.
-- WHERE: filter before grouping
SELECT dept, COUNT(*) as headcount
FROM employees
WHERE salary > 50000          -- filters rows first
GROUP BY dept
HAVING COUNT(*) > 5;          -- filters groups after
2
Explain all SQL JOIN types with examples.
Easy
v
INNER JOIN: Only matching rows from both tables. Most common.
LEFT JOIN: All rows from left table + matching from right. Non-matching right side gets NULL.
RIGHT JOIN: All rows from right table + matching from left. Rarely used - LEFT JOIN is preferred by convention.
FULL OUTER JOIN: All rows from both tables. NULL where no match on either side.
CROSS JOIN: Every row from left combined with every row from right (Cartesian product). Use with care.
SELF JOIN: Table joined to itself. Common for hierarchy/manager queries.
-- Manager hierarchy using SELF JOIN
SELECT e.name as employee, m.name as manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

What are window functions? Explain ROW_NUMBER, RANK, LAG with use cases.
Medium
v
Window functions perform calculations across related rows without collapsing them. Unlike GROUP BY, you keep all original rows.

ROW_NUMBER(): Unique sequential number per partition. Use for deduplication (keep latest row per ID).
RANK(): Rank with gaps (1,2,2,4). Use for "top N" leaderboards.
LAG/LEAD: Access previous/next row. Use for month-over-month growth calculations.
-- MoM growth using LAG
SELECT month, revenue,
       LAG(revenue, 1) OVER (ORDER BY month) AS prev_revenue,
       ROUND((revenue - LAG(revenue,1) OVER (ORDER BY month)) /
             LAG(revenue,1) OVER (ORDER BY month) * 100, 2) AS growth_pct
FROM monthly_sales;

How would you find duplicate records in a table?
Medium
v
Two approaches - GROUP BY counts or ROW_NUMBER deduplication: 
-- Find duplicate order_ids
SELECT order_id, COUNT(*) as cnt
FROM orders
GROUP BY order_id
HAVING COUNT(*) > 1;

-- Keep latest row per duplicate (dedup)
WITH ranked AS (
    SELECT *, ROW_NUMBER() OVER (
        PARTITION BY order_id ORDER BY updated_at DESC
    ) AS rn
    FROM orders
)
SELECT * FROM ranked WHERE rn = 1;
In PySpark: df.dropDuplicates(["order_id"]) or use Window + row_number()

WITH ranked AS (

What is a CTE and when would you use it over a subquery?
Medium
v
A CTE (Common Table Expression) is a named temporary result set defined with WITH. It improves readability and can be referenced multiple times in the query.

Use CTE when: The same subquery is referenced multiple times, the query has multiple logical steps, or you want to add recursion (hierarchies).
Use subquery when: The subquery is simple and only used once - a subquery is sometimes faster as the optimizer can inline it.
-- Recursive CTE for org hierarchy
WITH RECURSIVE org_tree AS (
    -- Base case: CEO
    SELECT id, name, manager_id, 0 AS level
    FROM employees WHERE manager_id IS NULL
    UNION ALL
    -- Recursive case
    SELECT e.id, e.name, e.manager_id, ot.level + 1
    FROM employees e
    JOIN org_tree ot ON e.manager_id = ot.id
)
SELECT *, REPEAT('  ', level) || name AS hierarchy
FROM org_tree ORDER BY level, name;

How do you optimize a slow SQL query? Walk through your approach.
Medium
v
1. Check EXPLAIN PLAN: Look for full table scans, missing indexes, expensive joins, and high row estimates.
2. Indexes: Add index on WHERE, JOIN, and ORDER BY columns. Composite index order matters - put highest cardinality column first.
3. Filter early: Apply WHERE filters before JOINs. Don't join then filter - filter then join.
4. Avoid functions on indexed columns: WHERE YEAR(date) = 2024 breaks the index. Use WHERE date BETWEEN '2024-01-01' AND '2024-12-31'.
5. Select only needed columns: SELECT * prevents index-only scans and increases I/O.
6. Partition pruning: If table is partitioned by date, always filter on the partition key.

Write a running total and a 3-month moving average using window functions.
Hard
v
SELECT
    month,
    revenue,
    -- Running total (cumulative sum)
    SUM(revenue) OVER (ORDER BY month
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total,
    -- 3-month moving average (current + 2 prior)
    AVG(revenue) OVER (ORDER BY month
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS moving_avg_3m,
    -- Year-to-date within each year
    SUM(revenue) OVER (PARTITION BY YEAR(month)
        ORDER BY month
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS ytd_revenue
FROM monthly_sales;

What is the difference between UNION and UNION ALL?
Easy
v
UNION: Combines results from two queries and removes duplicates. Requires a sort operation - slower.
UNION ALL: Combines results from two queries and keeps all rows including duplicates. No deduplication - faster.
Rule: Always use UNION ALL unless you specifically need deduplication. UNION ALL is faster because it avoids the extra sort/hash for dedup.
Both queries must have the same number of columns with compatible data types.

What is SCD (Slowly Changing Dimension) Type 2? How do you implement it?
Hard
v
SCD Type 2 preserves the full history of changes to a dimension record. Each change creates a new row with effective date ranges.

When a customer moves cities, SCD2 adds a new row for the new city with a start date, and closes the old row with an end date. You can then analyze orders using the customer's address at the time of purchase.
-- SCD Type 2 customer dimension
customer_key | customer_id | city    | valid_from | valid_to   | is_current
1            | C001        | NYC     | 2022-01-01 | 2023-06-30 | FALSE
2            | C001        | Boston  | 2023-07-01 | 9999-12-31 | TRUE

-- In Delta Lake (MERGE for SCD2)
MERGE INTO dim_customer AS target
USING staging AS source ON target.customer_id = source.customer_id
    AND target.is_current = TRUE
WHEN MATCHED AND target.city != source.city THEN UPDATE SET
    valid_to = source.effective_date, is_current = FALSE
WHEN NOT MATCHED THEN INSERT (customer_id, city, valid_from, is_current)
VALUES (source.customer_id, source.city, source.effective_date, TRUE);

What is the execution order of SQL clauses?
Medium
v
SQL is written in one order but executed in a different order:

Execution order: FROM - JOIN - WHERE - GROUP BY - HAVING - SELECT - DISTINCT - ORDER BY - LIMIT
This is why you can't use a SELECT alias in WHERE (SELECT runs after WHERE). But you CAN use a SELECT alias in ORDER BY (ORDER BY runs after SELECT).
Window functions execute after WHERE and GROUP BY but before ORDER BY. CTEs can be thought of as running before the main query.

Explain partitioning in SQL databases and when to use it.
Hard
v
Table partitioning divides a large table into smaller physical pieces (partitions) while still appearing as one logical table.

Range partitioning: Most common - partition by date. Queries filtering by date only scan the relevant partition(s) - partition pruning.
Hash partitioning: Distribute rows evenly by hash of a key - good for even distribution across partitions.
List partitioning: Explicit values per partition - e.g., partition by country.
When to use: Tables with 100M+ rows where you always filter on a column. Typically date for time-series data. Avoid over-partitioning - too many small partitions create overhead.

What is the difference between DELETE, TRUNCATE, and DROP?
Easy
v
DELETE: Removes specific rows (uses WHERE). Logged row-by-row. Can be rolled back. Does not reset identity/auto-increment. Slow on large tables.
TRUNCATE: Removes ALL rows instantly. Minimally logged (much faster). Cannot be filtered by WHERE. Resets identity counter. Can be rolled back in some DBs (PostgreSQL yes, MySQL no).
DROP: Removes the entire table structure AND data. Cannot be rolled back. The table no longer exists after DROP.
Interview rule: DELETE rows you want to keep audit trail for. TRUNCATE to empty a staging table. DROP to decommission a table.

What is a subquery vs a correlated subquery?
Medium
v
Subquery: Runs once, independently. The outer query uses its result. Fast because it executes only once.
Correlated subquery: References a column from the outer query. Runs once per row of the outer query. Can be very slow on large tables.
-- Correlated: finds employees earning above their dept average
SELECT name, salary, dept
FROM employees e
WHERE salary > (
    SELECT AVG(salary) FROM employees
    WHERE dept = e.dept  -- references outer query
);
Performance tip: Replace correlated subqueries with window functions: salary > AVG(salary) OVER (PARTITION BY dept) - runs in a single pass.

How would you calculate a 7-day rolling average in SQL?
Hard
v
SELECT
    sale_date,
    revenue,
    AVG(revenue) OVER (
        ORDER BY sale_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS rolling_7day_avg
FROM daily_sales;
ROWS BETWEEN 6 PRECEDING AND CURRENT ROW: The window is the current row plus the 6 rows before it - exactly 7 rows. ROWS is based on physical row position; RANGE is based on value distance.
Partial window at start: The first 6 rows will have averages over fewer than 7 days (day 1 is just 1 row, day 2 is 2 rows, etc.) unless you filter those out.

What is the difference between a primary key and a unique key?
Easy
v
Primary Key: Uniquely identifies each row. Cannot be NULL. Only one per table. Automatically creates a clustered index (in most DBs).
Unique Key: Enforces uniqueness but allows one NULL (in most DBs). Multiple unique keys per table are allowed. Creates a non-clustered index.
Common interview answer: Primary key = the row's identity. Unique key = an alternate business identifier (e.g., email can be unique but employee_id is the primary key).

What are indexes and what are the trade-offs of having too many?
Medium
v
What indexes do: Create a sorted copy of column(s) that the DB can binary-search instead of scanning all rows. Dramatically speeds up SELECT, WHERE, JOIN, and ORDER BY on indexed columns.
Cost of indexes: Every INSERT, UPDATE, DELETE must also update every index on the table. More indexes = slower writes. They also use disk space.
Rule of thumb: Index columns used in WHERE, JOIN ON, and ORDER BY. Don't index low-cardinality columns (e.g., boolean flags). Remove indexes not being used by any query.

Write a query to find the top 3 products by revenue in each category.
Hard
v
WITH ranked_products AS (
    SELECT
        category,
        product_name,
        revenue,
        DENSE_RANK() OVER (
            PARTITION BY category
            ORDER BY revenue DESC
        ) AS revenue_rank
    FROM product_sales
)
SELECT category, product_name, revenue, revenue_rank
FROM ranked_products
WHERE revenue_rank <= 3
ORDER BY category, revenue_rank;
Why DENSE_RANK not ROW_NUMBER: If two products tie for 2nd place, ROW_NUMBER would give one rank 2 and one rank 3 (arbitrary). DENSE_RANK gives both rank 2 and shows the next one as rank 3 - both correctly in top 3.

What is a materialized view and when would you use it?
Medium
v
Regular view: Virtual table - runs the query every time you SELECT from it. No stored data. Always fresh.
Materialized view: Stores the query result physically on disk. SELECT is instant. Must be refreshed to reflect source changes (manually or on schedule).
Use when: Complex aggregation query takes minutes to run and result is queried hundreds of times per day. Accept slightly stale data in exchange for sub-second response time.
In Delta Lake: Gold tables in Medallion architecture are essentially materialized views - pre-aggregated, refreshed by scheduled pipelines.

Explain OLTP vs OLAP and how they affect schema design.
Hard
v
OLTP (Online Transaction Processing): Many concurrent small reads/writes. Normalized schema (3NF) - minimize data duplication, maximize write performance. Examples: banking, e-commerce orders.
OLAP (Online Analytical Processing): Few complex queries scanning millions of rows. Denormalized schema (Star/Snowflake) - pre-join tables to maximize read performance. Examples: data warehouses, BI dashboards.
Star schema: One fact table (orders, events) surrounded by dimension tables (customers, products, dates). Optimized for aggregation queries.
Data Engineers bridge the two: Extract from OLTP (3NF), transform into OLAP-ready star schema, load into warehouse/lakehouse.

What is the difference between EXISTS and IN?
Medium
v
IN: Retrieves all matching values from the subquery and creates a list. The outer query checks if a value is in that list. Can be slow with large lists.
EXISTS: Stops as soon as it finds the first match (short-circuit evaluation). More efficient when the subquery returns many rows.
NULL behavior: IN returns no rows if the subquery contains a NULL (x IN (1, NULL) is unknown for x=2). EXISTS is not affected by NULLs.
General rule: Use EXISTS for large subqueries. Use IN for small fixed value lists. Modern query optimizers often convert one to the other anyway.

What are transactions and why are they important?
Medium
v
A transaction is a unit of work that either fully completes (COMMIT) or fully undoes (ROLLBACK). Guarantees the database moves from one valid state to another - never partial states.
Classic example: Bank transfer. Debit Account A, Credit Account B. Both must succeed or both must fail. Without a transaction, a crash between them leaves money missing.
ACID properties ensure: Atomicity (all or nothing), Consistency (valid state before and after), Isolation (concurrent transactions don't interfere), Durability (committed changes survive failures).

What is lazy evaluation in Spark and why is it important?
Easy
v
Lazy evaluation means Spark doesn't execute transformations immediately. It builds a DAG (plan) of all transformations, and only executes when an action is triggered.

Why it matters: Spark can see the entire plan before running, allowing optimizations like predicate pushdown (apply filters early), column pruning (only read needed columns), and stage fusion (combine multiple steps).
Transformations (lazy): filter(), select(), join(), groupBy(), withColumn()
Actions (trigger execution): count(), show(), collect(), write(), first(), take()
Interview tip: Calling count() multiple times reruns the computation. Cache/persist the DataFrame if you need to reuse it.

What is a shuffle in Spark? How do you minimize it?
Medium
v
A shuffle is a data redistribution operation across the network - Spark moves rows from one partition to another to co-locate related data.

What causes shuffles: groupBy(), join() (non-broadcast), distinct(), repartition(), orderBy()
Why shuffles are expensive: Network I/O, disk spill if data doesn't fit in memory, sorting overhead
Minimize by: Using broadcast joins for small tables (avoids shuffle entirely), filtering data before joining (fewer rows to shuffle), increasing spark.sql.shuffle.partitions for large shuffles, caching shared DataFrames to avoid recomputing them
Detect: Spark UI Exchange nodes in DAG indicate shuffle boundaries. High shuffle read/write bytes in Stage metrics.
3
Explain data skew in Spark and how to fix it.
Hard
v
Data skew happens when one partition has significantly more data than others - one task takes 10x longer than all others, wasting cluster resources.

Common cause: A key value is extremely frequent (NULL, "Unknown", a specific customer ID with millions of orders). All rows with that key go to one partition.
Fix 1 - Salting: Add a random suffix to the skewed key before groupBy, then remove it after. Spreads the load across multiple partitions.
Fix 2 - Broadcast join: If joining a large + small table and the join key is skewed, broadcast the small table instead of shuffle-joining.
Fix 3 - Skew hint (Databricks AQE): Adaptive Query Execution (Spark 3+) automatically splits skewed partitions. Enable with spark.sql.adaptive.enabled=true
-- Salting to fix groupBy skew
import pyspark.sql.functions as F
df = df.withColumn("salt", (F.rand() * 10).cast("int"))
df_grouped = df.groupBy("skewed_key", "salt").agg(F.sum("amount"))
df_final = df_grouped.groupBy("skewed_key").agg(F.sum("amount"))

When would you use cache() vs persist()? What storage levels are available?
Medium
v
cache() is shorthand for persist(StorageLevel.MEMORY_AND_DISK). Stores DataFrame in memory, spills to disk if not enough memory.
persist() lets you choose the storage level explicitly for fine-grained control.
MEMORY_ONLY: Fastest reads - recomputes if evicted. Use for small DataFrames that fit easily.
MEMORY_AND_DISK: (default cache) Spills to disk if evicted from memory. Safe choice for medium DataFrames.
DISK_ONLY: Slowest but doesn't use executor memory. Use for checkpointing long lineages.
When to cache: Use when a DataFrame is reused multiple times in the same application (multiple actions on the same DF). Always unpersist() when done.

What is the Catalyst Optimizer? How does it optimize queries?
Hard
v
Catalyst is Spark's query optimizer that converts your PySpark or SQL code into an optimized physical execution plan through 4 phases:

Phase 1 - Analysis: Resolve column names, check types, validate the logical plan.
Phase 2 - Logical Optimization: Apply rules like predicate pushdown (move filters as early as possible), column pruning (drop columns not used downstream), constant folding (evaluate 1+1 to 2 at planning time).
Phase 3 - Physical Planning: Select join strategies (broadcast vs sort-merge), decide on partition strategies, choose scan types.
Phase 4 - Code Generation: Generate JVM bytecode for the physical plan. Whole-stage code generation (WSCG) fuses multiple operators into a single tight loop - much faster than row-by-row processing.
Key insight: This is why DataFrame/SQL API is faster than RDD API - Catalyst can't optimize arbitrary Python lambda functions in RDDs.

How do you handle schema evolution in PySpark pipelines?
Medium
v
Schema evolution - the source adds/removes columns - is a common production problem. Several strategies:

Explicit schema definition: Define StructType schema and fail loudly on unexpected changes. Safest for production - no surprises.
mergeSchema (Delta Lake): .option("mergeSchema", "true") on write - new columns are added, existing columns keep their type. Cannot handle type changes.
Auto Loader with schema inference: Databricks Auto Loader tracks schema and rescues unknown columns into a _rescued_data JSON column instead of failing.
SELECT * approach: Select only the columns you need by name. New columns in source are silently ignored. Most forgiving.
df.write.option("mergeSchema", "true").format("delta").save(path)

What is AQE (Adaptive Query Execution) in Spark 3+?
Hard
v
AQE lets Spark re-optimize the physical plan at runtime based on actual statistics collected during execution - instead of relying only on estimates made at planning time.

Automatic partition coalescing: After a shuffle, if partitions are small, AQE merges them to reduce overhead. Solves "too many small tasks" without manual tuning.
Automatic broadcast join conversion: If a join side turns out to be small after a filter, AQE converts sort-merge join to broadcast join at runtime.
Automatic skew handling: Detects skewed partitions and splits them. The partition with 1M rows becomes multiple smaller partitions.
spark.conf.set("spark.sql.adaptive.enabled", "true")  # default in Spark 3.2+
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

What is the difference between repartition() and coalesce()?
Medium
v
repartition(n): Full shuffle - redistributes all data evenly across n partitions. Expensive but produces evenly sized partitions. Use to increase OR decrease partitions.
coalesce(n): No full shuffle - merges partitions locally. Can only decrease partition count. Much cheaper than repartition. Partitions may be uneven if sizes differ.
When to use repartition: Before a groupBy/join to avoid skew; when writing to partitioned storage (one file per partition); when increasing parallelism.
When to use coalesce: Before writing a small result to fewer output files. df.coalesce(1).write.csv(...) produces a single CSV file.

How do you write a custom UDF in PySpark and what are the performance implications?
Hard
v
from pyspark.sql.functions import udf
from pyspark.sql.types import StringType

# Regular Python UDF (slow)
def clean_phone(num):
    return ''.join(filter(str.isdigit, str(num))) if num else None

clean_phone_udf = udf(clean_phone, StringType())
df = df.withColumn("clean_phone", clean_phone_udf("phone_raw"))

# Pandas UDF (vectorized - much faster)
from pyspark.sql.functions import pandas_udf
import pandas as pd

@pandas_udf(StringType())
def clean_phone_fast(nums: pd.Series) -> pd.Series:
    return nums.apply(lambda x: ''.join(filter(str.isdigit, str(x))) if x else None)
Why UDFs are slow: Regular Python UDFs serialize each row from JVM to Python, process, and serialize back. Each row crosses the JVM-Python boundary.
Pandas UDF (arrow-based): 10-100x faster. Transfers data in columnar Apache Arrow batches instead of row-by-row. Use whenever possible.
Best practice: Always try to replace UDFs with built-in Spark functions first. If that's not possible, use Pandas UDF not regular UDF.

What is the difference between withColumn and select for transformations?
Medium
v
withColumn(name, expr): Keeps ALL existing columns, adds/replaces one column. Clean when adding derived columns to an existing DataFrame.
select(*cols): Returns ONLY the columns you specify. Use to reshape a DataFrame (project specific columns), rename multiple columns at once, or eliminate unwanted columns in one operation.
Performance note: Chaining many withColumn() calls creates a deep plan. For 50+ column transformations, prefer select() with a list comprehension - generates a flatter, more optimizable plan.
from pyspark.sql import functions as F
# Efficient: select with transformations
df_transformed = df.select(
    F.col("id"), F.col("name").alias("full_name"),
    (F.col("salary") * 1.1).alias("new_salary"),
    F.upper("dept").alias("dept")
)
11
How do you handle out-of-memory errors in a Spark job?
Hard
v
1. Identify which memory: Driver OOM vs Executor OOM are different problems. Driver OOM often from collect(). Executor OOM from large shuffles or caching too much.
2. Driver OOM: Replace collect() with write(). If you need results on driver, use take(n) or limit(n).toPandas() for small samples.
3. Executor OOM from shuffle: Increase spark.sql.shuffle.partitions (default 200, often too low). More partitions = smaller partition size = less memory per task.
4. Executor OOM from skew: One partition is huge. Use salting or AQE's skew join handling (spark.sql.adaptive.skewJoin.enabled=true).
5. Increase memory: spark.executor.memory, spark.driver.memory. Also increase spark.memory.fraction for more heap for execution.
6. Reduce data early: Apply filters and select only needed columns before joins/aggregations. Less data = less memory needed.

What is the difference between DataFrame.write.save() and saveAsTable()?
Medium
v
write.save(path): Writes files to a storage path (ADLS/S3/HDFS). No metadata registered. Only accessible by directly reading the path. Good for intermediate pipeline outputs.
saveAsTable("catalog.schema.table"): Writes files AND registers metadata in the catalog (Unity Catalog / Hive Metastore). Table becomes queryable by name from any notebook or SQL query. Good for final output tables that analysts/BI tools will query.
In Databricks: saveAsTable with Delta format creates a managed table (Unity Catalog controls the lifecycle). DROP TABLE deletes both metadata and files.

Explain Structured Streaming checkpointing and why it's critical for exactly-once processing.
Hard
v
What checkpointing stores: The offset (position) in the source stream that has been processed, plus the current state for stateful operations (e.g., running counts).
Exactly-once guarantee: If the streaming job crashes and restarts, it reads the checkpoint to find where it left off and resumes from exactly that point - no data skipped, no data reprocessed.
Without checkpointing: At-most-once (may miss data) or at-least-once (may reprocess and duplicate). Exactly-once requires both checkpointing AND an idempotent sink (e.g., Delta Lake MERGE or deduplication).
df.writeStream
  .format("delta")
  .option("checkpointLocation", "/checkpoints/my_stream")  # required!
  .outputMode("append")
  .start("/delta/output")

How do you read and write Parquet files efficiently in PySpark?
Medium
v
Why Parquet: Columnar format - Spark only reads the columns you SELECT, skipping the rest. Built-in compression. Schema embedded in file. 3-5x smaller than CSV. The standard format for data lakes.
# Read with predicate pushdown (Spark pushes filter to file level)
df = spark.read.parquet("/data/sales/")
df.filter(F.col("year") == 2024)  # only reads 2024 partitions

# Write with partitioning + compression
df.write \
  .mode("overwrite") \
  .partitionBy("year", "month") \
  .option("compression", "snappy") \
  .parquet("/output/sales")
Partition pruning: When you filter on the partition column, Spark skips entire directories it doesn't need to read - zero I/O for irrelevant data.

How would you debug a Spark job that's failing silently (no obvious error)?
Hard
v
Step 1 - Check Spark UI: Failed stages show red. Click the failed stage to see the exception on the executor. Executor logs often have more detail than driver logs.

Explain the concept of lineage in Spark and how it provides fault tolerance.
Hard
v
Lineage (RDD lineage graph): Spark remembers every transformation applied to create a DataFrame/RDD - the full chain from source data to current state. This is the DAG.
How fault tolerance works: If an executor crashes and a partition is lost, Spark doesn't need a data replica. It simply re-executes the lineage (the recorded transformations) on the source data to recompute that partition on another executor.
Wide transformations break lineage for checkpointing: Long lineage chains (100+ transformations) become expensive to recompute. Use df.checkpoint() to materialize data to disk and cut the lineage chain.
This is why caching helps more than just performance: A cached partition doesn't need to be recomputed from source if it's still in memory - it serves as a lineage shortcut.

How does Spark handle null values in joins and aggregations?
Medium
v
Joins on null keys: NULL != NULL in SQL/Spark. A row with a null join key will NEVER match any other row - including another null. It gets dropped in INNER JOIN, kept in LEFT JOIN with null values from right side.
Aggregations: COUNT(*) counts all rows including nulls. COUNT(column) counts only non-null values. SUM/AVG ignore nulls. If all values are null, AVG returns null.
NULL-safe equality: Use eqNullSafe() for joins where you want nulls to match nulls: df1.join(df2, df1.key.eqNullSafe(df2.key))
Best practice: Decide on null semantics explicitly. fillna() or coalesce() before joining to replace nulls with a sentinel value if nulls should join together.

Option 3 - Separate catalogs (Unity Catalog): Each tenant gets their own Unity Catalog namespace. Schema-level isolation. Shared infrastructure (cluster pools), isolated data. Good balance for up to hundreds of tenants.
Key requirements: Audit log of all data access. Deletion of all tenant data on offboarding (GDPR right to erasure). Tenant-specific retention policies. Encrypted at rest with per-tenant keys (BYOK).

How would you migrate a 10TB on-premise Oracle data warehouse to Azure Databricks?
Hard
v
Phase 1 - Assessment: Inventory tables (sizes, update frequency, Oracle-specific features). Identify T-SQL/Oracle-specific syntax to rewrite. Map Oracle objects to Delta Lake equivalents.
Phase 2 - Initial load: Use Azure Database Migration Service or ADF Copy Activity for initial bulk transfer. Throttle to not impact production Oracle. Transfer during low-traffic window.
Phase 3 - Parallel run: Both Oracle and Databricks active. Validate row counts, key metrics match. Fix discrepancies. Run new reports against both and compare output.
Phase 4 - Cutover: Switch pipelines to write to Databricks only. Freeze Oracle as read-only archive. Update BI tools to point to new Databricks Gold tables. Monitor closely for 2 weeks.
Phase 5 - Decommission: Archive Oracle data to cold storage. Cancel Oracle licensing. Document what was migrated and any schema changes made.

Explain how you would implement a data lineage tracking system.
Hard
v
What lineage tracks: For every piece of data: where did it come from (sources), what transformations were applied, where did it go (targets). Column-level lineage shows which source column contributed to each output column.
Unity Catalog automatic lineage: Tracks lineage automatically for all SQL queries and Spark operations in Databricks. No additional code needed. View in the Data Explorer UI.
Custom lineage metadata: In each pipeline, log input_table, output_table, pipeline_name, run_id, row_counts, and timestamp to a central lineage table. Build a graph from this metadata.
Why it matters: Impact analysis (if I change this source column, which 23 downstream tables break?). Root cause analysis (this KPI is wrong - trace back to which ETL step introduced the error). Compliance (prove this financial report's data came from the approved source system).
10
What is data mesh and when is it the right architectural choice?
Hard
v
Data mesh distributes data ownership to domain teams instead of a central data engineering team. Each domain (orders, payments, customers) owns and publishes their data as products.
4 principles: Domain ownership (orders team owns orders data), Data as a product (SLA-backed, documented, discoverable), Self-serve platform (teams can build without central DE), Federated governance (global policies applied centrally).
When it's right: Large organization (>500 engineers), multiple domains with different data needs, central data team is a bottleneck, domains have dedicated engineering capacity.
When it's wrong: Small company, single domain, teams don't have engineering capacity to own their data, no platform team to build self-serve infrastructure.
Reality: Data mesh is an organizational pattern more than a technical one. The technology (Unity Catalog, Delta Sharing) enables it but doesn't replace the cultural change required.
11
How do you handle timezone complexities in a global data pipeline?
Medium
v
Golden rule: Store all timestamps in UTC in the data lake. Convert to local timezone only at display/reporting layer.
Common bugs: Source system stores local time without timezone offset. DST (Daylight Saving Time) creates duplicate or missing hours. Joining data from two sources in different timezones without conversion.
PySpark handling: F.to_utc_timestamp(col, "America/New_York") converts local time to UTC. F.from_utc_timestamp(col, timezone) converts UTC to local for display. Store timezone as a separate column.
Partition by UTC date: Always partition by UTC date. If you partition by local date and the source timezone changes (summer/winter), your partition boundaries shift unpredictably.
12
How would you build a cost monitoring system for a data engineering platform?
Hard
v
Databricks system tables: system.billing.usage table contains per-cluster, per-job DBU consumption. system.compute.clusters shows cluster uptime. Join with Azure pricing to get dollar amounts.
Tagging strategy: Tag every Databricks cluster with team, project, environment, and pipeline_name. All usage is attributable to a cost center. Required for chargeback models.
Anomaly detection: Alert when a job costs 3x its historical average. Alert when total daily spend exceeds budget threshold. Alert on clusters with no auto-termination.
Optimization triggers: Auto-report top 10 most expensive jobs weekly. Identify idle clusters (running but no active notebooks). Find jobs running with larger clusters than needed (consistently low CPU/memory utilization).
13
Design a real-time fraud detection data pipeline.
Hard
v
Latency requirement: Fraud must be detected before the transaction is approved - typically within 200-500ms. This dictates the architecture: cannot use batch processing.
Real-time scoring path: Transaction API - feature lookup (Redis/Cosmos DB for millisecond lookups of precomputed user features) - ML model inference (MLflow serving endpoint) - rule engine - approve/decline decision.
Data engineering pipeline: Transaction events stream to Kafka - Spark Structured Streaming computes 1h/24h/30d rolling features (avg transaction amount, countries used, device fingerprints) - features written to low-latency feature store (Redis) - ML model trained nightly on historical labeled transactions.
Feedback loop: Declined transactions reviewed by fraud analysts. Labels flow back as training data. Model retrained daily with fresh labels. Feature drift monitored - alert when feature distributions shift significantly.
14
How would you handle PII data and GDPR compliance in a data lake?
Hard
v
Data classification: Tag all PII columns at ingest time. Unity Catalog column tags identify PII fields. Data catalog shows which tables contain PII.
Access control: Column masking on PII fields (email, phone, national ID). Only authorized roles see raw values. Analysts see masked values by default.
Right to erasure (GDPR Art. 17): Maintain a customer deletion request table. Daily job deletes/overwrites PII for deleted users. Delta Lake VACUUM purges physical files after retention period. Log every deletion for audit.
Data minimization: Don't collect PII you don't need. At Bronze, store raw. By Silver, replace PII with pseudonymous IDs (hash of customer_id). Gold tables have no raw PII - only aggregates and pseudonyms.
Encryption: At-rest encryption (ADLS default). In-transit TLS. Customer-Managed Keys (CMK) for highest sensitivity - customer holds the encryption key, can revoke at any time.
15
Explain the CAP theorem and how it applies to choosing data storage systems.
Hard
v
CAP theorem: A distributed system can guarantee at most 2 of: Consistency (all nodes see same data at same time), Availability (every request gets a response), Partition tolerance (system works despite network partition).
CP systems (Consistency + Partition tolerance): May be unavailable during partition. Examples: HBase, MongoDB with strong consistency, traditional relational DBs with synchronous replication. Choose for: financial transactions, inventory counts where accuracy is critical.
AP systems (Availability + Partition tolerance): May return stale data during partition. Examples: Cassandra, DynamoDB in eventual consistency mode. Choose for: user sessions, shopping carts, analytics where eventual consistency is acceptable.
Data lake perspective: Delta Lake on object storage is CP (strong consistency via transaction log). Event streaming (Kafka) is AP (messages available even during broker failures, with possible duplicate delivery).

Scenario-Based Questions
10 questions - Think-out-loud problems
1
Your Spark job ran fine yesterday but is 3x slower today. How do you investigate?
Hard
v
Step 1 - Check if data volume changed: 3x more input data = 3x longer is expected. Check row counts. Did a new data source go live? Is it a month-end dump?
Step 2 - Check Spark UI: Compare Stage metrics from today vs yesterday run. Look for a specific stage that got slower. High shuffle read bytes = shuffle increase. One task taking much longer = data skew.
Step 3 - Check data skew: Is one partition now much larger? Could be a new key value with many rows, or NULLs becoming frequent in a join key.
Step 4 - Check cluster health: Was the cluster resized? Is another job running simultaneously consuming resources? Check executor memory spill to disk (indicates memory pressure).
Step 5 - Check upstream changes: Did the source schema change? Were indexes dropped? Did a source query plan change?
Step 6 - Check for GC issues: High GC time in Spark UI indicates memory pressure - objects being collected frequently. Increase executor memory or reduce partitions size.
2
A downstream analyst reports that the numbers in your pipeline don't match their direct database query. How do you resolve?
Hard
v
Step 1 - Define the discrepancy precisely: How much difference? 5 rows or 5 million? Which date range? Which dimensions? What's the metric?
Step 2 - Align on the source of truth: Which source should win? Raw transactional DB? The pipeline? Depends on the business definition.
Step 3 - Reconcile at each stage: Check Bronze row count vs source. Silver vs Bronze. Gold vs Silver. The discrepancy appears at a specific transition - that's your bug.
Common causes: Filter applied incorrectly (filtering out cancelled orders when analyst includes them), timezone difference (pipeline UTC vs analyst local time), deduplication logic differs, JOIN type difference (inner vs outer), null handling difference.
Step 4 - Document the definition: After resolving, write down the agreed business definition. "Revenue = sum of completed orders, excluding refunds, in UTC+5." Prevents recurrence.
3
You need to load a 500GB CSV file into a Delta table. What's your approach?
Medium
v
Step 1 - Understand the file: One giant CSV or many files? Header present? Known schema or need inference? Encoding (UTF-8)? Delimiter (comma, pipe, tab)?
Step 2 - Define schema explicitly: Never infer schema on 500GB - expensive and can get types wrong. Define StructType schema upfront.
Step 3 - Read with parallelism: A single 500GB file can't be split easily (no natural split points except newlines). If possible, split the source into multiple files first. Spark reads multiple files in parallel.
Step 4 - Validate a sample first: Read 10,000 rows, validate types, check nulls, look for encoding issues. Fix problems before processing 500GB.
Step 5 - Write with partitioning: Partition by a date or category column. Provides query performance and allows partial rewrites later.
df = spark.read.schema(defined_schema).option("header","true").csv(path)
df.write.mode("overwrite").partitionBy("year","month").format("delta").save(target)
4
Your source system goes down mid-pipeline run. Half the data has been written to Delta. What do you do?
Hard
v
Immediate action - halt dependent jobs: Stop any downstream jobs from reading partial data. A corrupt Silver table feeding a Gold report is worse than a delayed report.
Delta time travel to the rescue: Use RESTORE TABLE to roll back to the last clean version: RESTORE TABLE bronze.orders TO VERSION AS OF 12. Delta's transaction log makes this instant - no data is actually deleted until VACUUM runs.
Identify what committed vs what didn't: Check the Delta transaction log (DESCRIBE HISTORY) to see exactly which files were committed. Anything uncommitted was never visible - Delta's atomicity means partial writes don't appear to readers.
Watermark / checkpoint strategy: After rollback, rerun from the last successful checkpoint/watermark. This is why watermarks exist - you reprocess from a known-good state, not from scratch.
Idempotency check: Make sure your pipeline is idempotent before rerunning. "MERGE based on primary key" is idempotent; "INSERT INTO" is not - it would double-count rows.
5
A stakeholder wants a new pipeline feature delivered in 2 days. Your honest estimate is 2 weeks. How do you handle this?
Hard
v
Don't agree to an unrealistic timeline: Committing to 2 days and delivering broken code in 5 days is worse than pushing back honestly. Failed commitments destroy trust faster than timeline negotiations.
Break it into an MVP: What is the minimum that unblocks the stakeholder? Often a simpler version solves 80% of the need. "I can deliver X in 3 days. Full feature in 2 weeks. Does X unblock you?"
Explain the complexity (briefly): Not to complain, but to calibrate. "This requires schema changes, backfill of 18 months of data, and testing against 3 downstream reports. Rushing increases risk of data quality issues in production."
Negotiate scope, not just time: Can we skip the historical backfill for now? Can we use an approximation for the edge case? What's the actual deadline driver - is there a meeting or a demo?
Escalate if needed: If they insist on 2 days and you're sure it's impossible safely, loop in your manager. Let the business decision be made at the right level with full information.
6
You discover your pipeline has been leaking PII to an unprotected storage location for the past 3 months. What do you do?
Hard
v
Step 1 - Contain immediately: Remove access to the exposed location NOW. Revoke any keys or SAS tokens. This is the highest priority - stop the bleeding before anything else.
Step 2 - Preserve evidence (don't delete yet): Before cleaning up, take logs of who accessed the location and when. This is needed for the incident report. Premature deletion can destroy evidence needed for legal/compliance assessment.
Step 3 - Escalate immediately: This is not a "fix quietly and move on" situation. Inform your manager and security team within the hour. GDPR requires breach notification within 72 hours. Do not try to handle this alone.
Step 4 - Assess scope: What data was exposed? Which individuals? What fields (names, emails, SSNs, financial data)? How many records? Who potentially accessed it?
Step 5 - Fix the root cause: Why did the pipeline write to the wrong location? Add a check to validate target paths. Add column-level access controls via Unity Catalog. Review all other pipelines for similar issues.
Step 6 - Post-incident review: Write a blameless postmortem. What process failed that allowed this to happen? What safeguards would prevent it in all future pipelines?
7
You inherit an undocumented legacy pipeline that runs daily. How do you understand and safely improve it?
Medium
v
First - observe, don't touch: Watch the pipeline run successfully at least 2-3 times before changing anything. Understand what "success" looks like. Check the logs for any warnings that get ignored. This is your baseline.
Map the data lineage: Source tables - transformations - target tables. Draw this manually if needed. Understanding the flow is more important than understanding the code at first.
Find the undocumented business rules: "Why is this filter here?" Often the original author has moved on, but the change was made for a reason (a data quality incident, a business rule exception). Git blame + commit messages are your first clue.
Add observability before changing code: Add row count logging, null checks, and timing metrics FIRST. Now when something changes, you'll know immediately.
Small changes with rollback plans: Change one thing at a time. Keep the old version running in parallel until the new version is proven. Never do a big bang rewrite of a legacy pipeline - too many unknown behaviors to safely replicate at once.
8
Two source systems are giving conflicting numbers for the same metric. Analysts are using both. What do you do?
Hard
v
This is a data governance problem, not just a technical one: There is no "right" answer from the code - the business needs to decide which definition wins. Your job is to facilitate that decision, not make it unilaterally.
Quantify the discrepancy: How different are the numbers? 0.1% = rounding. 15% = fundamentally different definitions. Knowing the magnitude helps prioritize urgency.
Find the definition difference: Common causes - different date/time filters, different currency conversion rates, different customer segments, different order statuses included, different aggregation levels.
Bring the right stakeholders: Schedule a meeting with the owners of both sources and the business stakeholders who consume the metric. Present both definitions side by side. The business decides which is "official."
Prevent recurrence: Once decided, document the official definition in a data catalog. Deprecate or clearly label the non-official source. Build a single "golden" metric table from the agreed definition going forward.
9
You get a 3 AM alert that your Azure Databricks cluster cost has spiked 10x above normal. What's your runbook?
Hard
v
Step 1 - Check running clusters immediately: Azure Portal - Databricks workspace - Clusters tab. Look for any cluster that should have been terminated (job clusters that didn't auto-terminate, interactive clusters left running overnight).
Step 2 - Kill obviously rogue resources: Terminate any cluster you can't explain. A large all-purpose cluster left running overnight at $5/DBU x 20 DBUs = $100/hour = $1000+ by morning.
Step 3 - Check Azure Cost Analysis: Filter by resource type and resource group. Identify which specific resource caused the spike. Was it compute, storage (ADLS operations), or networking?
Step 4 - Investigate cause: A runaway job in an infinite retry loop? A scheduled job with a bug causing it to spawn many jobs? An auto-scaling cluster that grew unexpectedly due to data volume increase?
Step 5 - Fix and safeguard: Add auto-termination to all clusters (never leave it unset). Add cluster policies that cap max workers. Set Azure Cost Alerts with stricter thresholds. Add cluster activity monitoring.
10
Your team is asked to migrate a legacy on-premise Oracle data warehouse to Azure. How do you approach this?
Medium
v
Phase 1 - Discovery (2-4 weeks): Catalog all source tables, their sizes, update frequencies, and downstream consumers. Interview the business to understand which reports are critical vs rarely used. Map the Oracle-specific features used (packages, sequences, partitioning).
Phase 2 - Architecture design: Target architecture - ADLS Gen2 as data lake (Bronze/Silver/Gold), Delta Lake format, Databricks for transformation, Synapse Analytics for SQL consumers. Map Oracle tables to Delta tables with appropriate partitioning.
Phase 3 - Parallel run strategy: Never big-bang cut over. Run Oracle and Azure in parallel. Build reconciliation queries that compare row counts and key metrics between both systems daily. Run parallel for 2-4 weeks minimum.
Phase 4 - Migrate in waves: Historical batch tables first (low risk), then daily batch, then near-real-time, then real-time last. Each wave runs parallel and validates before consumers switch over.
Key risks to flag: Oracle-specific SQL that won't run in Spark (ROW_NUMBER with custom RANGE, custom PL/SQL functions, Oracle-specific date formats). Budget ~20% of timeline for these rewrites. Test with actual data volumes, not samples.
