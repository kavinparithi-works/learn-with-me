Q1 What is the difference between WHERE and HAVING? Easy ›

WHERE filters rows before GROUP BY aggregation. HAVING filters after aggregation. WHERE cannot use aggregate functions because it runs before aggregation. HAVING operates on grouped results and can reference aggregate functions like COUNT, SUM, AVG.

Use WHERE to reduce rows before grouping for better performance. Use HAVING to filter groups based on aggregate conditions. Both can appear in the same query.

SELECT dept, COUNT(*) as cnt
FROM employees
WHERE salary > 50000
GROUP BY dept
HAVING COUNT(*) > 5;
Follow-up questions:
  • Can you use both WHERE and HAVING in the same query?
  • What happens if you omit GROUP BY when using HAVING?
  • Can WHERE filter on a SELECT alias?
Q2 Explain window functions with a real example Medium ›

Window functions compute values across a set of rows related to the current row without collapsing them into a single output row like GROUP BY does. They use the OVER() clause with optional PARTITION BY and ORDER BY.

Common window functions include ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, SUM, AVG, and NTILE. The FRAME clause (ROWS BETWEEN or RANGE BETWEEN) controls which rows are included in the window calculation. Unlike GROUP BY, window functions preserve all original rows while adding a computed column alongside existing data.

SELECT emp_id, dept, salary,
  AVG(salary) OVER (
    PARTITION BY dept
    ORDER BY salary
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) as running_avg
FROM employees;
Follow-up questions:
  • What is the difference between ROWS and RANGE frames?
  • Can you nest window functions?
  • When would you use PARTITION BY without ORDER BY?
Q3 How would you find the second highest salary? Easy ›

There are multiple approaches to this classic problem. The DENSE_RANK approach handles ties correctly by assigning the same rank to equal salaries. A subquery approach works by excluding the maximum salary and finding the max of the remaining rows.

DENSE_RANK is generally preferred because it correctly handles duplicate salary values -- if two employees share the top salary, the second distinct value is returned.

WITH ranked AS (
  SELECT salary,
    DENSE_RANK() OVER (ORDER BY salary DESC) as rnk
  FROM employees
)
SELECT MIN(salary) FROM ranked WHERE rnk = 2;
Follow-up questions:
  • What if multiple employees share the second highest salary?
  • How would you get the Nth highest salary?
  • How would you do this per department?
Q4 What is a CTE and when would you use it over a subquery? Medium ›

A Common Table Expression (CTE) is a named temporary result set defined with the WITH keyword before the main query. CTEs improve readability by giving a meaningful name to complex subqueries. They can be referenced multiple times in the same query without repeating the logic.

Recursive CTEs enable traversal of hierarchical data like org charts or bill of materials. Subqueries are fine for simple one-off filters but CTEs shine when logic is complex or reused. Some databases materialize CTEs as temporary tables, others inline them -- check your database's behavior.

WITH dept_avg AS (
  SELECT department, AVG(salary) as avg_sal
  FROM employees
  GROUP BY department
),
ranked AS (
  SELECT e.*, d.avg_sal,
    DENSE_RANK() OVER (ORDER BY e.salary DESC) as rnk
  FROM employees e
  JOIN dept_avg d ON e.department = d.department
)
SELECT * FROM ranked WHERE rnk <= 3;
Follow-up questions:
  • What is a recursive CTE?
  • Does a CTE always improve performance over a subquery?
  • Can you use multiple CTEs in one query?
Q5 Explain INNER JOIN vs LEFT JOIN with an example Easy ›

INNER JOIN returns only rows where the join condition matches in both tables, discarding unmatched rows from either side. LEFT JOIN (also called LEFT OUTER JOIN) returns all rows from the left table and matching rows from the right table.

Where no match exists in the right table, NULL values fill the right table columns. Use INNER JOIN when you only care about matched records. Use LEFT JOIN when you want all records from the left table regardless of whether a match exists on the right side.

-- INNER JOIN: only matched rows
SELECT o.order_id, c.customer_name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.customer_id;

-- LEFT JOIN: all orders, NULLs where no customer
SELECT o.order_id, c.customer_name
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.customer_id;
Follow-up questions:
  • What is a CROSS JOIN?
  • What is a FULL OUTER JOIN?
  • How does NULL handling differ in each join type?
Q6 How does GROUP BY work internally? Medium ›

GROUP BY works by collecting all rows with the same combination of grouped column values and combining them into a single output row. Internally, databases use either a hashing approach or a sorting approach.

In the hash approach, rows are placed into hash buckets based on their group key, then aggregates are computed per bucket -- this works well when data fits in memory. In the sort approach, rows are sorted by the group key, then aggregated in a single pass -- better for very large datasets or when an index already provides sorted order. Query optimizers choose the approach based on data size and available indexes.

SELECT department,
  COUNT(*) as headcount,
  AVG(salary) as avg_salary,
  MAX(salary) as max_salary
FROM employees
WHERE active = true
GROUP BY department
ORDER BY avg_salary DESC;
Follow-up questions:
  • Can you GROUP BY a column not in SELECT?
  • What happens when you GROUP BY multiple columns?
  • How does GROUPING SETS differ from GROUP BY?
Q7 Write a query to find duplicate records Easy ›

Duplicates can be found using GROUP BY with HAVING COUNT(*) greater than 1. This identifies which values repeat. To retrieve the actual duplicate rows with all columns, use a subquery or window function approach.

ROW_NUMBER() partitioned by the duplicate-identifying columns allows flagging and removing extra copies while keeping one. The GROUP BY approach is simpler for detection. The ROW_NUMBER approach is better for deduplication because you can choose which copy to keep based on a business rule like most recent timestamp.

-- Find which emails are duplicated:
SELECT email, COUNT(*) as cnt
FROM users
GROUP BY email
HAVING COUNT(*) > 1;

-- Retrieve full duplicate rows using ROW_NUMBER:
SELECT * FROM (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY email
      ORDER BY created_at DESC
    ) as rn
  FROM users
) t
WHERE rn > 1;
Follow-up questions:
  • How would you delete duplicates while keeping the most recent?
  • What if duplicates span multiple columns?
  • How would you handle duplicates in a very large table?
Q8 What is the difference between RANK, DENSE_RANK, and ROW_NUMBER? Medium ›

All three assign a number to each row within a window partition, but they differ in how they handle ties. ROW_NUMBER assigns a unique sequential integer to every row -- tied rows get different numbers based on arbitrary ordering, so no two rows share the same number.

RANK assigns the same rank to tied rows but then skips numbers (1, 1, 3 for two ties then the next). DENSE_RANK assigns the same rank to tied rows and does not skip numbers (1, 1, 2 for two ties). Use ROW_NUMBER for unique row identification, RANK for competition-style ranking where gaps matter, DENSE_RANK when you need contiguous rank values.

SELECT name, salary,
  ROW_NUMBER() OVER (ORDER BY salary DESC) as row_num,
  RANK()       OVER (ORDER BY salary DESC) as rnk,
  DENSE_RANK() OVER (ORDER BY salary DESC) as dense_rnk
FROM employees;

-- Example output for salary 100, 100, 80:
-- row_num: 1, 2, 3
-- rnk:     1, 1, 3
-- dense_rnk: 1, 1, 2
Follow-up questions:
  • When would you choose RANK over DENSE_RANK?
  • Can you use these without PARTITION BY?
  • What happens with NULL values in the ORDER BY?
Q9 How would you pivot rows to columns in SQL? Medium ›

Pivoting transforms unique values in one column into multiple output columns. The CASE WHEN approach with conditional aggregation works in all SQL databases: use SUM or MAX with a CASE expression for each target column.

The PIVOT syntax (available in SQL Server and Oracle) provides cleaner syntax for the same operation. Dynamic pivoting -- when column values are not known in advance -- requires dynamic SQL to generate the query string at runtime. The reverse operation, columns to rows, is called UNPIVOT and can also be done with UNION ALL.

-- Conditional aggregation (works everywhere):
SELECT month,
  SUM(CASE WHEN product = 'A' THEN revenue END) as product_a,
  SUM(CASE WHEN product = 'B' THEN revenue END) as product_b,
  SUM(CASE WHEN product = 'C' THEN revenue END) as product_c
FROM sales
GROUP BY month;

-- SQL Server PIVOT syntax:
SELECT month, [A] as product_a, [B] as product_b
FROM sales
PIVOT (SUM(revenue) FOR product IN ([A],[B])) p;
Follow-up questions:
  • How would you pivot when you do not know the column values in advance?
  • What is the equivalent in Python/Pandas?
  • How does PIVOT differ from crosstab in PostgreSQL?
Q10 What are indexes? When should you and should you NOT use them? Medium ›

An index is a data structure (typically a B-tree) that allows the database to locate rows matching a condition without scanning every row in the table. Clustered indexes determine the physical row order on disk -- a table can have only one. Non-clustered indexes are separate structures with pointers back to the main rows.

Use indexes on columns frequently used in WHERE clauses, JOIN conditions, and ORDER BY. Avoid indexes on low-cardinality columns (boolean flags, status columns with few values), heavily written tables where each INSERT/UPDATE/DELETE also updates all indexes, and very wide columns. Composite indexes should match your most common query filter order.

-- Create a single-column index:
CREATE INDEX idx_employees_dept ON employees(department);

-- Create a composite index:
CREATE INDEX idx_orders_composite ON orders(customer_id, order_date);

-- Check if index is used:
EXPLAIN ANALYZE
SELECT * FROM employees WHERE department = 'Engineering';
Follow-up questions:
  • What is a covering index?
  • How do you identify missing indexes?
  • What is the difference between an index scan and a full table scan?
Q11 Write a query to calculate a 7-day rolling average Medium ›

A rolling average uses a window function with a ROWS BETWEEN frame clause. ROWS BETWEEN 6 PRECEDING AND CURRENT ROW includes the current row and the 6 rows before it, giving a 7-row window.

For date-based rolling windows where some dates may have no data, RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW is more semantically accurate since it operates on date values rather than row counts. Always confirm with your interviewer whether the rolling average should cover exactly 7 calendar days or 7 data points, as the answer differs when there are gaps in the data.

SELECT
  event_date,
  daily_revenue,
  AVG(daily_revenue) OVER (
    ORDER BY event_date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) as rolling_7day_avg
FROM daily_sales
ORDER BY event_date;
Follow-up questions:
  • How would you compute a 30-day rolling average?
  • How does ROWS BETWEEN differ from RANGE BETWEEN for date windows?
  • What if there are gaps in the date series?
Q12 How would you detect data gaps in a time-series table? Hard ›

Detecting gaps requires comparing each row's timestamp to the next expected timestamp. Using LAG or LEAD, you can find rows where the difference to the adjacent row exceeds the expected interval.

In PostgreSQL, generate_series produces a complete date spine, and a LEFT JOIN against the actual data reveals missing dates as NULL rows -- this is the most explicit approach. In Spark, you can generate an expected key range using a sequence or cross-join approach. Always clarify the expected granularity (hourly, daily, per user) and whether gaps are normal for your data.

-- LAG approach: find gaps greater than 1 day:
SELECT curr_date, prev_date,
  curr_date - prev_date as gap_days
FROM (
  SELECT
    event_date as curr_date,
    LAG(event_date) OVER (ORDER BY event_date) as prev_date
  FROM time_series
) t
WHERE curr_date - prev_date > 1;

-- Date spine approach (PostgreSQL):
SELECT d.date, t.value
FROM generate_series(
  '2024-01-01'::date,
  '2024-12-31'::date,
  '1 day'
) d(date)
LEFT JOIN time_series t ON t.event_date = d.date
WHERE t.event_date IS NULL;
Follow-up questions:
  • How would you detect gaps per user or per device?
  • How do you handle irregular expected intervals?
  • What if the table has billions of rows?
Q13 Explain query execution order Easy ›

SQL queries execute in a logical order that differs from the written order. The processing sequence is: FROM and JOIN (identify and combine tables), then WHERE (filter individual rows), then GROUP BY (group rows into aggregation buckets), then HAVING (filter groups), then SELECT (compute output expressions and aliases), then DISTINCT (remove duplicate rows), then ORDER BY (sort results), then LIMIT/OFFSET (restrict output count).

This order explains why you cannot use a SELECT alias in a WHERE clause -- WHERE runs before SELECT evaluates the alias -- but you can use a SELECT alias in ORDER BY because ORDER BY runs after SELECT.

-- Logical execution order demonstrated with comments:
SELECT dept, COUNT(*) as headcount    -- 5: compute output columns
FROM employees                         -- 1: load the table
WHERE active = true                    -- 2: filter individual rows
GROUP BY dept                          -- 3: group into buckets
HAVING COUNT(*) > 2                    -- 4: filter groups
ORDER BY headcount DESC                -- 6: sort results
LIMIT 10;                              -- 7: restrict to 10 rows
Follow-up questions:
  • Why can you not reference a SELECT alias in a WHERE clause?
  • When is DISTINCT applied?
  • How does this order affect where window functions can appear?
Q14 What is a self-join and when would you use it? Medium ›

A self-join joins a table to itself by aliasing it twice with different names. This is useful when rows in a table have relationships with other rows in the same table. Classic use cases include hierarchical data (employee-manager relationships where manager_id references the same employees table), finding pairs of rows that satisfy a condition (customers in the same city), and comparing rows to their neighbors.

Before window functions became widely available, self-joins were used for row-to-row comparisons. Window functions like LAG and LEAD are now more efficient alternatives for sequential row comparisons.

-- Employee-manager self-join:
SELECT
  e.name as employee,
  e.salary as emp_salary,
  m.name as manager,
  m.salary as mgr_salary
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id;

-- Find employees earning more than their manager:
SELECT e.name
FROM employees e
JOIN employees m ON e.manager_id = m.emp_id
WHERE e.salary > m.salary;
Follow-up questions:
  • How does a recursive CTE compare to a self-join for deep hierarchies?
  • Can you self-join on multiple conditions?
  • How would you find all pairs of employees in the same department?
Q15 How would you optimize a slow query? Hard ›

Start with EXPLAIN ANALYZE to view the actual execution plan. Look for sequential scans on large tables (need indexes), high estimated versus actual row count discrepancies (stale statistics), expensive sort and hash operations, and nested loop joins on large datasets.

Add indexes on filter and join columns. Rewrite correlated subqueries as JOINs which are usually much faster. Avoid SELECT star to reduce I/O. Use CTEs or temp tables to break complex queries into steps the optimizer can handle better. Check table statistics freshness and run ANALYZE if needed. For recurring expensive queries, consider materialized views.

-- Step 1: view execution plan:
EXPLAIN ANALYZE
SELECT o.order_id, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.order_date > '2024-01-01';

-- Step 2: add targeted index:
CREATE INDEX idx_orders_date ON orders(order_date, customer_id);

-- Step 3: rewrite correlated subquery as JOIN:
-- Before (slow):
SELECT name FROM employees e
WHERE salary > (SELECT AVG(salary) FROM employees WHERE department = e.department);

-- After (fast):
SELECT e.name FROM employees e
JOIN (
  SELECT department, AVG(salary) as avg_sal
  FROM employees GROUP BY department
) d ON e.department = d.department
WHERE e.salary > d.avg_sal;
Follow-up questions:
  • What does nested loop vs hash join vs merge join mean in the execution plan?
  • How would you tune a query that is slow only for certain parameter values?
  • What is a stale query plan and how does it cause issues?
Q16 What is a materialized view? Medium ›

A materialized view stores the query result physically on disk, unlike a regular view which re-executes the underlying query each time it is queried. This makes reads much faster for expensive aggregations or complex joins that run frequently.

The tradeoff is data staleness -- the materialized view reflects the state at the last refresh, not real-time data. Refresh strategies include complete refresh (full recompute from scratch), fast/incremental refresh (apply only changes since last refresh), and ON COMMIT versus ON DEMAND scheduling. In modern data platforms, materialized views are used to pre-aggregate metrics tables that power dashboards.

-- PostgreSQL: create materialized view:
CREATE MATERIALIZED VIEW monthly_revenue AS
SELECT
  DATE_TRUNC('month', order_date) as month,
  SUM(amount) as revenue,
  COUNT(*) as order_count
FROM orders
GROUP BY 1;

-- Manual refresh (blocks reads):
REFRESH MATERIALIZED VIEW monthly_revenue;

-- Concurrent refresh (no lock, requires unique index):
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_revenue;
Follow-up questions:
  • When would a regular view perform better than a materialized view?
  • How do you handle incremental refresh efficiently?
  • What is the equivalent concept in Databricks Delta Lake?
Q17 Explain SCD Type 1 and SCD Type 2 Hard ›

Slowly Changing Dimensions describe how dimension table data changes over time. SCD Type 1 overwrites old values with new ones, keeping no history. Use Type 1 when history is irrelevant such as correcting typos or updating phone numbers.

SCD Type 2 preserves complete history by inserting a new row for each change, with effective_from and effective_to date columns and an is_current flag. The original row has its effective_to date closed and is_current set to false. Use Type 2 when historical accuracy matters, for example tracking customer address changes that affect past order attribution. SCD Type 3 adds a previous_value column but only tracks one change back.

-- SCD Type 1 (overwrite, no history):
MERGE INTO dim_customer t
USING staging s ON t.customer_id = s.customer_id
WHEN MATCHED THEN
  UPDATE SET t.email = s.email, t.phone = s.phone
WHEN NOT MATCHED THEN
  INSERT (customer_id, email, phone)
  VALUES (s.customer_id, s.email, s.phone);

-- SCD Type 2 (history): close old row:
UPDATE dim_customer
SET effective_to = CURRENT_DATE, is_current = false
WHERE customer_id = 123 AND is_current = true AND email != 'new@email.com';

-- SCD Type 2: insert new row:
INSERT INTO dim_customer
  (customer_id, email, effective_from, effective_to, is_current)
VALUES
  (123, 'new@email.com', CURRENT_DATE, '9999-12-31', true);
Follow-up questions:
  • What is SCD Type 3 and when would you use it?
  • How do you query a SCD Type 2 table for a point-in-time snapshot?
  • How does Delta Lake time travel relate to SCD Type 2?
Q18 How would you write an upsert (MERGE) in SQL? Medium ›

An upsert inserts a row if it does not exist or updates it if it does. The MERGE statement (standard SQL supported in SQL Server, Oracle, Spark SQL, and Delta Lake) is the most flexible form. PostgreSQL uses INSERT ON CONFLICT syntax. MySQL uses INSERT ON DUPLICATE KEY UPDATE.

The source table or subquery provides new data. The MERGE compares on a business key, then applies WHEN MATCHED (update existing rows) and WHEN NOT MATCHED (insert new rows) clauses. Delta Lake extends MERGE with WHEN NOT MATCHED BY SOURCE for soft deletes of rows missing from the source.

-- Standard MERGE (SQL Server, Spark, Delta Lake):
MERGE INTO target_table t
USING source_table s ON t.id = s.id
WHEN MATCHED THEN
  UPDATE SET t.value = s.value, t.updated_at = CURRENT_TIMESTAMP
WHEN NOT MATCHED THEN
  INSERT (id, value, created_at)
  VALUES (s.id, s.value, CURRENT_TIMESTAMP);

-- PostgreSQL INSERT ON CONFLICT:
INSERT INTO users (user_id, email, updated_at)
VALUES (1, 'new@email.com', NOW())
ON CONFLICT (user_id)
DO UPDATE SET
  email = EXCLUDED.email,
  updated_at = EXCLUDED.updated_at;
Follow-up questions:
  • How does MERGE perform on large tables?
  • What are the risks of MERGE in a concurrent write environment?
  • How do you add a soft-delete clause to MERGE?
Q19 What is the difference between DELETE and TRUNCATE? Easy ›

DELETE removes rows one by one, logging each deletion in the transaction log. It fires row-level triggers, can target specific rows with a WHERE clause, and supports rollback within a transaction. TRUNCATE removes all rows by deallocating entire data pages -- it is much faster because it uses minimal transaction logging.

TRUNCATE cannot be filtered with WHERE. TRUNCATE resets identity and auto-increment counters. In most databases TRUNCATE is DDL and cannot be rolled back outside an explicit transaction. PostgreSQL is an exception and supports transactional TRUNCATE. TRUNCATE does not fire row-level triggers.

-- DELETE with filter (logged, can rollback, triggers fire):
DELETE FROM logs WHERE created_at < '2023-01-01';

-- TRUNCATE (fast, minimal log, resets identity):
TRUNCATE TABLE staging_load;

-- PostgreSQL transactional TRUNCATE:
BEGIN;
TRUNCATE TABLE temp_data;
ROLLBACK; -- This works in PostgreSQL only
Follow-up questions:
  • Can TRUNCATE be rolled back?
  • Why is TRUNCATE faster than DELETE on large tables?
  • What happens to foreign key constraints when you TRUNCATE a referenced table?
Q20 Explain partitioning in SQL and databases Medium ›

Partitioning divides a large table into smaller physical segments called partitions based on a column value. Range partitioning splits rows by value ranges such as by year for date columns. List partitioning splits by discrete values such as by region or status. Hash partitioning distributes rows evenly based on a hash of the partition key -- useful when no natural range exists.

Queries filtering on the partition key benefit from partition pruning: the database reads only relevant partitions, skipping the rest entirely. In cloud data platforms like Delta Lake and BigQuery, partitioning is a primary performance lever. Over-partitioning creates the small file problem.

-- PostgreSQL range partitioning:
CREATE TABLE orders (
  order_id INT,
  order_date DATE,
  amount DECIMAL
) PARTITION BY RANGE (order_date);

CREATE TABLE orders_2024
  PARTITION OF orders
  FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE orders_2025
  PARTITION OF orders
  FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- Delta Lake partitioning:
df.write.format("delta").partitionBy("year", "month").save("/data/orders")
Follow-up questions:
  • What is the small file problem and how does it relate to over-partitioning?
  • How does partition pruning work in the query planner?
  • What is the difference between partitioning and bucketing in Hive/Spark?
Q21 How would you find customers who made purchases in both January and February? Medium ›

There are two clean approaches. INTERSECT finds customer IDs appearing in both month result sets -- it is the most readable. A GROUP BY approach counts distinct months per customer and filters for those with exactly 2 distinct months from the set of January and February.

Avoid using OR in a WHERE clause as that would find customers in either month. For the INTERSECT approach, note it automatically deduplicates, so a customer making 5 purchases in January appears only once in each set.

-- INTERSECT approach (clean, readable):
SELECT customer_id FROM orders WHERE MONTH(order_date) = 1
INTERSECT
SELECT customer_id FROM orders WHERE MONTH(order_date) = 2;

-- GROUP BY approach (more flexible):
SELECT customer_id
FROM orders
WHERE MONTH(order_date) IN (1, 2)
GROUP BY customer_id
HAVING COUNT(DISTINCT MONTH(order_date)) = 2;
Follow-up questions:
  • How would you extend this to find customers active every month of the year?
  • How does INTERSECT differ from INNER JOIN?
  • What if a customer made multiple purchases in each month?
Q22 Write a query to calculate month-over-month growth Medium ›

Month-over-month (MoM) growth compares the current period's metric to the prior period. LAG() retrieves the previous period's value within the window. The growth rate formula is: (current minus previous) divided by previous times 100.

Wrap the denominator in NULLIF to handle division by zero when previous month revenue is zero. For clarity, compute the LAG value in a CTE first rather than nesting LAG multiple times in the arithmetic expression. Always verify whether gaps (months with no revenue) should appear as NULL growth or zero growth.

WITH monthly AS (
  SELECT
    month,
    revenue,
    LAG(revenue) OVER (ORDER BY month) as prev_revenue
  FROM monthly_revenue
)
SELECT
  month,
  revenue,
  prev_revenue,
  ROUND(
    (revenue - prev_revenue) / NULLIF(prev_revenue, 0) * 100,
    2
  ) as mom_growth_pct
FROM monthly
ORDER BY month;
Follow-up questions:
  • How would you calculate year-over-year growth?
  • How would you compute MoM growth per region or product line?
  • What if some months have zero revenue?
Q23 How do you handle NULL values in aggregations? Easy ›

COUNT(*) counts all rows including those with NULLs. COUNT(column) skips NULLs in that column. SUM, AVG, MIN, and MAX all silently ignore NULL values. This means AVG of (1, NULL, 3) returns 2.0 -- it divides by 2 not 3 -- which can produce misleading results if NULLs represent missing data that should count as zero.

Use COALESCE(column, 0) to substitute zero for NULL before aggregating when that matches your business logic. Use NULLIF(value, 0) to convert zeros to NULL when you want to exclude zeros from averages. Always be explicit about NULL handling in business metric calculations.

SELECT
  COUNT(*) as total_rows,
  COUNT(email) as rows_with_email,
  COUNT(*) - COUNT(email) as null_email_count,
  SUM(COALESCE(amount, 0)) as total_amount_nulls_as_zero,
  AVG(amount) as avg_ignoring_nulls,
  AVG(COALESCE(amount, 0)) as avg_nulls_as_zero,
  AVG(NULLIF(amount, 0)) as avg_excluding_zeros
FROM orders;
Follow-up questions:
  • What does AVG return if all values in the column are NULL?
  • How does NULL propagate through arithmetic expressions?
  • How would you count the number of NULLs in a column?
Q24 What is the difference between EXISTS and IN? Medium ›

IN checks if a value matches any value in a list or subquery result set and retrieves the full result set before checking. EXISTS checks if a subquery returns at least one row and short-circuits on the first match, never retrieving the full result.

For large subquery result sets, EXISTS is generally faster because it stops at the first match. Critically, IN has a NULL trap: if the subquery or list contains any NULL, NOT IN returns NULL (not FALSE) for all rows, silently filtering out all records. NOT EXISTS handles NULLs correctly. Modern query optimizers often convert between EXISTS and IN automatically, but understanding the NULL difference is important.

-- IN (retrieves full subquery result):
SELECT * FROM orders
WHERE customer_id IN (
  SELECT customer_id FROM vip_customers
);

-- EXISTS (short-circuits on first match, faster for large sets):
SELECT * FROM orders o
WHERE EXISTS (
  SELECT 1 FROM vip_customers v
  WHERE v.customer_id = o.customer_id
);

-- NULL trap with NOT IN -- always add IS NOT NULL guard:
SELECT * FROM orders
WHERE customer_id NOT IN (
  SELECT customer_id FROM banned

Q1 Lazy evaluation -- what it is and why it matters Medium ›

PySpark uses lazy evaluation for all transformations. When you call filter, map, select, join, or withColumn, no data is moved or computed. Instead, Spark builds a Directed Acyclic Graph (DAG) representing the sequence of operations.

The DAG is only executed when an action is called such as collect, count, write, or show. This enables the Catalyst optimizer to inspect the full transformation chain before executing, applying optimizations like predicate pushdown (filter early), column pruning (read only needed columns), and join reordering. Lazy evaluation also avoids computing intermediate results that are never used, and allows Spark to pipeline multiple operations into a single pass over the data.

from pyspark.sql import SparkSession
from pyspark.sql.functions import hour, col

spark = SparkSession.builder.appName("demo").getOrCreate()

# These lines do NOT execute yet (lazy transformations):
df = spark.read.parquet("/data/events")
filtered = df.filter(df.event_type == "click")
with_col = filtered.withColumn("hour", hour(col("event_time")))

# This triggers execution of the FULL DAG:
with_col.write.parquet("/output/clicks")  # action

Q2 Transformations vs actions Easy ›

Transformations create a new RDD or DataFrame and are lazy -- they define what to compute but do not execute immediately. Actions trigger actual computation of the accumulated DAG and return a result to the driver or write data to storage.

Common transformations include filter, map, select, withColumn, join, groupBy, distinct, union, and cache. Common actions include collect, count, show, write, save, take, first, foreach, and reduce. A critical mistake is calling collect on a large DataFrame -- it sends all data to the driver and causes out-of-memory errors. Always prefer distributed write actions over collect for large data.

Q3 Partitions in Spark -- how to decide the right number Medium ›

Partition count directly controls parallelism. A common rule of thumb is 2 to 4 partitions per CPU core available in your cluster. For shuffle partitions generated after groupBy or join operations, the default is 200 -- tune this with spark.sql.shuffle.partitions.

Q4 What causes a shuffle? How to minimize it Hard ›

A shuffle is a full redistribution of data across the cluster over the network -- the most expensive Spark operation because it involves serialization, network transfer, and disk I/O. Shuffles are triggered by groupBy, join (when not a broadcast join), repartition, distinct, orderBy for global sort, and aggregations.

To minimize shuffles: use broadcast joins for small dimension tables to avoid shuffling the large table; pre-partition data on the join key before joining so Spark can do a local join; use coalesce instead of repartition when reducing partition count; avoid unnecessary distinct and global orderBy; use reduceByKey instead of groupByKey in the RDD API since reduceByKey does partial aggregation locally. AQE can optimize some shuffles at runtime.

Q5 Broadcast joins and when to use them Medium ›

Q6 cache() vs persist() Easy ›

Both cache and persist store a DataFrame in memory to avoid recomputing it on subsequent actions. cache() is shorthand for persist(StorageLevel.MEMORY_AND_DISK) -- the most common storage level. persist() accepts an explicit StorageLevel for fine-grained control.

MEMORY_ONLY stores deserialized Java objects in memory -- fastest to access but uses the most memory and spills nothing. MEMORY_AND_DISK spills overflow partitions to disk when memory is insufficient. MEMORY_ONLY_SER serializes data to use less memory at the cost of CPU overhead for deserialization. DISK_ONLY avoids heap memory entirely. Always call unpersist() when the cached data is no longer needed to free resources for other jobs.

# Explicit storage level:
df.persist(StorageLevel.MEMORY_AND_DISK)

# Off-heap (avoids GC pressure):
df.persist(StorageLevel.OFF_HEAP)

# Available levels:
# MEMORY_ONLY, MEMORY_ONLY_SER
# MEMORY_AND_DISK, MEMORY_AND_DISK_SER
# DISK_ONLY, OFF_HEAP

# Release when done:
df.unpersist()

Q7 Handling skewed data in a join Hard ›

Data skew occurs when one partition contains far more rows than others, causing one task to run much longer than all others. In joins, skew happens when a join key has severe cardinality imbalance -- for example one customer_id accounts for 90% of orders.

The salting technique addresses this: add a random integer suffix (0 to N-1) to the skewed key in the large table, distributing that key's data across N partitions. Then explode the small table to N copies with each suffix value, so each salted partition has a matching small table copy. AQE in Spark 3+ can automatically detect and split skewed partitions without manual salting.

from pyspark.sql.functions import (
  concat, lit, col, floor, rand, explode, array
)

N = 10  # salt factor

# Salt the large (skewed) table:
large_salted = large_df.withColumn(
  "salted_key",
  concat(col("customer_id"), lit("_"),
         (floor(rand() * N)).cast("string"))
)

# Explode small table to N copies:
small_exploded = small_df.withColumn(
  "salt", explode(array([lit(i) for i in range(N)]))
).withColumn(
  "salted_key",
  concat(col("customer_id"), lit("_"), col("salt").cast("string"))
)

# Join on salted key:
result = large_salted.join(small_exploded, "salted_key") \
  .drop("salted_key", "salt")

Q10 UDFs -- how to write them and performance tradeoffs Medium ›

Python UDFs defined with udf() run Python code row by row, requiring serialization of each value from the JVM to the Python interpreter and back -- this is the main performance bottleneck and can be 10x to 100x slower than built-in Spark functions.

Pandas UDFs (also called vectorized UDFs, defined with pandas_udf) operate on pandas Series objects, batching the serialization overhead significantly and leveraging Apache Arrow for efficient transfer. Built-in Spark SQL functions run natively in the JVM with zero serialization overhead. Always prefer built-in functions first, then Pandas UDFs for custom logic, then Python UDFs only as a last resort when vectorization is impossible.

from pyspark.sql.functions import udf, pandas_udf
from pyspark.sql.types import StringType
import pandas as pd

# Python UDF (slow -- row by row, JVM/Python serialization):
@udf(StringType())
def categorize(amount):
    return "high" if amount and amount > 1000 else "low"

# Pandas UDF (fast -- vectorized, Arrow-based batching):
@pandas_udf(StringType())
def categorize_pandas(amounts: pd.Series) -> pd.Series:
    return amounts.apply(lambda x: "high" if x > 1000 else "low")

# Use Pandas UDF:
df = df.withColumn("category", categorize_pandas(col("amount")))

# Register for use in SQL:
spark.udf.register("categorize", categorize_pandas, StringType())

Q13 Incremental data loading in PySpark Hard ›

Q14 Structured Streaming and watermarking Hard ›

Structured Streaming processes data as an unbounded DataFrame, treating streaming data like a continuously growing table. Watermarking tells Spark how long to wait for late-arriving events before finalizing a time window. Without watermarks, stateful aggregations accumulate state forever and eventually OOM. Triggers control micro-batch frequency. Output modes: append (new finalized rows only), update (rows that changed), complete (full result table each micro-batch).

  .groupBy(

Q16 MERGE INTO in Delta Lake Medium ›

Q18 repartition() vs coalesce() Medium ›

repartition(n) performs a full shuffle, redistributing all data evenly to create exactly n balanced partitions. It can increase or decrease partition count. Use repartition when you need balanced output or when distributing by a specific column key for downstream operations.

Q19 Medallion Architecture Medium ›

The Medallion Architecture organizes a data lakehouse into three layers with increasing data quality. Bronze is the raw ingestion layer: data arrives as-is from source systems preserving original format, types, and values. It is append-only and serves as the permanent source of truth for replaying pipelines.

Silver is the cleansed and conformed layer: data quality rules are applied, duplicates removed, schemas standardized, and tables enriched via joins to reference data. Silver represents validated, trustworthy data. Gold is the business-ready layer: business-specific aggregations, metrics, and domain models built for analytics consumption. BI tools, data scientists, and downstream applications consume Gold.

Q7 Delta Live Tables -- when to use it over regular notebooks Medium ›

Q14 SCD Type 2 in Delta Lake using MERGE INTO Hard ›

SCD Type 2 in Delta Lake requires two atomic steps. First, expire existing active rows that have changed: update their effective_to timestamp and set is_current to false. Second, insert new current versions of changed rows plus any brand new rows. Delta Lake's ACID guarantees make each MERGE operation atomic.

The merge condition must include both the business key and is_current = true to only touch active records and avoid accidentally updating historical rows. A practical optimization: add a partition filter on the effective_to column in the merge condition to enable partition pruning and avoid scanning the entire table.

from delta.tables import DeltaTable
from pyspark.sql.functions import current_timestamp, lit

delta_tbl = DeltaTable.forName(spark, "silver.dim_customer")

# Step 1: expire rows that changed:
delta_tbl.alias("t").merge(
  source_df.alias("s"),
  "t.customer_id = s.customer_id "
  "AND t.is_current = true "
  "AND t.email != s.email"
).whenMatchedUpdate(set={
  "effective_to": "current_timestamp()",
  "is_current": "false"
}).execute()

# Step 2: insert new current versions:
(source_df
  .withColumn("effective_from", current_timestamp())
  .withColumn("effective_to",
    lit("9999-12-31 00:00:00").cast("timestamp"))
  .withColumn("is_current", lit(True))
  .write.format("delta").mode("append")
  .saveAsTable("silver.dim_customer"))

Feature Store has pre-computed user and merchant risk features refreshed hourly. ML Scoring uses an online UDF calling a Databricks Model Serving endpoint. Alert Service writes high-risk transactions (score above 0.85) to Cosmos DB for the case management system.

Scalability

  • Event Hubs partitions scale ingestion horizontally -- add partitions to match Spark executor count
  • Auto-scaling Databricks cluster grows with event volume during peak hours
  • Cosmos DB scales writes elastically with request units; partition by card_id for hot path queries
  • Feature Store lookups use in-memory Delta cache for sub-millisecond latency

Failure Handling

  • Event Hubs checkpoint offsets stored in ADLS Gen2 for exactly-once processing guarantees
  • Idempotent Cosmos DB upserts on transaction_id prevent duplicate alerts on restart
  • Dead letter queue for unparseable events -- routed to separate Bronze table for manual review
  • Model Serving endpoint has a fallback to a rule-based scorer if ML endpoint is unavailable

POS Systems (15)
→
ADF / Auto Loader
→
Bronze Delta
→
Silver Delta
→
Gold Delta
→
Power BI (50+ users)

Architecture Rationale

At 100TB scale, file layout and compaction are critical. All Delta tables partition by date at all layers. Bronze is append-only, Silver applies deduplication and SCD Type 2 for product dimension. Gold has daily and weekly aggregation tables.

OPTIMIZE runs nightly with ZORDER BY customer_id and date on the most-queried Silver and Gold tables. This co-locates data for the most common query patterns (filter by customer, filter by date range) into fewer files, reducing I/O dramatically. VACUUM with 7-day retention removes old Parquet files while preserving time travel capability. Unity Catalog governs access across 8 teams with row-level security by region.

Scalability

  • Target 200MB to 1GB file size after OPTIMIZE -- verified via Delta table properties
  • Databricks SQL Pro warehouse with 4 clusters for BI workloads; Databricks Jobs cluster for ETL
  • AQE enabled on all ETL jobs -- coalesces shuffle partitions dynamically at runtime
  • Incremental OPTIMIZE using WHERE clause on recent partitions to avoid full table scan nightly

Failure Handling

  • Delta transaction log guarantees ACID -- partial writes are rolled back automatically
  • Bronze replay: if Silver job fails for any partition, reprocess from Bronze without data loss
  • Azure Monitor alerts for job duration SLA (Gold job must complete by 06:00 local time for morning reports)

Scenario 5: Incremental Data Loading Framework

Source DB
→
Debezium CDC
→
Event Hubs
→
Bronze: Raw CDC
→
Silver: MERGE dedup
→
Gold: Incremental Agg

Architecture Rationale

Debezium captures row-level INSERT, UPDATE, DELETE changes from the source database's transaction log (no query load on source). Changes are published to Event Hubs topics partitioned by table name. Schema Registry enforces Avro schemas for CDC event payloads, preventing downstream breakage from source schema changes.

Bronze layer appends all CDC events with full payload and metadata (op_type: I/U/D, source_timestamp). Control table tracks the watermark (last processed Event Hubs offset) per source table. Silver applies MERGE INTO on the business key: inserts for new rows, updates for changed rows, soft deletes (is_deleted = true) for deleted rows. The MERGE is idempotent: processing the same CDC event twice produces the same result. Gold layer uses incremental aggregations -- only re-aggregates partitions touched by the latest Silver changes.

Scalability

  • Event Hubs throughput units scale for high-volume sources -- auto-inflate enabled
  • Silver MERGE targets only recent date partitions using a partition filter to avoid full table scan
  • Dead letter queue in ADLS Gen2 captures events that fail schema validation or MERGE for manual review

Failure Handling

  • Structured Streaming checkpoints store Event Hubs offsets -- restart from last checkpoint on failure
  • Idempotent MERGE ensures re-processing the same batch produces no duplicates
  • Monitor source-to-Gold lag: alert if Gold data is more than 30 minutes behind source transaction timestamp

Q1: Tell me about a time you handled a production data incident.

Why interviewers ask this: Tests calm under pressure, systematic debugging, communication skills, and whether you write runbooks after incidents.
Situation

Describe production impact: dashboard wrong, pipeline down, data missing. What was the business consequence?

Task

Your specific role in the response -- were you on-call, the subject matter expert, the incident lead?

Action

Diagnostic steps, root cause identification, fix deployed, stakeholder communication timeline.

Result

Fixed in X hours, runbook created, incident review completed, preventive measure shipped.

Example outline: Production dashboard showing incorrect weekly revenue. Finance team about to use numbers for board presentation. Root cause: timezone conversion bug -- timestamps converted twice in the transformation layer. Fixed by deploying corrected transformation and backfilling 30 days of data. Created runbook and added data quality assertions checking revenue within 5% of previous week as an automated guard.
Coaching tips: Show you communicate proactively -- what did you tell stakeholders and when? Mention what you learned and what systemic change you made. Interviewers want ownership, not blame-shifting.

Q2: What is the most complex data pipeline you have built?

Why interviewers ask this: Assesses technical depth, architecture decisions, and ability to handle real-world complexity at scale.
Situation

The business problem that required the pipeline. What data, at what scale, serving what consumers?

Task

Your design and implementation scope. Did you own end to end or collaborate?

Action

Architecture choices, key technical decisions, challenges you overcame.

Result

Business impact, scale achieved, reliability metrics, latency improvement.

Example outline: Real-time inventory pipeline aggregating from 15 store POS systems, 3 ERP APIs, and a supplier EDI feed. CDC from SQL Server via Debezium into Event Hubs. Bronze via Auto Loader. Silver applied SCD Type 2 for product dimension changes. Gold computed real-time available-to-promise inventory. 10-minute watermarks for late events. Reduced inventory discrepancy latency from next-day batch to 15 minutes.
Coaching tips: Show breadth (multiple sources, layers) and depth (specific technical decisions and why you made them). Quantify scale in TB processed and business impact in dollars or time saved.

Q3: Tell me about a time you disagreed with a technical decision.

Why interviewers ask this: Tests whether you can hold a position under pushback, communicate data-driven arguments, and collaborate without conflict.
Situation

The decision being made and why you disagreed. What was the technical risk or business risk?

Task

Your position and what was at stake if the wrong approach was chosen.

Action

How you raised concerns -- data, benchmarks, prototypes, written alternatives. Did you propose a time-boxed pilot?

Result

What happened -- did you win, lose, or reach a compromise? What did you learn from the outcome?

Example outline: Team decided to process a 50GB daily file without partitioning. Raised OOM risk and 4-hour job duration concerns. Prepared a 10-minute presentation with benchmarks from a similar-sized table. Proposed a 2-day pilot comparing both approaches. Pilot showed 40-minute runtime with partitioning vs 4+ hours without. Team adopted the partitioned approach as the standard for all similar jobs going forward.
Coaching tips: Argue with evidence, not opinion. Accept outcomes gracefully even when you disagree. Never make the other party look foolish in your story.

Q4: Describe a time you improved a slow or unreliable pipeline.

Why interviewers ask this: Tests debugging skills, performance optimization mindset, and ability to translate technical work into business impact.
Situation

How slow or unreliable was it? What was the downstream business impact of the failure or slowness?

Task

Your mandate: were you assigned this, or did you self-identify and propose the fix?

Action

Diagnostic steps, what profiling/tooling you used, specific root causes found, changes made.

Result

Before and after metrics: runtime, failure rate, cost, business value unlocked.

Example outline: Daily Spark job processing 200GB events was running 8 hours and failing 30% of runs due to OOM. Spark UI identified three issues: no partitioning on the large input table, two unnecessary repartition calls creating 2000 small partitions, and a cache on a DataFrame accessed only once. Removed unnecessary operations, added date partitioning, enabled AQE. Job now runs in 45 minutes with zero OOM failures in 3 months.
Coaching tips: Be specific about what profiling or debugging tool showed you the problem. Show you measured rather than guessed. Quantify technical improvement and business value separately.

Q5: Tell me about a time you learned something new quickly.

Why interviewers ask this: Tests learning agility, self-direction, and ability to become productive without hand-holding.
Situation

Task

Your specific learning goal and what you had to deliver by when.

Action

Your structured learning approach: courses, documentation, prototypes, mentors. Day-by-day breakdown shows intentionality.

Result

Time to first production delivery, quality of the output, and what you still do differently because of that learning.

Example outline: Joined a project midway requiring Delta Live Tables. Never used DLT before. Two weeks to first deliverable. Completed Databricks Academy DLT course in 2 days, built a prototype medallion pipeline in 3 days, had a code review session with a senior engineer on day 6, delivered a production Bronze-to-Gold pipeline on day 14. Coaching tips: Name specific learning resources. Show genuine curiosity about the technology. Mention one specific insight that changed how you work -- that signals depth, not just completion.

Q6: Describe a time you had conflicting priorities from stakeholders.

Why interviewers ask this: Tests prioritization framework, communication, ability to navigate politics, and stakeholder management skill.
Situation

Two stakeholders with competing requests, same timeline, fixed capacity. What were the stakes for each?

Task

Your responsibility -- you could not satisfy both, so what framework did you use to decide?

Action

How you met with each stakeholder, clarified constraints, communicated tradeoffs, and negotiated the outcome.

Result

What you delivered and when. What process improvement came out of the experience?

Met individually with both. Finance date was externally fixed. Data Science deadline was internally set and movable by 2 weeks. Delivered Finance dashboard on time. Data Science feature work shipped 2 weeks later. Now run a weekly stakeholder sync to surface conflicts earlier.
Coaching tips: Show proactive communication of tradeoffs rather than overpromising to both sides. Mention the process improvement that prevents the same situation recurring.

Q7: Have you mentored or helped a junior colleague?

Why interviewers ask this: Tests leadership potential, ability to explain complex concepts, patience, and investment in team growth.

Task

Action

Ran three pair programming sessions walking through the Spark UI together -- reading the DAG, identifying skew, interpreting memory metrics. Q8: Tell me about a time you had to work with unclear requirements.

Why interviewers ask this: Tests initiative, ability to ask the right clarifying questions, comfort with ambiguity, and judgment about when to proceed vs when to stop and ask.

Task

Action

Result

What you delivered that solved the underlying need -- and how it differed from the original vague request.

Q9: Tell me about a project that failed or did not go as planned.

Coaching tips: Own the failure genuinely. Show a concrete change in behavior afterward -- not "I learned to be more careful" but a specific process you now follow. Interviewers want a real story with real learning.
