ETL Pipeline Patterns

Production ETL pipelines follow consistent patterns: ingest raw data, validate and clean it, transform for business use, and load to target systems - with error handling and idempotency at every step.

Key Principles of Production Pipelines

  • Idempotency: Running a pipeline twice produces the same result as running it once. Use MERGE instead of INSERT, write to partitioned paths, track watermarks.
  • Incremental Loading: Process only new or changed data since the last run. Track via watermark column (updated_at), partition date, or Change Data Capture (CDC).
  • Error Isolation: Quarantine bad records to a dead-letter table instead of failing the whole pipeline. Process the 99% that's clean; fix and replay the 1%.
  • Lineage: Track where every piece of data came from, how it was transformed, and where it went. Required for debugging and compliance.
  • Observability: Log row counts, durations, anomalies, and SLA breaches. Alert before business users notice problems.
Incremental Load Pattern with Watermark
from pyspark.sql import functions as F
from delta.tables import DeltaTable

def run_incremental_load(spark, source_path, target_path, watermark_col="updated_at"):
    # 1. Read current watermark from target
    target = DeltaTable.forPath(spark, target_path)
    last_ts = target.toDF() \
        .agg(F.max(watermark_col).alias("max_ts")) \
        .collect()[0]["max_ts"]

    # 2. Read only new/changed records from source
    source_df = spark.read.parquet(source_path) \
        .filter(F.col(watermark_col) > last_ts)

    if source_df.count() == 0:
        print("No new records - skipping")
        return

    # 3. Validate
    validated_df = validate(source_df)

    # 4. MERGE (upsert) into target
    target.alias("t").merge(
        validated_df.alias("s"),
        "t.id = s.id"
    ) \
    .whenMatchedUpdateAll() \
    .whenNotMatchedInsertAll() \
    .execute()

    print(f"Loaded {validated_df.count()} records")

def validate(df):
    # Quarantine nulls in required fields
    good = df.filter(F.col("id").isNotNull() & F.col("amount").isNotNull())
    bad = df.filter(F.col("id").isNull() | F.col("amount").isNull())
    if bad.count() > 0:
        bad.withColumn("quarantine_reason", F.lit("null_required_field")) \
           .write.mode("append").save("/quarantine/bad_records")
    return good

Medallion Architecture

The Medallion Architecture organizes data into Bronze, Silver, and Gold layers - each progressively more refined, trusted, and business-ready. It's the standard pattern for modern lakehouses.

Bronze - Silver - Gold Data Quality Flow
🥉
Bronze - Raw Ingestion
Exact copy of source data, never modified. JSON/CSV/Parquet as-is. Append-only. The "source of truth" for replay. May have duplicates, nulls, bad data types.
↓ validate, deduplicate, cast types
🥈
Silver - Cleansed and Conformed
Deduplicated, schema-validated, proper data types. Nulls handled. Business rules applied. Ready for analysts. Can be used for operational dashboards.
↓ aggregate, join, enrich
🥇
Gold - Business Aggregates
Pre-aggregated, denormalized for performance. Daily KPIs, customer 360 views, revenue summaries. Directly powering BI dashboards and reports. Highly refined.

bronze_df = spark.read.format("delta").load("/bronze/orders")

silver_df = bronze_df \
    .filter(F.col("order_id").isNotNull()) \
    .withColumn("amount", F.col("amount").cast("double")) \

    .format("delta") \

Data Quality Checks

Data quality is the difference between a pipeline that runs and a pipeline that's trusted. Implement expectations at every layer.

Data Quality Framework
from dataclasses import dataclass
from typing import List, Dict
import json

@dataclass
class QualityResult:
    check_name: str
    passed: bool
    actual: float
    threshold: float
    message: str

def run_quality_checks(df, checks: List[Dict]) -> List[QualityResult]:
    results = []
    total = df.count()

    for check in checks:
        if check["type"] == "not_null":
            col = check["column"]
            null_count = df.filter(F.col(col).isNull()).count()
            null_pct = (null_count / total) * 100
            passed = null_pct <= check.get("max_null_pct", 0)
            results.append(QualityResult(
                f"not_null_{col}", passed, null_pct,
                check.get("max_null_pct", 0),
                f"{col}: {null_pct:.1f}% null"
            ))

        elif check["type"] == "row_count":
            passed = total >= check["min_rows"]
            results.append(QualityResult(
                "row_count", passed, total,
                check["min_rows"],
                f"Row count: {total}"
            ))

        elif check["type"] == "unique":
            col = check["column"]
            dup_count = total - df.dropDuplicates([col]).count()
            passed = dup_count == 0
            results.append(QualityResult(
                f"unique_{col}", passed, dup_count, 0,
                f"{col}: {dup_count} duplicates"
            ))

    failed = [r for r in results if not r.passed]
    if failed:
        raise ValueError(f"Quality checks failed: {[r.check_name for r in failed]}")
    return results

# Usage
checks = [
    {"type": "row_count", "min_rows": 100},
    {"type": "not_null", "column": "order_id", "max_null_pct": 0},
    {"type": "not_null", "column": "amount", "max_null_pct": 1.0},
    {"type": "unique", "column": "order_id"}
]
results = run_quality_checks(df, checks)
Delta Live Tables (DLT) Quality: In Databricks, use @expect and @expect_or_drop decorators on DLT tables - they enforce expectations declaratively and track violations in a quality dashboard automatically.

Pipeline Monitoring with Structured Logging
import logging
import json
from datetime import datetime

class PipelineMonitor:
    def __init__(self, pipeline_name: str):
        self.pipeline_name = pipeline_name
        self.metrics = {}
        self.start_time = datetime.utcnow()
        self.logger = logging.getLogger(pipeline_name)

    def log_metric(self, name: str, value, unit: str = ""):
        self.metrics[name] = {"value": value, "unit": unit, "ts": datetime.utcnow().isoformat()}
        self.logger.info(json.dumps({"metric": name, "value": value, "unit": unit}))

    def check_volume(self, df, expected_min: int, expected_max: int = None):
        count = df.count()
        self.log_metric("row_count", count, "rows")
        if count < expected_min:
            self.alert(f"LOW VOLUME: {count} rows (expected >= {expected_min})", severity="critical")
        elif expected_max and count > expected_max:
            self.alert(f"HIGH VOLUME: {count} rows (expected <= {expected_max})", severity="warning")
        return count

    def alert(self, message: str, severity: str = "warning"):
        alert_payload = {
            "pipeline": self.pipeline_name,
            "severity": severity,
            "message": message,
            "timestamp": datetime.utcnow().isoformat()
        }
        self.logger.error(json.dumps(alert_payload))
        # In production: send to PagerDuty, Teams, Slack webhook, etc.

    def finish(self):
        duration = (datetime.utcnow() - self.start_time).total_seconds()
        self.log_metric("duration_seconds", duration, "seconds")
        self.log_metric("status", "success")

# Usage
monitor = PipelineMonitor("daily_orders_etl")
count = monitor.check_volume(df, expected_min=1000, expected_max=100000)
monitor.finish()

ML Feature Store
Reusable feature pipeline for data science teams
Feature Engineering MLflow Databricks Point-in-Time
Problem: Without a feature store, every ML team re-derives the same features (customer LTV, 30-day spend, churn score) independently - wasting compute, causing inconsistency between training and serving.

Solution: Centralized feature tables with point-in-time correct lookups. Features computed once, reused by any model. Training uses historical feature values - no data leakage.
Building and Registering Features
from databricks.feature_store import FeatureStoreClient
fs = FeatureStoreClient()

# Compute customer features
def compute_customer_features(orders_df):
    w30 = Window.partitionBy("customer_id") \
        .orderBy(F.col("order_date").cast("long")) \
        .rangeBetween(-30 * 86400, 0)
    return orders_df.withColumn(
        "spend_30d", F.sum("amount").over(w30)
    ).withColumn(
        "order_count_30d", F.count("*").over(w30)
    ).withColumn(
        "days_since_last_order", F.datediff(F.current_date(), F.max("order_date").over(w30))
    ).groupBy("customer_id").agg(
        F.last("spend_30d").alias("spend_30d"),
        F.last("order_count_30d").alias("order_count_30d"),
        F.last("days_since_last_order").alias("days_since_last_order")
    )

features_df = compute_customer_features(orders)

# Register in Feature Store
fs.create_table(
    name="ecommerce.customer_features",
    primary_keys=["customer_id"],
    timestamp_keys=["feature_date"],  # for point-in-time lookup
    df=features_df
)

# Training: get features with point-in-time join
training_set = fs.create_training_set(
    labels_df,
    feature_lookups=[FeatureLookup("ecommerce.customer_features", "customer_id")],
    label="churned",
    timestamp_lookup_key="label_date"
)

Data Mesh Implementation
Domain-oriented decentralized data ownership
Data Mesh Domain-Oriented Data Products Self-Service
Core Data Mesh Principles:
  • Domain ownership: The orders team owns and publishes the orders data product - not a central DE team
  • Data as a product: Each domain team publishes SLA-backed, documented, discoverable data products
  • Self-serve platform: Infrastructure (Databricks, Unity Catalog) enables any domain to build their pipeline without help from platform team
  • Federated governance: Global policies (PII handling, retention) applied centrally; domain-specific rules owned by domains

Implementation with Unity Catalog: Each domain gets their own catalog (orders_domain, customers_domain, marketing_domain). Cross-domain reads go through published APIs (approved Delta Sharing tables). Unity Catalog tracks lineage across domains automatically.
