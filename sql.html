1 What is SQL

SQL (Structured Query Language) is the language used to communicate with relational databases. It lets you store, retrieve, update, and analyze data. Every data engineer, analyst, and scientist uses SQL every day. It is consistently ranked the most in-demand data skill worldwide.

2 Why It Exists

Before SQL, retrieving data required writing low-level navigation code that depended on physical storage layout. Edgar Codd's insight: users should describe what data they want, not how to find it. SQL is declarative: you state the goal, the database engine figures out the plan.

3 Real World Analogy
🍽️
SQL is like ordering at a restaurant

You do not go into the kitchen and cook your meal yourself (imperative). You tell the waiter "I want pasta with chicken, no onions" (declarative). The kitchen figures out how to prepare it. SQL works the same way: you describe the result you want, and the query optimizer figures out the fastest execution path.

SQL has evolved from a research project to the universal language of data engineering over 50 years.

What is SQL?

Beginner ⏱ ~25 min
1 Concept Introduction

SQL stands for Structured Query Language. It is the standard language for interacting with relational database management systems (RDBMS). SQL lets you: query data (SELECT), modify data (INSERT/UPDATE/DELETE), define schema (CREATE/ALTER/DROP), and control access (GRANT/REVOKE).

2 Why It Exists

Before SQL, each database vendor had its own proprietary query language. ANSI standardized SQL in 1986, so the same syntax works across PostgreSQL, MySQL, SQL Server, Snowflake, and BigQuery (with minor dialect differences). This portability is why SQL has outlasted every "SQL killer" technology for 50 years.

SQL works the same way: you describe the result, the database uses indexes and statistics to retrieve it efficiently.

Interview Questions
SQL Basics
Easy What does SQL stand for and what is it used for?▶
Answer: SQL stands for Structured Query Language. It is the standard language for interacting with relational databases. It is used to query data (SELECT), modify data (INSERT/UPDATE/DELETE), define database structure (CREATE/ALTER/DROP), and control access (GRANT/REVOKE). SQL is declarative: you describe the result you want and the database engine determines how to retrieve it efficiently.
Easy What is the difference between SQL and NoSQL?▶
Answer: SQL databases store data in structured tables with a fixed schema and use SQL to query. They enforce ACID transactions and are great for structured, relational data. NoSQL databases (MongoDB, Cassandra, DynamoDB) use flexible schemas and store data as documents, key-value pairs, or graphs. They trade some consistency for scalability and flexibility. SQL is still the dominant choice for analytics and reporting even in NoSQL-heavy shops.
Medium What is ACID and why does it matter?▶
Answer: ACID stands for Atomicity (a transaction completes fully or not at all), Consistency (data always moves from one valid state to another), Isolation (concurrent transactions do not interfere), and Durability (committed data survives crashes). SQL databases enforce ACID by default. This is critical for financial transactions, order systems, and any use case where partial writes would cause data corruption.

SELECT Statement

Beginner⏱ ~30 min
1 Concept Introduction

The SELECT statement is the most used command in SQL. It retrieves data from one or more tables. You can select all columns, specific columns, renamed columns (aliases), computed expressions, and distinct values.

2 Why It Exists

Tables can have dozens of columns and millions of rows. SELECT lets you project only the columns you care about, reducing data transfer and making results easier to work with. Column aliases make output readable for downstream consumers like dashboards or APIs.

7 Common Mistakes
Mistake

Using SELECT * in production pipelines. When the table schema changes (new columns added), queries silently return different data.

Fix

Always name the specific columns you need: SELECT id, name, salary FROM employees. This is explicit and pipeline-safe.

Mistake

Using DISTINCT carelessly on large tables. DISTINCT requires sorting or hashing the entire result set, which is expensive.

Fix

Only use DISTINCT when you genuinely need unique values. Often duplicates indicate a JOIN problem that should be fixed at the JOIN level instead.

9 Interview Questions
Interview Questions
SELECT
Easy What is the difference between SELECT * and SELECT column_name?▶
Answer: SELECT * returns all columns in the table. SELECT column_name returns only the specified columns. In production, always prefer naming columns explicitly: it makes queries self-documenting, prevents breaking changes when columns are added/removed, and often reduces data transfer (especially important in columnar warehouses like Snowflake where you pay per byte scanned).
Easy What does the AS keyword do in SQL?▶
Answer: AS creates an alias, renaming a column or table in the output. For columns: SELECT salary AS annual_pay renames the column header. For tables: FROM employees AS e creates a shorter reference. AS is optional in most databases (SELECT salary annual_pay also works) but including it makes intent clear. Aliases are useful for expressions, long column names, and disambiguating columns from joined tables.
Medium When would you use DISTINCT and what are its performance implications?▶
Answer: Use DISTINCT when you genuinely need unique combinations of the selected columns, such as finding all unique departments, or all unique (customer, product) pairs. Performance-wise, DISTINCT requires the database to sort or hash the entire result set to find duplicates, which is O(n log n). On large tables this is expensive. If you find yourself using DISTINCT to fix duplicate results from a JOIN, fix the JOIN condition instead. Also, COUNT(DISTINCT col) is significantly slower than COUNT(col).

WHERE Filtering

Beginner+⏱ ~35 min
1 Concept Introduction

The WHERE clause filters rows before they are returned. Only rows where the condition evaluates to TRUE are included. It is evaluated on every row in the table (unless an index allows early filtering).

2 Why It Exists

Without WHERE, every query returns every row. Tables in production can have billions of rows. WHERE is the primary tool for telling the database to return only the relevant subset, which is critical for both correctness and performance.

3 Real World Analogy
🕵️
WHERE is like a bouncer at the door

The bouncer checks each person (row) against the rules (condition). Only people who meet the criteria (TRUE) get in. Everyone else is turned away. The bouncer does not change the people — just decides who passes through.

9 Interview Questions
Interview Questions
WHERE
Easy Why can't you use = NULL in a WHERE clause?▶
Answer: NULL represents the absence of a value — it is unknown. Any comparison with NULL (NULL = NULL, NULL != NULL, NULL > 5) evaluates to NULL, not TRUE or FALSE. SQL's three-valued logic means WHERE col = NULL never returns any rows because the condition is always NULL (neither true nor false). Use IS NULL or IS NOT NULL instead.
Easy What is the difference between IN and BETWEEN?▶
Answer: IN checks if a value matches any item in a list: WHERE dept IN ('Engineering', 'Marketing'). It is equivalent to multiple OR conditions. BETWEEN checks if a value falls within a range (inclusive on both ends): WHERE salary BETWEEN 70000 AND 100000 is equivalent to WHERE salary >= 70000 AND salary <= 100000. Use IN for discrete values, BETWEEN for continuous ranges.
Medium Explain SQL's three-valued logic.▶
Answer: Standard SQL uses three-valued logic: TRUE, FALSE, and NULL (unknown). This affects WHERE clauses — only rows evaluating to TRUE pass through. A row with a NULL comparison evaluates to NULL (unknown) and is excluded, just like FALSE. This is why NOT (col = NULL) does not work as expected — NOT NULL is still NULL, not TRUE. It also affects aggregate functions: most aggregates ignore NULL values entirely.
Knowledge Check
Which is the correct SQL to find employees with no bonus assigned?
Correct! IS NULL is the only correct way to check for NULL in standard SQL. The = operator always returns NULL (not TRUE) when comparing to NULL, so = NULL never matches any rows. IS NULL is a special predicate designed specifically for NULL detection.

ORDER BY

Beginner⏱ ~25 min
1 Concept Introduction

ORDER BY sorts the result rows. By default it sorts ascending (ASC). Use DESC for descending. Without ORDER BY, rows are returned in no guaranteed order (the database can return them in whatever order is most efficient).

2 Why It Exists

Data in tables has no inherent order. When building reports ("top 10 customers"), leaderboards, or paginated UIs, you need predictable, sorted results. ORDER BY also unlocks window functions and TOP-N patterns essential in data engineering.

3 Real World Analogy
🏆
ORDER BY is like a leaderboard

All players exist in the database (the table), but you want to see them ranked by score from highest to lowest. ORDER BY score DESC does exactly that — reorders the rows for display without changing the underlying data.

7 Common Mistakes
Mistake

Relying on row order without ORDER BY. Without it, the database returns rows in any order it chooses and it can change between runs.

Fix

Always add ORDER BY when the order of results matters to your application or report.

Mistake

Using ORDER BY without LIMIT on a large table. Sorting 100M rows is expensive and slow if you only need the top 10.

Fix

Always pair ORDER BY with LIMIT when you only need the top-N rows. This allows the database to use a heap structure and stop after N results.

8 Interview Questions
Interview Questions
ORDER BY
Easy What is the default sort order for ORDER BY?▶
Answer: The default is ASC (ascending). For numbers: 1, 2, 3... For strings: A, B, C... For dates: oldest to newest. You must explicitly specify DESC for descending order. It is best practice to always write ASC or DESC explicitly for clarity, even though ASC is the default.
Easy How are NULLs sorted in ORDER BY?▶
Answer: NULL sort behavior varies by database. In PostgreSQL, NULLs sort LAST in ASC and FIRST in DESC by default. In MySQL, NULLs sort FIRST in ASC. You can control this explicitly with NULLS FIRST or NULLS LAST in PostgreSQL/Oracle: ORDER BY bonus ASC NULLS LAST. This is important in rankings where NULL bonuses should not appear at the top of the list.
Medium How do you implement pagination in SQL?▶
Answer: Use LIMIT and OFFSET: LIMIT 10 OFFSET 0 for page 1, LIMIT 10 OFFSET 10 for page 2, etc. Formula: OFFSET = (page_number - 1) * page_size. Important caveat: OFFSET-based pagination is inefficient on large datasets because the database still reads all skipped rows. For high-performance pagination on large tables, use keyset pagination: WHERE id > last_seen_id ORDER BY id LIMIT 10. This uses an index and is O(1) regardless of page depth.
Knowledge Check
Which query returns the 3 most recently hired employees (newest first)?
Correct! ORDER BY hire_date DESC sorts newest first (descending), and LIMIT 3 returns only the top 3. ASC would give the oldest. LIMIT must come after ORDER BY. The third option has invalid syntax (WHERE cannot sort). The fourth option has LIMIT before ORDER BY which is invalid in most databases.

Aggregation Functions

Intermediate⏱ ~30 min
1 Concept Introduction

Aggregate functions compute a single value from a group of rows. They collapse many rows into one summary number. The most common are COUNT, SUM, AVG, MIN, MAX, and ROUND.

2 Why They Exist

Raw data has thousands of rows. Business users need summaries: total revenue, average order value, max response time. Aggregate functions compute these summaries directly in the database, far more efficiently than downloading all rows to your application and computing there.

3 Real World Analogy
🧮
Aggregates are like a calculator on a column

Imagine a column of salary numbers written on paper. SUM adds them all up. AVG divides the sum by count. MIN picks the smallest. MAX picks the largest. COUNT counts how many there are. The individual rows disappear into a single answer — just like pressing = on a calculator gives you one result.

⚠️
NULL Behavior in Aggregates
COUNT(*) counts ALL rows including NULLs. COUNT(col) counts only non-NULL values in that column. SUM, AVG, MIN, MAX all IGNORE NULL values. This means AVG(salary) on 5 rows where 1 salary is NULL will compute the average of only 4 values — which may not be what you want. Use COALESCE to handle NULLs before aggregating: AVG(COALESCE(salary, 0)).
7 Common Mistakes
Mistake

Using COUNT(col) when you mean COUNT(*). If col has NULLs, COUNT(col) returns fewer rows than you expect.

Fix

Use COUNT(*) to count rows. Use COUNT(col) only when you specifically want to count non-NULL values in that column.

Mistake

Forgetting that AVG ignores NULLs, making the denominator smaller and the average appear higher than the true average.

Fix

Use AVG(COALESCE(col, 0)) to treat NULL as zero, or SUM(col) / COUNT(*) to explicitly control the denominator.

8 Interview Questions
Interview Questions
Aggregates
Easy What is the difference between COUNT(*) and COUNT(column)?▶
Answer: COUNT(*) counts all rows regardless of NULL values. COUNT(column) counts only rows where that specific column is NOT NULL. Example: if you have 100 employees but 10 have no bonus set (NULL bonus), COUNT(*) = 100 but COUNT(bonus) = 90. Use COUNT(*) to count rows, COUNT(col) to count how many rows have a value in a column.
Medium How does AVG handle NULL values?▶
Answer: AVG ignores NULL values in both the sum and the count. If 10 rows have salary values and 5 have NULL salary, AVG(salary) computes the average of the 10 non-NULL values only — dividing their sum by 10, not 15. This can be misleading. To include NULLs as 0 in the average: AVG(COALESCE(salary, 0)) or SUM(COALESCE(salary, 0)) / COUNT(*).
Medium How would you compute the median salary in SQL?▶
Answer: SQL does not have a built-in MEDIAN function in most databases. In PostgreSQL: SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) FROM employees. In Snowflake: MEDIAN(salary). In MySQL/older systems: use a subquery with ROW_NUMBER to find the middle row(s). The PERCENTILE_CONT function is part of the SQL:2003 standard and available in PostgreSQL, Snowflake, and BigQuery.
Knowledge Check
A table has 10 employees. 3 have NULL salary. What does COUNT(*) return vs COUNT(salary)?
Correct! COUNT(*) = 10 because it counts all rows regardless of NULLs. COUNT(salary) = 7 because it only counts rows where salary is NOT NULL (3 rows have NULL salary, so 10 - 3 = 7). This is one of the most common interview tricks with aggregate functions.

GROUP BY

Group rows by shared values and compute aggregates per group. The engine of every summary report and KPI dashboard.

⏱️ ~35 min🟡 Intermediate

GROUP BY

Intermediate⏱ ~35 min
1 Concept Introduction

GROUP BY collapses multiple rows that share the same value in a column into a single summary row. Aggregate functions then compute values for each group separately. The result has one row per unique group.

2 Why It Exists

A single AVG(salary) gives you the company-wide average. But what if you need the average per department? GROUP BY dept tells SQL to split the data into groups first, then run AVG on each group independently.

3 Real World Analogy
📚
GROUP BY is like sorting papers into piles

Imagine 100 student papers spread on a table. You sort them into piles: one pile per class. Then you count each pile (COUNT), compute the average grade per pile (AVG). GROUP BY does this automatically: it sorts rows into groups by shared column value, then aggregates each group.

7 Common Mistakes
Mistake

Selecting a column that is not in GROUP BY and not inside an aggregate. This causes a syntax error in PostgreSQL/Snowflake (though MySQL allows it with unpredictable results).

Fix

Every column in SELECT must either be in GROUP BY or wrapped in an aggregate function (COUNT, SUM, AVG, MIN, MAX).

Mistake

Trying to filter on aggregate values using WHERE: WHERE COUNT(*) > 5 — this causes a syntax error.

Fix

Use HAVING to filter on aggregates: GROUP BY dept HAVING COUNT(*) > 5. WHERE runs before grouping; HAVING runs after.

8 Interview Questions
Interview Questions
GROUP BY
Easy What rule governs which columns can appear in a SELECT with GROUP BY?▶
Answer: Every column in SELECT must either appear in the GROUP BY clause OR be wrapped in an aggregate function (COUNT, SUM, AVG, MIN, MAX). This rule exists because after grouping, multiple rows collapse into one — so any non-grouped, non-aggregated column would have multiple possible values, making the result ambiguous. MySQL has a non-standard mode that allows this, but the result is undefined (picks an arbitrary value from the group).
Medium What is the execution order of SELECT, FROM, WHERE, GROUP BY, HAVING, ORDER BY?▶
Answer: The logical execution order is: (1) FROM — identify the source table, (2) WHERE — filter individual rows, (3) GROUP BY — group remaining rows, (4) HAVING — filter groups, (5) SELECT — compute output columns and aliases, (6) ORDER BY — sort results, (7) LIMIT — restrict row count. This is why you cannot use a SELECT alias in WHERE or HAVING (those run before SELECT). You CAN use aliases in ORDER BY in most databases.
Medium How do you count distinct values per group?▶
Answer: Use COUNT(DISTINCT col): SELECT dept, COUNT(DISTINCT manager_id) AS num_managers FROM employees GROUP BY dept. This counts unique manager IDs per department. COUNT(DISTINCT) is significantly slower than COUNT() because it requires deduplication within each group. On large tables, consider approximate count functions like HLL_COUNT.DISTINCT in BigQuery or APPROX_COUNT_DISTINCT in Snowflake if exact precision is not required.
Knowledge Check
SELECT dept, name, COUNT(*) FROM employees GROUP BY dept — what is wrong?
Correct! The column name is in SELECT but not in GROUP BY and not inside an aggregate. After grouping by dept, each group could have multiple names — which one should be returned? PostgreSQL throws an error. MySQL may pick an arbitrary name. Fix: either add name to GROUP BY, or use MIN(name)/MAX(name) to aggregate it.

L1.7 - Aggregation 🔬

HAVING

Filter groups after aggregation. Understand the critical difference between WHERE (pre-group filter) and HAVING (post-group filter).

⏱️ ~25 min🟡 Intermediate

HAVING

Intermediate⏱ ~25 min
1 Concept Introduction

HAVING filters groups after aggregation. It is like a WHERE clause but for groups rather than individual rows. Use it to keep only groups that meet a condition based on an aggregate value.

2 Why It Exists

WHERE filters rows before grouping, so it cannot reference aggregate functions (COUNT, SUM, AVG) because those values do not exist yet. HAVING was added to SQL specifically to filter after grouping, when aggregate values are available.

3 Real World Analogy
🏭
HAVING is like inspecting boxes after packing

WHERE checks individual items before packing them into boxes (filter rows). HAVING checks the full boxes after packing (filter groups). "Only keep boxes with more than 10 items inside" is a HAVING condition — you can only count items after the box is packed.

7 Common Mistakes
Mistake

Using HAVING without GROUP BY. HAVING without GROUP BY treats the entire table as one group, which is valid but unusual.

Fix

In most use cases, HAVING pairs with GROUP BY. When you see HAVING without GROUP BY, ask yourself if the logic makes sense as a WHERE clause instead.

Mistake

Using a SELECT alias in HAVING: SELECT AVG(salary) AS avg_sal ... HAVING avg_sal > 80000 — this fails in PostgreSQL (aliases not available in HAVING).

Fix

8 Interview Questions
Interview Questions
HAVING
Easy What is the difference between WHERE and HAVING?▶
Answer: WHERE filters individual rows BEFORE grouping. HAVING filters groups AFTER aggregation. You cannot use aggregate functions in WHERE. You can use aggregate functions in HAVING. Mnemonic: WHERE selects rows that go INTO the groups; HAVING selects which groups come OUT.
Medium Can you use HAVING without GROUP BY?▶
Answer: Yes, HAVING without GROUP BY is valid. The entire table is treated as a single group. For example: SELECT COUNT(*) FROM orders HAVING COUNT(*) > 1000 — this returns the count only if there are more than 1000 orders, otherwise returns no rows. In practice, this pattern is rare; usually you would use a subquery or WHERE EXISTS instead for readability.
Medium When should you use WHERE vs HAVING for performance?▶
Answer: Always filter rows with WHERE when possible, before grouping. This reduces the number of rows that need to be grouped and aggregated. WHERE can use indexes; HAVING cannot filter before the full-table scan is done. Example: WHERE dept = 'Engineering' reduces rows from 1 million to 50,000 before grouping — HAVING would group all 1 million rows first, then discard most of them. Rule: push filters as early as possible in the query.
Knowledge Check
Which clause filters groups that have more than 5 employees?
Correct! HAVING COUNT(*) > 5 filters groups after GROUP BY. You cannot use aggregate functions like COUNT() in a WHERE clause — it causes a syntax error because WHERE runs before grouping, when no aggregate values exist yet. FILTER is not a top-level SQL clause (it is used differently in window functions).

L1.8 - Joins 🔗

INNER JOIN

Combine rows from two tables where a matching key exists in both. The most commonly used join type in production SQL.

⏱️ ~35 min🟡 Intermediate

INNER JOIN

Intermediate⏱ ~35 min
1 Concept Introduction

INNER JOIN returns only rows where the join condition is true in both tables. Rows with no matching key in the other table are excluded from the result. It is the intersection of the two tables on the specified key.

2 Why It Exists

Relational databases store data in separate tables to avoid duplication (normalization). An orders table has customer_id but not customer names. A customers table has names. INNER JOIN lets you combine them at query time to get the full picture without storing redundant data.

3 Real World Analogy
🪪
INNER JOIN is a guest list intersection

Imagine two lists: employees registered for a conference (List A) and employees who paid their fee (List B). INNER JOIN returns only people on BOTH lists. Anyone who registered but did not pay (or paid but is not registered) is excluded. Only the confirmed attendees.

7 Common Mistakes
Mistake

Forgetting the ON clause produces a Cartesian product: FROM customers, orders returns every customer paired with every order (N x M rows). With large tables this can produce billions of rows.

Fix

Always specify the JOIN condition: INNER JOIN orders ON customers.id = orders.customer_id. If you need a cross join intentionally, use CROSS JOIN to be explicit.

Mistake

Not using table aliases with duplicate column names. SELECT id, name... when both tables have id causes ambiguity errors.

Fix

Always alias tables (c for customers, o for orders) and qualify every column: c.id, o.id, c.name. This also makes queries easier to read.

8 Interview Questions
Interview Questions
INNER JOIN
Easy What rows does INNER JOIN return?▶
Answer: INNER JOIN returns only rows where the join condition evaluates to TRUE in both tables. If a row in the left table has no match in the right table (or vice versa), it is excluded from the result. The result set is the intersection of the two tables based on the specified key. Using just JOIN (without INNER) is equivalent — INNER is the default join type.
Medium What is a Cartesian product and when does it happen?▶
Answer: A Cartesian product (cross join) pairs every row from table A with every row from table B. If A has 1,000 rows and B has 1,000 rows, the result has 1,000,000 rows. It happens when: (1) you list multiple tables in FROM without a JOIN condition, (2) you use CROSS JOIN explicitly, or (3) your JOIN condition always evaluates to TRUE. In production, an accidental Cartesian join can cause queries to run for hours or crash the database. Always verify your JOIN conditions are correct and selective.
Hard How does the query optimizer decide the join order for multiple table joins?▶
Answer: The optimizer uses statistics (row counts, cardinality, distinct values, data distribution histograms) to estimate the cost of different join orders and pick the cheapest plan. It considers: (1) start with the smallest filtered table, (2) join in order of result size reduction, (3) use available indexes. For N tables, there are N! possible join orders — the optimizer uses heuristics and dynamic programming to avoid evaluating all possibilities. You can influence join order with hints or by rewriting the query. Use EXPLAIN to see the chosen plan.
Knowledge Check
Table A has 3 rows. Table B has 5 rows. INNER JOIN on A.id = B.a_id returns 2 rows. How many rows had no match?
Correct! The result has 2 rows. Since A has 3 rows and only 2 matched, at least 1 row in A had no match. Since B has 5 rows and only 2 appear in the result, at least 3 rows in B had no match. INNER JOIN silently drops all non-matching rows. To see unmatched rows, use LEFT JOIN or FULL OUTER JOIN.

L1.9 - Joins ⬅️

LEFT JOIN

Return all rows from the left table plus matching rows from the right. Unmatched right-side values become NULL. The join you will use most often.

⏱️ ~30 min🟡 Intermediate

LEFT JOIN

Intermediate⏱ ~30 min
1 Concept Introduction

LEFT JOIN (also called LEFT OUTER JOIN) returns every row from the left (first) table, plus matched rows from the right table. When there is no match in the right table, the right-side columns are filled with NULL. No left-side row is ever dropped.

2 Why It Exists

LEFT JOIN is for "keep all left rows even if there is no match." Classic use case: list all customers with their most recent order. Customers with no orders still appear in the list (with NULL for order columns), unlike INNER JOIN which would drop them silently.

3 Real World Analogy
📋
LEFT JOIN is a class roll call

The teacher calls every student on the roll (left table). For each student, she checks if there is a grade in the gradebook (right table). If there is a grade, she records it. If the student has no grade yet, she writes "not submitted" (NULL). Every student is listed regardless. No student is dropped because they are missing a grade.

7 Common Mistakes
Mistake

Putting a filter for the right table in WHERE instead of ON. WHERE o.status = 'active' applied after LEFT JOIN converts it back to an INNER JOIN (NULL rows are filtered out by the WHERE condition).

Fix

Move right-table conditions into the ON clause: LEFT JOIN orders o ON c.id = o.customer_id AND o.status = 'active'. Now customers with no active orders still appear (with NULLs) rather than being dropped.

Mistake

Using COUNT(*) instead of COUNT(col) after LEFT JOIN. COUNT(*) counts NULL rows as 1, giving the wrong count for unmatched left-side rows.

Fix

Use COUNT(o.order_id) or any non-nullable right-table column. COUNT(col) skips NULLs, so unmatched rows correctly count as 0.

8 Interview Questions
Interview Questions
LEFT JOIN
Easy How do you find all records in table A with no match in table B?▶
Answer: Use a LEFT JOIN and filter for NULL on a non-nullable right-table column: SELECT a.* FROM a LEFT JOIN b ON a.id = b.a_id WHERE b.id IS NULL. After the LEFT JOIN, rows with no match in B get NULL for all B columns. Filtering WHERE b.id IS NULL isolates those unmatched rows. This anti-join pattern is more readable than NOT IN (which has NULL traps) and often faster than NOT EXISTS in practice.
Medium Why can a WHERE clause on the right table cancel a LEFT JOIN?▶
Answer: After LEFT JOIN, unmatched left rows have NULL for all right-side columns. A WHERE clause like WHERE right.col = 'X' evaluates NULL = 'X' as UNKNOWN (not TRUE), so those rows are filtered out — exactly what INNER JOIN does. To filter right-table rows without turning it into an inner join, put the condition in the ON clause instead of WHERE: LEFT JOIN orders o ON c.id = o.customer_id AND o.status = 'active'. Now non-active orders produce NULL rather than causing the customer row to disappear.
Hard What is LATERAL JOIN and when would you use it?▶
Answer: LATERAL allows a subquery in the FROM clause to reference columns from tables listed earlier in the same FROM. It is like a correlated subquery in the FROM position. Use case: for each customer, get the most recent order -- LEFT JOIN LATERAL (SELECT * FROM orders WHERE orders.customer_id = c.id ORDER BY order_date DESC LIMIT 1) o ON true. Without LATERAL, a subquery cannot reference c.id. Supported in PostgreSQL, Snowflake, BigQuery. Useful for per-row operations that need a subset of rows.
Knowledge Check
You write: SELECT c.name, o.amount FROM customers c LEFT JOIN orders o ON c.id = o.customer_id WHERE o.status = 'shipped'. A customer with no orders -- does she appear?
Correct! The WHERE o.status = 'shipped' filters after the LEFT JOIN. The customer with no orders gets NULL for o.status. Then NULL = 'shipped' is UNKNOWN, which is treated as FALSE -- so that row is dropped. The WHERE clause effectively converts the LEFT JOIN back into an INNER JOIN. Fix: move the condition to ON: LEFT JOIN orders o ON c.id = o.customer_id AND o.status = 'shipped'.

RIGHT JOIN

Mirror of LEFT JOIN. Returns all rows from the right table plus matched rows from the left. Rarely needed -- most queries rewrite it as a LEFT JOIN.

⏱️ ~20 min🟡 Intermediate

RIGHT JOIN

Intermediate⏱ ~20 min
1 Concept Introduction

RIGHT JOIN (RIGHT OUTER JOIN) is the mirror image of LEFT JOIN. It returns every row from the right (second) table plus matched rows from the left table. When there is no match in the left table, left-side columns are NULL.

2 Why It Exists

RIGHT JOIN exists for completeness and symmetry in the SQL standard. In practice, most developers rewrite RIGHT JOINs as LEFT JOINs by swapping the table order -- it is easier to reason about queries when all the "all rows" tables are on the left side.

3 Real World Analogy
🗄️
RIGHT JOIN is the same pantry check, from the other direction

You have a recipe list (right table) and a pantry (left table). RIGHT JOIN shows every recipe, with pantry items filled in where you have them. Recipes with no matching pantry item still appear (with NULL for pantry). You could get the same result by doing a LEFT JOIN with recipes on the left.

L1.11 - Joins 🔄

FULL OUTER JOIN

Return ALL rows from both tables. Rows with no match get NULLs on the missing side. Perfect for data reconciliation and finding gaps between two datasets.

⏱️ ~25 min🟡 Intermediate

FULL OUTER JOIN

Intermediate⏱ ~25 min
1 Concept Introduction

FULL OUTER JOIN returns every row from both tables. Matched rows are combined as in INNER JOIN. Unmatched rows from the left table have NULLs on the right side. Unmatched rows from the right table have NULLs on the left side. No row from either table is lost.

2 Why It Exists

FULL OUTER JOIN is the union of LEFT JOIN and RIGHT JOIN results. It is essential for data reconciliation: "show me everything from both systems, and flag where they diverge." Use it to find rows that exist in A but not B, rows in B but not A, and rows in both.

3 Real World Analogy
📊
FULL OUTER JOIN is two team rosters merged

Team A and Team B both submit player rosters. Some players are on both (matches). Some are only on Team A's list (left-only). Some are only on Team B's list (right-only). FULL OUTER JOIN shows the complete merged list with notes for who is missing from which side.

7 Common Mistakes
Mistake

Using FULL OUTER JOIN in MySQL -- it is not supported and causes a syntax error.

Fix

Emulate it with UNION of LEFT JOIN and RIGHT JOIN as shown above. In PostgreSQL, Snowflake, BigQuery, and SQL Server, FULL OUTER JOIN is natively supported.

Mistake

Not using COALESCE to get the ID. When both sides can be NULL, selecting a.id alone misses rows where only b exists.

Fix

Interview Questions
FULL OUTER JOIN
Easy When would you use FULL OUTER JOIN over LEFT or INNER JOIN?▶
Answer: Use FULL OUTER JOIN when you need to see ALL rows from both tables, regardless of matches. INNER JOIN drops non-matching rows entirely. LEFT JOIN preserves only left-side unmatched rows. FULL OUTER JOIN preserves both.

UNION removes duplicate rows (rows that match from both sides appear twice -- once in each SELECT). If you want all duplicates, use UNION ALL and then deduplicate with a CTE or DISTINCT. This approach works in any database.

Table A has 4 rows. Table B has 3 rows. 2 rows match. How many rows does FULL OUTER JOIN return?
Correct! A has 4 rows: 2 match + 2 don't. B has 3 rows: 2 match + 1 doesn't. FULL OUTER JOIN returns: 2 matched rows + 2 A-only rows (with NULL B columns) + 1 B-only row (with NULL A columns) = 5 rows total. Note: 7 would be the Cartesian product. INNER JOIN returns 2. LEFT JOIN returns 4.

Window Functions

Compute values across a sliding window of rows without collapsing them. The secret weapon of every senior SQL analyst: rankings, running totals, moving averages, lag/lead comparisons.

Window Functions

Advanced⏱ ~45 min
1 Concept Introduction

A window function performs a calculation across a set of rows related to the current row (the "window"), without collapsing those rows into one like GROUP BY does. Each input row keeps its own output row, but gains a new computed column based on the window definition.

2 Why It Exists

Before window functions, ranking or running totals required complex self-joins or correlated subqueries. Window functions make these computations declarative, readable, and far more performant. They are indispensable for analytics SQL in modern warehouses.

For each seat (row), you can compute something based on that view -- but you do NOT merge all seats into one. Every passenger keeps their own seat while gaining context from nearby stations.

Mistake

Using a window function alias in WHERE or HAVING. Window functions are computed after WHERE and GROUP BY, so their aliases are not available there.

Fix

Mistake

Confusing ROWS BETWEEN and RANGE BETWEEN. RANGE treats equal ORDER BY values as the same position; ROWS treats each physical row independently.

Fix

8 Interview Questions
Interview Questions
Window Functions
Easy What is the difference between ROW_NUMBER, RANK, and DENSE_RANK?▶
Answer: All three assign numbers to rows ordered by a column. They differ on ties: ROW_NUMBER gives unique sequential numbers regardless of ties (1,2,3,4). RANK skips numbers after ties: two rows tied for rank 2 both get 2, and the next row gets 4 (1,2,2,4). DENSE_RANK never skips: two tied rows get 2 and the next gets 3 (1,2,2,3). Use ROW_NUMBER when you need a unique ID. Use RANK when you want "competition ranking." Use DENSE_RANK when you want no gaps in rank numbers.
Medium How do you select the top 1 row per group using window functions?▶
Answer: Use ROW_NUMBER() with PARTITION BY: WITH ranked AS (SELECT *, ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn FROM employees) SELECT * FROM ranked WHERE rn = 1. This selects the highest-paid employee per department. If you want ties included, use RANK() instead and filter WHERE rn = 1 -- but note RANK can return multiple rows per partition when there are salary ties. For exactly one row per partition, use ROW_NUMBER().
Hard What is the PARTITION BY clause and how does it differ from GROUP BY?▶
Answer: PARTITION BY divides rows into groups for window function calculations, similar to GROUP BY -- but crucially, rows are NOT collapsed. Each row still appears in the output. The window function is computed independently within each partition. GROUP BY collapses rows into one per group; PARTITION BY does not. Example: GROUP BY dept with AVG(salary) gives one row per dept. PARTITION BY dept with AVG(salary) gives one row per employee, each with their department's average salary alongside their individual salary -- enabling instant comparison of individual vs group.
Knowledge Check
Three employees in Engineering all earn $90,000. What does RANK() OVER (PARTITION BY dept ORDER BY salary DESC) return for them?
Correct! RANK() gives tied rows the same rank and then skips the intermediate numbers. Three employees tied for first get rank 1, 1, 1 -- and the next-lower salary employee would get rank 4. DENSE_RANK would give 1,1,1 and then 2 (no gap). ROW_NUMBER would give 1,2,3 arbitrarily.

Subqueries

A query nested inside another query. Use subqueries to filter, compute, or generate data within a larger SQL statement. Three types: scalar, row, and table subqueries.

⏱️ ~35 min🔴 Advanced

Subqueries

Advanced⏱ ~35 min
1 Concept Introduction

A subquery is a SELECT statement embedded inside another SQL statement. It can appear in SELECT (scalar subquery), FROM (derived table / inline view), WHERE (filter subquery), or HAVING. The inner query runs first and its result is used by the outer query.

2 Why It Exists

Subqueries let you break complex problems into pieces. "Find employees who earn more than the company average" -- the average is computed by an inner query, and the outer query filters against it. Before CTEs existed, subqueries were the only way to write multi-step logic in a single SQL statement.

3 Real World Analogy
🗃️
Subqueries are questions within questions

"Who earns above average?" requires two questions: (1) What is the average salary? (2) Who earns more than that number? A subquery computes (1) and passes the result to (2). It is like asking your accountant for last year's average before deciding who gets a bonus -- you ask the inner question first, then use the answer to answer the outer question.

7 Common Mistakes
Mistake

Correlated subqueries inside loops (e.g., WHERE col IN (SELECT ... WHERE outer.col = inner.col)) execute once per outer row, making them O(N) expensive on large tables.

Fix

Rewrite correlated subqueries as JOINs or CTEs when possible. JOINs are optimized by the planner; correlated subqueries often force row-by-row execution.

Mistake

NOT IN with a subquery that could return NULLs. NOT IN returns no rows if the subquery returns any NULL value, due to three-valued logic.

Fix

Use NOT EXISTS instead of NOT IN when the subquery might return NULLs. NOT EXISTS correctly handles NULLs and is often faster.

8 Interview Questions
Interview Questions
Subqueries
Easy What is the difference between a correlated and a non-correlated subquery?▶
Answer: A non-correlated (independent) subquery executes once and its result is used by the outer query. Example: WHERE salary > (SELECT AVG(salary) FROM employees) -- the inner AVG runs once. A correlated subquery references columns from the outer query, so it re-executes for every row the outer query processes. Example: WHERE salary > (SELECT AVG(salary) FROM employees e2 WHERE e2.dept = e1.dept) -- the inner query runs once per employee in the outer query. Correlated subqueries are more powerful but can be slow at scale.
Medium Why does NOT IN fail when the subquery returns NULLs?▶
Answer: IN checks if a value equals any member of the list using =. NOT IN is equivalent to != for every member. If any member is NULL, then value != NULL evaluates to UNKNOWN, not FALSE. When NOT IN encounters ANY NULL in its list, the whole expression becomes UNKNOWN and SQL returns no rows (UNKNOWN is treated as FALSE in WHERE). Example: WHERE id NOT IN (1, 2, NULL) returns zero rows even if id=3, because 3 != NULL is UNKNOWN. Fix: use NOT EXISTS, which is NULL-safe, or add WHERE col IS NOT NULL to the subquery.
Knowledge Check
A correlated subquery references the outer query's columns. What is the main performance concern?
Correct! A correlated subquery runs once per outer row. For a table with 1 million rows, the inner subquery executes 1 million times. This results in O(N*M) complexity. The fix is often to rewrite as a JOIN (which the optimizer can execute more efficiently) or as a window function (for per-partition aggregates). Always check EXPLAIN output when using correlated subqueries on large tables.

CTEs (WITH Clause)

Common Table Expressions let you name and reuse subqueries. Write complex multi-step logic that reads like a recipe rather than a maze of nested parentheses.

⏱️ ~35 min🔴 Advanced

CTEs (WITH Clause)

Advanced⏱ ~35 min
1 Concept Introduction

A Common Table Expression (CTE) is a named temporary result set defined with the WITH keyword at the top of a query. It is like a named subquery you can reference by name later in the same statement. Multiple CTEs can be chained together to build multi-step logic step by step.

2 Why It Exists

CTEs make complex SQL readable by breaking it into named, self-contained steps. Instead of deeply nested subqueries, you write: first compute department averages (step 1 CTE), then rank employees vs their department average (step 2 CTE), then filter to top performers (final SELECT). Each step has a name and can be reasoned about independently.

3 Real World Analogy
🍳
CTEs are recipe steps written out before the main instruction

Instead of saying "take the result of cooking the sauce you got from simmering the reduction of the broth from step 1," you write: (1) Make broth. (2) Reduce it to a sauce. (3) Use the sauce. Each step has a name. CTEs do the same -- name each intermediate result so the final query reads cleanly: "FROM dept_averages JOIN top_earners ON ..."

4 Interactive Animation
Building Blocks Stack: Each CTE Becomes a Named Layer
Step 1 CTE: dept_avgs
SELECT dept, AVG(salary) AS avg_sal FROM employees GROUP BY dept
Step 2 CTE: ranked_emps
SELECT e.*, d.avg_sal, RANK() OVER (PARTITION BY e.dept ORDER BY salary DESC) AS rnk FROM employees e JOIN dept_avgs d ON e.dept=d.dept
Final SELECT
SELECT name, dept, salary FROM ranked_emps WHERE rnk = 1
Each CTE is a named building block. The final SELECT uses the clean CTE names instead of nested subquery noise.
5 Syntax
SQL

dept_avgs AS (                             -- Step 1
    SELECT dept, AVG(salary) AS avg_sal
    FROM   employees
    GROUP BY dept
),
ranked_emps AS (                           -- Step 2 (references Step 1)
    SELECT
        e.*,
        d.avg_sal,
        RANK() OVER (PARTITION BY e.dept ORDER BY salary DESC) AS rnk

    JOIN   dept_avgs   d ON e.dept = d.dept
)
SELECT name, dept, salary, avg_sal         -- Final SELECT
FROM   ranked_emps
WHERE  rnk = 1;                            -- top earner per dept
6 Recursive CTEs
SQL - Recursive CTE: manager hierarchy
WITH RECURSIVE org_chart AS (
    -- Anchor: start with the CEO (no manager)
    SELECT id, name, manager_id, 0 AS level
    FROM   employees
    WHERE  manager_id IS NULL

    UNION ALL

    -- Recursive step: join each employee to their manager
    SELECT e.id, e.name, e.manager_id, oc.level + 1
    FROM   employees e
    JOIN   org_chart  oc ON e.manager_id = oc.id
)
SELECT REPEAT('  ', level) || name AS indented_name, level
FROM   org_chart
ORDER BY level, name;
7 Common Mistakes
Mistake

Treating CTEs as materialized (cached) views. In most databases (PostgreSQL, BigQuery, Snowflake), CTEs are NOT materialized by default -- they are inlined at the point of use, meaning if referenced twice, they execute twice.

Fix

In PostgreSQL 12+, use the MATERIALIZED keyword: WITH expensive_cte AS MATERIALIZED (...). In BigQuery/Snowflake, consider temp tables or subquery factoring if reuse is important. Always check with EXPLAIN.

Mistake

Recursive CTEs without a termination condition cause infinite loops.

Fix

Always ensure the recursive part has a reducing condition (like level < 10 or a DISTINCT check) and verify your data has no circular references before running.

8 Interview Questions
Interview Questions
CTEs
Easy What is the difference between a CTE and a subquery?▶
Answer: Both define a temporary result set. The main differences are: (1) Readability -- CTEs are defined at the top with a name and referenced cleanly; subqueries are embedded inline and nested in parentheses. (2) Reusability -- a CTE can be referenced multiple times in the same query by name; a subquery must be copy-pasted if needed twice. (3) Recursion -- only CTEs support recursive queries (WITH RECURSIVE). Performance-wise they are often equivalent, as CTEs are inlined in most databases (not cached).
Medium What is a recursive CTE and what kind of data does it work well with?▶
Answer: A recursive CTE has two parts: an anchor member (base case, runs once) and a recursive member (references the CTE itself, runs repeatedly until no new rows are produced). It works perfectly for hierarchical or graph data: organizational charts (employee-manager), file system trees, bill of materials (product components), graph traversal (who is connected to whom). Without recursive CTEs, this data requires multiple self-joins or application-level looping. Supported in PostgreSQL, SQL Server, MySQL 8+, Snowflake, BigQuery.
Knowledge Check
In most databases (PostgreSQL, Snowflake), if you reference the same CTE twice in a query, what happens?
Correct! By default, CTEs in PostgreSQL, Snowflake, BigQuery, and most modern databases are NOT materialized -- they are inlined (expanded in place) each time they are referenced. If you reference an expensive CTE twice, it runs twice. Use WITH expensive AS MATERIALIZED (...) in PostgreSQL 12+ to force caching. In Snowflake, consider creating a temporary table for heavy reuse.

Intermediate⏱ ~30 min
1 Concept Introduction

CASE WHEN is SQL's conditional expression. It evaluates conditions in order and returns the first matching THEN value, falling through to ELSE if none match. It can appear in SELECT, WHERE, ORDER BY, GROUP BY, and aggregate functions. It is an expression, not a statement -- it returns a value.

7 Common Mistakes
Mistake

Not including ELSE in a CASE expression. When no WHEN condition matches and there is no ELSE, the expression returns NULL silently.

Fix

Always add an ELSE clause with a meaningful default (like 'Unknown' or 0) to avoid unexpected NULLs in your output.

Mistake

Overlapping conditions in the wrong order. If WHEN salary > 50000 comes before WHEN salary > 100000, employees earning over 100k get labeled "Junior" because the first matching condition wins.

Fix

Order WHEN clauses from most specific/restrictive to least specific. Put the highest threshold first.

Easy What does CASE WHEN return if no WHEN matches and there is no ELSE?▶
Answer: NULL. If no WHEN condition evaluates to TRUE and no ELSE clause is provided, the CASE expression returns NULL. This is a common source of unexpected NULLs in reports. Indexes

Speed up queries by orders of magnitude. Understand how B-tree indexes work, when to add them, and when they hurt more than they help.

⏱️ ~40 min🔴 Advanced

Indexes

Advanced⏱ ~40 min
1 Concept Introduction

An index is a separate data structure (usually a B-tree) that stores a sorted copy of a column's values along with pointers to the original rows. When a query filters or joins on an indexed column, the database jumps directly to matching rows instead of scanning every row in the table.

2 Why It Exists

Without indexes, every query does a full sequential scan: reading every row to find matches. On a 100 million row table this takes seconds or minutes. With an index on the filter column, the database traverses a B-tree in O(log N) steps, finding matches in milliseconds.

3 Real World Analogy
📖
Indexes are like a book's index at the back

To find "database normalization" in a textbook without an index, you read every page. With an index, you look up the term alphabetically, see "page 247," and jump there in two steps. A database index does the same: sorted lookup structure pointing to the row location on disk.

6 When Indexes Help vs Hurt
Indexes HELP when:
  • WHERE clause filters on the indexed column (high selectivity)
  • JOIN ON uses the indexed column
  • ORDER BY on the indexed column (avoids sort step)
  • Table is large and queries are read-heavy
Indexes HURT when:
  • Table is small (full scan is faster than index lookup + row fetch)
  • Column has very low selectivity (e.g., boolean true/false)
  • Table has heavy write workload (every INSERT/UPDATE/DELETE must update indexes)
  • Too many indexes -- each one slows writes and consumes disk space
7 Interview Questions
Interview Questions
Indexes
Easy What is a B-tree index and how does it speed up queries?▶
Answer: A B-tree (balanced tree) index is the default index type in PostgreSQL, MySQL, and most databases. It stores column values in sorted order in a tree structure where each node has multiple children. A search starts at the root, compares the target value to node values, and navigates left or right until reaching the leaf node with the row pointer. This traversal is O(log N) -- for 1 million rows, only ~20 comparisons are needed. The B-tree stays balanced through splits and merges, ensuring consistent performance regardless of insertion order.
Medium What is column order significance in a composite index?▶
Answer: A composite index on (dept, salary) can be used by queries that filter on dept, or on both dept AND salary. It CANNOT efficiently serve a query that filters ONLY on salary (the second column) because the index is sorted by dept first, then by salary within each dept. This is the "leftmost prefix" rule: the index is usable when your query includes the leftmost column(s) of the index. Build composite indexes with the most frequently filtered column first, or the most selective column first if both are equally used.
Hard What is an index covering query and how does it eliminate heap fetches?▶
Answer: A covering index contains all columns needed by a query (in WHERE, SELECT, and ORDER BY) so the database never has to fetch the actual row from the heap (main table storage). Example: query is SELECT email, salary FROM employees WHERE dept = 'Engineering'. A composite index on (dept, email, salary) covers the entire query -- all needed values are in the index leaf nodes. No heap fetch needed. This is called an "index-only scan" and is significantly faster than a regular index scan + heap fetch, especially when rows are spread across many disk pages.
Knowledge Check
You create an index on (dept, salary). Which query can use this index efficiently?
Correct! The composite index (dept, salary) uses the leftmost prefix rule. It works when you filter on the leading column (dept) or both columns (dept AND salary). Queries that skip the leading column and only use salary cannot efficiently use this index -- the database would need to scan all entries in the index regardless. To support salary-only queries, create a separate index on (salary).

Read the database's execution plan to understand how it runs your query. Find slow scans, bad joins, and missing indexes before they become production problems.

Query Plans (EXPLAIN)

Advanced⏱ ~40 min
1 Concept Introduction

EXPLAIN (or EXPLAIN ANALYZE) shows the query execution plan -- the tree of operations the database engine chose to run your query. It reveals which indexes were used, what join algorithms were chosen, how many rows were estimated vs actual, and where time was spent. It is the primary diagnostic tool for slow queries.

2 Why It Exists

The query optimizer makes educated guesses based on statistics. Sometimes it guesses wrong -- choosing a full scan when an index exists, or a nested loop join when a hash join would be faster. EXPLAIN lets you see what decision it made and why, so you can help it make a better choice through indexes, statistics updates, or query rewrites.

3 Real World Analogy
🗺️
EXPLAIN is the GPS turn-by-turn before you drive

Before starting a road trip, GPS shows you the route: which highway, which exits, estimated time. EXPLAIN ANALYZE is GPS with live traffic: it shows the planned route AND the actual time taken at each step after the drive. You can see if a shortcut actually saved time or if the optimizer chose a detour that cost you 20 minutes.

🔎
Seq Scan

Sequential (full) table scan. Reads every row. Acceptable for small tables or when most rows match. Bad on large tables with selective filters.

⚡

Uses an index to find matching rows. Good for selective queries. Note: still fetches heap rows (unlike Index Only Scan).

🏎️

All needed columns are in the index -- no heap fetch. Fastest read path for covering indexes.

Builds a hash table from one input, probes with the other. Good for medium-to-large unsorted inputs.

🔄

For each row in outer input, scan inner input. Fast when inner input is small and indexed. Slow for large tables.

7 Common Mistakes
Mistake

Wrapping an indexed column in a function: WHERE UPPER(email) = 'BOB@X.COM' prevents index use because the index stores email values, not UPPER(email) values.

Fix

Either store data consistently (lowercase at insert time), use a function-based index: CREATE INDEX ON employees (UPPER(email)), or use ILIKE in PostgreSQL which can use indexes with the right operator class.

Mistake

Using EXPLAIN with no ANALYZE on production writes (UPDATE/DELETE). Without ANALYZE, the query is not executed so you only see estimates, not actual costs.

Fix

For read queries, use EXPLAIN ANALYZE to get actual timing. Wrap in a transaction and rollback to safely EXPLAIN ANALYZE a write: BEGIN; EXPLAIN ANALYZE DELETE ...; ROLLBACK;

8 Interview Questions
Interview Questions
Query Plans
Easy What is the difference between EXPLAIN and EXPLAIN ANALYZE?▶
Answer: EXPLAIN shows the query plan the optimizer chose based on statistics, with estimated row counts and costs -- but does NOT execute the query. EXPLAIN ANALYZE actually executes the query and shows both the estimated plan AND the actual execution time, actual row counts, and loop counts. Use EXPLAIN for a quick check without running the query. Use EXPLAIN ANALYZE to see actual performance and compare estimated vs actual rows (a big gap indicates outdated statistics). Warning: EXPLAIN ANALYZE on INSERT/UPDATE/DELETE modifies data, so wrap in BEGIN/ROLLBACK.
Medium What does a large difference between "Rows" (estimated) and "Actual Rows" indicate?▶
Answer: A large gap indicates the optimizer's statistics are stale -- the table has changed significantly since the last statistics collection. The optimizer made a bad plan based on old data distribution. Fix: run ANALYZE table_name (PostgreSQL) or ANALYZE TABLE table_name (MySQL) to update column statistics. In PostgreSQL, autovacuum runs ANALYZE automatically but may lag on fast-changing tables. Stale statistics often cause the optimizer to choose nested loops instead of hash joins, or to miss indexes.

EXPLAIN shows "Seq Scan" on a large table for your WHERE email = 'bob@x.com' query, despite an index on email. What is the most likely cause?

10 SQL Performance Tips

Advanced⏱ ~45 min

These are the 10 most impactful techniques for speeding up SQL queries, drawn from real-world data engineering and analytics practice. Each tip includes the "why" and a concrete code example.

01SELECT only the columns you need -- never SELECT *▶

SELECT * fetches every column including large TEXT/BLOB columns you do not need. This increases I/O, network transfer, and prevents covering index usage. On columnar databases (BigQuery, Snowflake, Redshift), SELECT * scans all column files -- massively more expensive than selecting 3 specific columns.

SQL
-- Bad
SELECT * FROM events;
-- Good
SELECT user_id, event_type, event_time FROM events;
02Filter early -- push WHERE conditions as close to the source as possible▶

The fewer rows entering each stage of the plan, the less work every subsequent step does. Apply selective filters before joins, before aggregations, and before window functions. In CTEs, filter within each CTE rather than in the final outer query.

SQL
-- Push filter into the CTE, not just the outer query
WITH recent_orders AS (
    SELECT * FROM orders
    WHERE order_date >= '2024-01-01'  -- filter here
)
SELECT customer_id, SUM(amount) FROM recent_orders
GROUP BY customer_id;
03Use indexed columns in JOIN and WHERE -- avoid wrapping them in functions▶

Applying a function to an indexed column defeats the index. The database computes the function for every row and cannot use the pre-sorted index. Store data in the form you query it (lowercase emails at write time, truncated dates at write time), or create function-based indexes.

SQL
-- Bad: index on email is NOT used
WHERE LOWER(email) = 'user@x.com'
-- Good: index IS used (store emails lowercased)
WHERE email = 'user@x.com'
04Use EXISTS instead of IN for large subquery result sets▶

IN with a large subquery materializes the entire subquery result into memory, then checks each outer row against it. EXISTS short-circuits as soon as one match is found, never fully materializing the inner result. EXISTS also correctly handles NULL values, while NOT IN with NULLs returns no rows.

SQL
-- Slower for large subquery
WHERE customer_id IN (SELECT id FROM vip_customers)
-- Faster + NULL-safe
WHERE EXISTS (SELECT 1 FROM vip_customers v WHERE v.id = o.customer_id)
05Avoid SELECT DISTINCT as a crutch -- fix the root cause of duplicates▶

DISTINCT performs a full sort or hash deduplication on all output rows. On large result sets this is expensive. If you are using DISTINCT to fix duplicates caused by a JOIN, the join condition is probably wrong (fan-out from a one-to-many relationship). Fix the join instead.

SQL
-- Bad: DISTINCT hiding a join problem
SELECT DISTINCT c.id FROM customers c JOIN orders o ON c.id = o.customer_id
-- Better: use EXISTS
SELECT c.id FROM customers c
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.id)
06Use LIMIT for exploratory queries and pagination▶

Never run SELECT * FROM huge_table in a REPL or notebook without LIMIT. Use LIMIT 10 to preview data. For cursor-based pagination (large offset gets slow), use keyset pagination: WHERE id > last_seen_id ORDER BY id LIMIT 100, rather than OFFSET which forces the database to skip N rows.

SQL
-- Slow offset pagination (must skip rows)
SELECT * FROM events ORDER BY id LIMIT 100 OFFSET 10000;
-- Fast keyset pagination
SELECT * FROM events WHERE id > 10099 ORDER BY id LIMIT 100;
07In data warehouses, partition pruning is your best friend▶

BigQuery, Snowflake, and Redshift partition large tables by date/region. If your WHERE clause includes the partition column, only the relevant partitions (file chunks) are scanned -- skipping terabytes of data. Always filter on the partition column. Avoid implicit casting that prevents pruning (e.g., comparing a DATE column to a string).

SQL
-- Good: partition pruning on date column
WHERE event_date BETWEEN '2024-01-01' AND '2024-01-31'
-- Bad: function on partition column defeats pruning
WHERE EXTRACT(YEAR FROM event_date) = 2024
08Prefer window functions over self-joins for row comparisons▶

A common pattern is comparing each row to a neighbor or to a group average. Before window functions, this required joining the table to itself. Self-joins double data processing. Window functions compute the same result in a single pass through the data.

SQL
-- Bad: self-join for previous month comparison
SELECT a.month, a.revenue, b.revenue AS prev
FROM monthly m1 JOIN monthly m2 ON m2.month_num = m1.month_num - 1;
-- Good: LAG window function
SELECT month, revenue, LAG(revenue,1) OVER (ORDER BY month) AS prev
FROM monthly;
09Keep statistics fresh -- run ANALYZE after bulk loads▶

The query optimizer relies on column statistics (min, max, most common values, histogram) to estimate row counts and choose plans. After bulk inserts or batch jobs that change large portions of a table, statistics may be stale. Run ANALYZE (PostgreSQL) or UPDATE STATISTICS (SQL Server) to refresh them and avoid bad query plans.

SQL
-- PostgreSQL
ANALYZE large_events_table;
-- Also useful: VACUUM ANALYZE (reclaims dead rows + refreshes stats)
VACUUM ANALYZE large_events_table;
10Use approximate aggregates for massive datasets where exact precision is not required▶

Exact COUNT(DISTINCT user_id) on billions of rows is extremely expensive (must track every unique value). Approximate functions like HLL_COUNT.DISTINCT (BigQuery), APPROX_COUNT_DISTINCT (Snowflake/BigQuery), and APPROX_PERCENTILE use probabilistic algorithms (HyperLogLog, t-Digest) that compute in O(1) memory with ~1-2% error -- usually acceptable for dashboards.

SQL
-- Exact (expensive on large tables)
SELECT COUNT(DISTINCT user_id) FROM events;
-- Approximate (BigQuery -- ~1-2% error, much faster)
SELECT APPROX_COUNT_DISTINCT(user_id) FROM events;
-- Snowflake
SELECT APPROX_COUNT_DISTINCT(user_id) FROM events;
Knowledge Check
You query BigQuery: WHERE EXTRACT(YEAR FROM event_date) = 2024. The table is partitioned by event_date. What problem does this cause?
Correct! Applying EXTRACT() to the partition column prevents partition pruning. BigQuery cannot determine which date partitions are relevant when the column is wrapped in a function, so it scans ALL partitions. Fix: WHERE event_date BETWEEN '2024-01-01' AND '2024-12-31'. This allows the query engine to prune to only 2024 partitions, potentially skipping years of data.

Project 8

End-to-End: Multi-Layer Analytics Pipeline

Expert

This capstone project combines everything: raw events table, staging CTEs, business logic CTEs, window functions, and a final aggregated output in a single SQL statement. Mimics a real dbt model.

Requirements
  • Stage 1 CTE: deduplicate raw events by (user_id, session_id)
  • Stage 2 CTE: sessionize events -- assign session boundaries (new session = >30 min gap using LAG)
  • Stage 3 CTE: compute session-level metrics (duration, events per session, converted)
  • Final output: daily cohort table with avg session duration, conversion rate, active users
SQL - Sessionization with LAG
WITH deduped AS (
    SELECT DISTINCT user_id, session_id, event_type, event_time
    FROM raw_events
),
with_gaps AS (
    SELECT *,
        LAG(event_time) OVER (
            PARTITION BY user_id
            ORDER BY     event_time
        ) AS prev_event_time
    FROM deduped
),
sessionized AS (
    SELECT *,
        SUM(CASE WHEN
            prev_event_time IS NULL
            OR event_time - prev_event_time > INTERVAL '30 minutes'
            THEN 1 ELSE 0 END)
        OVER (PARTITION BY user_id ORDER BY event_time)
        AS session_num
    FROM with_gaps
)
SELECT
    user_id,
    session_num,
    MIN(event_time)            AS session_start,
    MAX(event_time)            AS session_end,
    COUNT(*)                   AS events,
    MAX(event_time) - MIN(event_time) AS duration
FROM     sessionized
GROUP BY user_id, session_num
ORDER BY user_id, session_num;
