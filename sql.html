What is Data?

Data is raw information - facts, figures, and observations that can be collected, stored, and processed. In the digital world, data is the foundation of every business decision.

At its core, data is simply recorded facts. A number, a name, a date, a yes/no answer - these are all data points. When we collect and organize these facts into structured formats, we can query them, analyze them, and extract insights that drive real decisions.

Database Fundamentals

A database is an organized collection of structured information stored electronically. Understanding the core building blocks is essential before writing your first query.

Think of a database like a digital library.

Tables are like books organized by topic, rows are individual records (like pages), and columns define what information each record stores.

📊
Table
A structured set of data organized in rows and columns. Like a spreadsheet tab, each table stores one type of entity (e.g., employees, products, orders).
➡
Row / Record
A single instance of data in a table. One employee, one order, one product. Also called a tuple in relational theory.
↕
Column / Field
A named attribute of a table. Each column has a data type (INT, VARCHAR, DATE, DECIMAL) that constrains what values it can hold.
🔑
Primary Key
A column (or set of columns) that uniquely identifies each row. Cannot contain NULL values. Ensures no duplicate records exist.
🔗
Foreign Key
A column that references the primary key of another table. Creates relationships between tables and enforces referential integrity.

GROUP BY Rule:

Any column in SELECT that is not an aggregate function must appear in the GROUP BY clause. Otherwise SQL doesn't know which row's value to show for the group.

WHERE - Filters Rows
  • Runs before GROUP BY
  • Cannot use aggregates (COUNT, SUM...)
  • Filters individual rows
HAVING - Filters Groups
  • Runs after GROUP BY
  • Can use aggregate functions
  • Filters entire groups
-- Group by department
SELECT
    department,
    COUNT(*)       AS headcount,
    AVG(salary)    AS avg_salary,
    MAX(salary)    AS top_salary
FROM employees
GROUP BY department
ORDER BY avg_salary DESC;
-- HAVING filters groups (not rows)
SELECT department, COUNT(*) AS headcount
FROM employees
GROUP BY department
HAVING COUNT(*) >= 2;
-- WHERE + GROUP BY + HAVING together
SELECT
    department,
    COUNT(*)    AS headcount,
    AVG(salary) AS avg_salary
FROM employees
WHERE hire_date >= '2019-01-01'    -- filter rows first
GROUP BY department
HAVING AVG(salary) > 75000;         -- then filter groups
-- Multiple GROUP BY columns
SELECT department, hire_date, COUNT(*)
FROM employees
GROUP BY department, hire_date
ORDER BY department, hire_date;

Anatomy of OVER()

PARTITION BYDivides rows into groups (like GROUP BY but keeps all rows)
ORDER BYDefines row order within each partition
ROWS BETWEENFrame clause: which rows to include in calculation window
SUM(salary) OVER (
  PARTITION BY department
  ORDER BY hire_date
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
) 
-- ROW_NUMBER: unique sequential number
SELECT
    name,
    salary,
    ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num
FROM employees;
-- RANK vs DENSE_RANK (difference with ties)
-- If Alice and Grace both have $95k:
-- RANK:       1, 1, 3, 4...  (skips 2)
-- DENSE_RANK: 1, 1, 2, 3...  (no gap)
SELECT
    name,
    salary,
    RANK()       OVER (ORDER BY salary DESC) AS rnk,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rnk
FROM employees;
-- LAG and LEAD: compare to adjacent rows
SELECT
    name,
    hire_date,
    salary,
    LAG(salary,  1) OVER (ORDER BY hire_date) AS prev_salary,
    LEAD(salary, 1) OVER (ORDER BY hire_date) AS next_salary
FROM employees;
-- Running total
SELECT
    name,
    hire_date,
    salary,
    SUM(salary) OVER (
        ORDER BY hire_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
FROM employees;
-- Partition by department: rank within each dept
SELECT
    name,
    department,
    salary,
    DENSE_RANK() OVER (
        PARTITION BY department
        ORDER BY salary DESC
    ) AS dept_rank
FROM employees;

Query Optimization

Slow queries cost money and frustrate users. Understanding indexing and query patterns can make the difference between a query taking 8 seconds vs 0.04 seconds.

Full Table Scan vs Index Lookup
Without Index
0.0 sec
With Index
0.000 sec
205x faster with index!
6 Key Optimization Strategies
  1. Use indexes on filter and join columns - Index foreign keys, columns used in WHERE, and columns in JOIN ON clauses.
  2. Select only needed columns - Avoid SELECT *. Fetch only the columns your application actually needs.
  3. Filter early with WHERE - Apply WHERE clauses that reduce the row count as early as possible in the query.
  4. Avoid functions on indexed columns - WHERE YEAR(hire_date) = 2020 breaks the index. Use range instead: hire_date BETWEEN '2020-01-01' AND '2020-12-31'.
  5. Use EXPLAIN to understand execution plans - Always check the query plan before deploying a new query to production.
  6. Partition large tables - For tables with billions of rows, partitioning by date or region limits how much data is scanned.
-- Bad: function on column prevents index use
SELECT * FROM employees
WHERE YEAR(hire_date) = 2020;

-- Good: range query uses index
SELECT name, department, salary
FROM employees
WHERE hire_date BETWEEN '2020-01-01' AND '2020-12-31';
-- Create indexes on commonly filtered columns
CREATE INDEX idx_employees_dept
  ON employees(department);

CREATE INDEX idx_employees_hire_date
  ON employees(hire_date);

-- Composite index for multi-column filters
CREATE INDEX idx_emp_dept_salary
  ON employees(department, salary);
-- EXPLAIN shows the query execution plan
EXPLAIN SELECT name, salary
FROM employees
WHERE department = 'Engineering';

-- EXPLAIN ANALYZE actually runs the query
EXPLAIN ANALYZE SELECT name, salary
FROM employees
WHERE department = 'Engineering';

-- Seq Scan = reading every row (slow on large tables)
-- Index Scan = using an index (fast)

SQL Interview Questions

Easy
Easy What is the difference between WHERE and HAVING? ⌄

WHERE filters individual rows before grouping occurs. It runs first in the query execution order and cannot reference aggregate functions like COUNT, SUM, or AVG.

HAVING filters groups after GROUP BY has executed. It can reference aggregate functions.

-- WHERE: filter rows before grouping
SELECT department, COUNT(*)
FROM employees
WHERE hire_date >= '2020-01-01'   -- filters rows
GROUP BY department
HAVING COUNT(*) > 1;               -- filters groups
Easy What is a PRIMARY KEY? ⌄

A PRIMARY KEY is a column (or combination of columns) that uniquely identifies each row in a table. It has two constraints:

  • Unique - no two rows can have the same primary key value
  • NOT NULL - primary key columns cannot contain NULL

Each table can have only one primary key. Primary keys are automatically indexed by most databases.

CREATE TABLE employees (
    id         INT PRIMARY KEY,  -- simple primary key
    name       VARCHAR(100),
    email      VARCHAR(255) UNIQUE
);

-- Composite primary key (multiple columns)
CREATE TABLE order_items (
    order_id   INT,
    product_id INT,
    quantity   INT,
    PRIMARY KEY (order_id, product_id)
);
Easy What does DISTINCT do? ⌄

DISTINCT removes duplicate rows from query results. When applied to a column, it returns only unique values.

If the employees table has Alice, Bob, and Carol in Engineering plus Frank in Marketing, SELECT DISTINCT department returns just 2 rows: Engineering and Marketing.

-- Without DISTINCT: 8 rows
SELECT department FROM employees;

-- With DISTINCT: 4 rows (Engineering, Marketing, Sales, HR)
SELECT DISTINCT department FROM employees;

-- DISTINCT on multiple columns
SELECT DISTINCT department, hire_date
FROM employees;
Easy UNION vs UNION ALL - what is the difference? ⌄

UNION combines results from two queries and removes duplicate rows. This requires sorting the data to find duplicates, making it slower.

UNION ALL combines results and keeps all rows including duplicates. It is faster because no deduplication step is needed.

Use UNION ALL by default unless you specifically need deduplication. Both require matching column count and compatible data types.

-- UNION: removes duplicates (slower)
SELECT name FROM employees_2023
UNION
SELECT name FROM employees_2024;

-- UNION ALL: keeps all rows (faster)
SELECT name FROM employees_2023
UNION ALL
SELECT name FROM employees_2024;
Easy How do you handle NULL values in SQL? ⌄

NULL represents unknown or missing data. It is not zero, not an empty string - it is the absence of a value.

Key rule: any comparison using = with NULL returns NULL (not TRUE or FALSE). Always use IS NULL or IS NOT NULL.

-- Wrong: this returns no rows!
WHERE manager_id = NULL

-- Correct
WHERE manager_id IS NULL
WHERE manager_id IS NOT NULL

-- COALESCE: substitute a default for NULL
SELECT name, COALESCE(manager_id, 0) AS mgr
FROM employees;

-- NULLIF: returns NULL if values are equal
SELECT NULLIF(discount, 0) AS discount
FROM orders;

Medium Explain INNER JOIN vs LEFT JOIN with an example ⌄

INNER JOIN returns only rows where the join condition matches in both tables. Unmatched rows are excluded.

LEFT JOIN returns all rows from the left table plus matched rows from the right table. Where there is no match, right-side columns are NULL.

-- INNER JOIN: excludes employees with no department
SELECT e.name, d.name AS dept
FROM employees e
INNER JOIN departments d ON e.dept_id = d.id;
-- Result: 4 rows (Eva with NULL dept_id is excluded)

-- LEFT JOIN: includes ALL employees
SELECT e.name, d.name AS dept
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id;
-- Result: 5 rows (Eva appears with NULL dept)
Medium What is a CTE (Common Table Expression)? ⌄

A CTE is a temporary named result set defined with the WITH clause. It exists only for the duration of the query. CTEs improve readability and allow you to reference the same subquery multiple times.

-- CTE: calculate department averages, then filter
WITH dept_averages AS (
    SELECT
        department,
        AVG(salary) AS avg_salary,
        COUNT(*)    AS headcount
    FROM employees
    GROUP BY department
)
SELECT department, avg_salary, headcount
FROM dept_averages
WHERE avg_salary > 75000
ORDER BY avg_salary DESC;

-- Multiple CTEs chained
WITH
  high_earners AS (
    SELECT * FROM employees WHERE salary > 80000
  ),
  eng_high AS (
    SELECT * FROM high_earners WHERE department = 'Engineering'
  )
SELECT name, salary FROM eng_high;
Medium Difference between ROW_NUMBER(), RANK(), and DENSE_RANK()? ⌄

All three assign numbers to rows based on ORDER BY, but they handle ties differently:

  • ROW_NUMBER() - always unique, no ties. Arbitrary tiebreak.
  • RANK() - ties get the same rank, then skips the next number(s).
  • DENSE_RANK() - ties get the same rank, no gaps in sequence.
-- If two employees both earn $95,000:
SELECT name, salary,
    ROW_NUMBER() OVER (ORDER BY salary DESC) AS rn,
    RANK()       OVER (ORDER BY salary DESC) AS rnk,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS drnk
FROM employees;

-- name       salary  rn  rnk  drnk
-- Alice      95000    1    1     1
-- Grace      95000    2    1     1
-- Carol      88000    3    3     2   <-- RANK skips 2
-- Frank      75000    4    4     3
Medium How do you find duplicate records in a table? ⌄

Use GROUP BY on the columns that should be unique, then filter with HAVING COUNT(*) > 1 to find groups with more than one row.

-- Find duplicate emails
SELECT email, COUNT(*) AS occurrences
FROM users
GROUP BY email
HAVING COUNT(*) > 1
ORDER BY occurrences DESC;

-- Find the actual duplicate rows
SELECT *
FROM users
WHERE email IN (
    SELECT email
    FROM users
    GROUP BY email
    HAVING COUNT(*) > 1
)
ORDER BY email;
Medium What is a correlated subquery? ⌄

A correlated subquery references a column from the outer query. It is re-executed for every row of the outer query, which can be slow on large tables but is sometimes necessary.

-- Find employees who earn above their dept average
-- The subquery references e1.department from outer query
SELECT e1.name, e1.department, e1.salary
FROM employees e1
WHERE e1.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department = e1.department  -- correlated!
)
ORDER BY e1.department, e1.salary DESC;
Medium Write a query to calculate a running total ⌄

Use SUM() as a window function with ORDER BY. The default frame is ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, which accumulates all previous rows plus the current one.

-- Running total of sales by date
SELECT
    sale_date,
    amount,
    SUM(amount) OVER (
        ORDER BY sale_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
FROM sales
ORDER BY sale_date;

-- Running total per customer
SELECT
    customer_id,
    sale_date,
    amount,
    SUM(amount) OVER (
        PARTITION BY customer_id
        ORDER BY sale_date
    ) AS customer_running_total
FROM sales;
Medium What is the difference between clustered and non-clustered index? ⌄

Clustered index determines the physical order of rows on disk. The data rows are stored sorted by the clustered index key. There can be only one per table (usually the primary key). Looking up by clustered index is fastest because the data is right there.

Non-clustered index is a separate structure that contains the index key plus a pointer to the actual row. You can have many non-clustered indexes per table. They add a lookup step but are flexible.

Think of a clustered index like a dictionary (words sorted alphabetically = data order), and non-clustered like the index at the back of a textbook (separate list pointing to page numbers).

Medium How to find the second highest salary? ⌄

Multiple approaches work. The most robust uses DENSE_RANK() to handle ties correctly:

-- Method 1: DENSE_RANK (handles ties correctly)
WITH ranked AS (
    SELECT name, salary,
           DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
    FROM employees
)
SELECT name, salary
FROM ranked
WHERE rnk = 2;

-- Method 2: Subquery approach
SELECT MAX(salary) AS second_highest
FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

-- Method 3: LIMIT/OFFSET (may miss ties)
SELECT DISTINCT salary
FROM employees
ORDER BY salary DESC
LIMIT 1 OFFSET 1;
Medium What is COALESCE and when do you use it? ⌄

COALESCE returns the first non-NULL value from a list of arguments. It is the standard way to substitute a default value when a column might be NULL.

-- Replace NULL with a default
SELECT name, COALESCE(bonus, 0) AS bonus
FROM employees;

-- Fallback chain: use first non-null
SELECT COALESCE(preferred_name, first_name, 'Unknown') AS display_name
FROM users;

-- Useful in aggregations to avoid NULL propagation
SELECT
    dept,
    COALESCE(SUM(revenue), 0) AS total_revenue
FROM sales
GROUP BY dept;

Hard How do you pivot rows into columns in SQL? ⌄

Pivoting transforms row values into column headers. In standard SQL, use conditional aggregation with CASE WHEN. Some databases (SQL Server, Snowflake) have a PIVOT keyword.

-- Pivot: show each department as a column
-- Input: department, salary per row
-- Output: one row per year, columns per dept

SELECT
    YEAR(hire_date) AS year,
    SUM(CASE WHEN department = 'Engineering' THEN salary END) AS engineering,
    SUM(CASE WHEN department = 'Marketing'   THEN salary END) AS marketing,
    SUM(CASE WHEN department = 'Sales'       THEN salary END) AS sales,
    SUM(CASE WHEN department = 'HR'          THEN salary END) AS hr
FROM employees
GROUP BY YEAR(hire_date)
ORDER BY year;
Hard Find employees who earn more than their manager ⌄

This is a classic self-join problem. Join the employees table to itself using manager_id, then compare salaries.

-- Self-join: emp table joined to itself
SELECT
    e.name        AS employee,
    e.salary      AS employee_salary,
    m.name        AS manager,
    m.salary      AS manager_salary
FROM employees e
JOIN employees m ON e.manager_id = m.id
WHERE e.salary > m.salary
ORDER BY e.salary DESC;

-- The table structure assumed:
-- employees (id, name, salary, manager_id)
-- manager_id references id in the same table
Hard What is an execution plan and how do you read it? ⌄

An execution plan shows how the database will physically execute your query - which indexes it uses, what operations it performs, and what the estimated cost is. It is the most important tool for query optimization.

-- Generate execution plan (PostgreSQL)
EXPLAIN SELECT name FROM employees
WHERE department = 'Engineering';

-- Output example:
-- Seq Scan on employees (cost=0.00..1.10 rows=3)
--   Filter: (department = 'Engineering')

-- With actual runtime stats
EXPLAIN ANALYZE SELECT name FROM employees
WHERE department = 'Engineering';

-- Key things to look for:
-- Seq Scan   = reading every row (bad on large tables)
-- Index Scan = using index lookup (fast)
-- cost=X..Y  = estimated startup cost .. total cost
-- rows=N     = estimated rows returned
-- actual time= actual milliseconds spent
Hard A query takes 5 minutes to run. Walk me through how you optimize it. ⌄

This is a systematic process. Never guess - always measure first.

  1. EXPLAIN ANALYZE - get the actual execution plan. Identify the slowest operations.
  2. Check for Seq Scans - if a large table uses a sequential scan, add an index on the filter column.
  3. Examine JOINs - are join columns indexed? Large hash joins on unindexed columns are expensive.
  4. COUNT rows at each step - is the query returning millions of rows unnecessarily? Add a LIMIT or tighten the WHERE clause.
  5. Move filters earlier - filter before joining when possible to reduce intermediate result size.
  6. Avoid functions on indexed columns - rewrite YEAR(date) = 2023 as a range.
  7. Consider materialized views - for repeated complex aggregations, store the result.
  8. Check statistics freshness - run ANALYZE to update table statistics so the planner makes better decisions.
Hard What are ACID properties? ⌄

ACID describes the four properties that guarantee database transactions are processed reliably.

A - Atomicity: A transaction is all-or-nothing. If a bank transfer deducts from account A but crashes before crediting account B, the whole transaction is rolled back.
C - Consistency: A transaction brings the database from one valid state to another. Constraints and rules are always enforced.
I - Isolation: Concurrent transactions execute as if they were sequential. One transaction's work-in-progress is invisible to others.
D - Durability: Once committed, a transaction persists even through system failure. Data is written to disk, not just memory.
Hard What is the difference between a view and a materialized view? ⌄
View
  • Virtual - no stored data
  • Runs the query every time
  • Always shows fresh data
  • No storage cost
  • Can be slow on complex queries
Materialized View
  • Physical - stores results on disk
  • Reads pre-computed results
  • Data can be stale until refresh
  • Uses storage space
  • Extremely fast for dashboards

Use a regular view for simplicity and freshness. Use a materialized view for expensive aggregations that are queried frequently but don't need real-time data (e.g., daily sales dashboards).

Hard Write a query to get the top 3 earners per department ⌄

Use ROW_NUMBER() with PARTITION BY department to rank within each group, then filter to the top 3. Use DENSE_RANK() if you want to include ties.

-- Top 3 earners per department
WITH dept_ranked AS (
    SELECT
        name,
        department,
        salary,
        ROW_NUMBER() OVER (
            PARTITION BY department
            ORDER BY salary DESC
        ) AS salary_rank
    FROM employees
)
SELECT name, department, salary, salary_rank
FROM dept_ranked
WHERE salary_rank <= 3
ORDER BY department, salary_rank;

-- Use DENSE_RANK instead if you want ties included:
-- DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC)
Hard What is index selectivity and why does it matter? ⌄

Selectivity is the ratio of distinct values to total rows in a column. High selectivity means the index narrows down results to few rows quickly. Low selectivity means the index is not worth using.

High selectivity (good for indexes): email, user_id, order_number - nearly every value is unique. An index on email immediately narrows millions of rows to one.
Low selectivity (bad for indexes): gender (2 values), is_active (true/false), country (200 values in a 100M row table). The database may skip the index and do a full scan anyway.

Rule of thumb: an index is worth using if it filters out more than 80-90% of rows. Always check selectivity before adding an index.

Hard OLTP vs OLAP - what is the difference? ⌄
OLTP
Online Transaction Processing
  • Many small, fast transactions
  • INSERT/UPDATE/DELETE heavy
  • Normalized schema (3NF)
  • Optimized for writes
  • e.g. e-commerce orders, banking
OLAP
Online Analytical Processing
  • Complex queries, large scans
  • SELECT heavy, mostly reads
  • Denormalized (star/snowflake schema)
  • Optimized for reads/aggregations
  • e.g. data warehouses, dashboards
Hard When would you use a subquery vs JOIN vs CTE? ⌄

Each approach has tradeoffs in readability, performance, and reusability:

Subquery - best for one-time filters or scalar values. Easy to write for simple cases. Can be hard to read when nested deeply. Correlated subqueries can be slow.
JOIN - best when combining columns from multiple tables. Generally the most performant because the optimizer can choose efficient join algorithms. Harder to read with complex multi-table joins.
CTE - best for readability and breaking complex logic into named steps. Performance is usually the same as a subquery (optimizer may inline it). Ideal for multi-step transformations or when you need to reference the same intermediate result twice. 
-- Same logic, three styles:

-- Subquery
SELECT name FROM employees
WHERE dept_id IN (SELECT id FROM departments WHERE name = 'Engineering');

-- JOIN (usually preferred for performance)
SELECT e.name FROM employees e
JOIN departments d ON e.dept_id = d.id
WHERE d.name = 'Engineering';

-- CTE (best for readability in complex queries)
WITH eng_dept AS (
    SELECT id FROM departments WHERE name = 'Engineering'
)
SELECT e.name FROM employees e
JOIN eng_dept d ON e.dept_id = d.id;
