7.1 Azure Basics
Global infrastructure, the Azure hierarchy, and why it matters for data engineering

🌏 Azure Global Infrastructure

Microsoft Azure operates 60+ regions worldwide, making it one of the largest cloud providers globally. Each region contains Availability Zones (AZs) - three or more physically separate datacenters within a region, each with independent power, cooling, and networking. This allows you to build fault-tolerant applications that survive datacenter-level outages.

  • Region: Geographic area (e.g., East US, West Europe, Southeast Asia)
  • Availability Zone: Physically separate datacenter within a region
  • Region Pair: Two regions paired for disaster recovery (e.g., East US ↔ West US)

🔒 Azure Organizational Hierarchy

Azure organizes resources in a strict four-level hierarchy. Understanding this is critical for permissions, billing, and resource management.

👤 Azure AD Tenant   - org-level identity provider (e.g., yourcompany.onmicrosoft.com)
  └── 💳 Subscription - billing unit; you can have multiple (Dev, Staging, Prod)
       └── 📁 Resource Group - logical container for related resources
             └── 🛠️ Resources - Storage, Databricks, ADF, Key Vault, etc.

7.2 Storage Account + Blob Storage
Azure's object storage service - the foundation of every data lake

💾 Azure Storage Account

A Storage Account is an umbrella service that provides access to multiple Azure Storage services under a single namespace. Think of it as the "account" that contains everything. Within a Storage Account you can have:

  • Blob Storage - unstructured object storage (CSV, Parquet, JSON, images, videos)
  • File Storage - managed file shares (SMB/NFS protocol)
  • Queue Storage - message queuing for async communication
  • Table Storage - NoSQL key-value store

🗄 Blob Storage Hierarchy

Blob Storage holds unstructured data. The structure is:

  • Container - top-level folder (like an S3 bucket). Can be public or private.
  • Blob - the actual file. Three types: Block Blob (large files), Append Blob (logs), Page Blob (VHDs)
  • Virtual directory - folders are simulated using "/" in blob names (no real hierarchy in plain Blob)

7.3 ADLS Gen2
Azure Data Lake Storage Gen2 - hierarchical namespace, ACLs, and why it's the recommended data lake storage

🌏 What is ADLS Gen2?

Azure Data Lake Storage Gen2 is Blob Storage with Hierarchical Namespace (HNS) enabled. This single setting transforms flat object storage into a true filesystem with real directories - not simulated ones.

  • Hierarchical Namespace: True directory structure like a Linux filesystem. Rename or delete a directory is a single atomic operation regardless of how many files are inside.
  • ACL Permissions: Unix-style Access Control Lists applied per file or per directory - not just at container level.
  • ABFS Driver: Azure Blob File System driver, Hadoop-compatible. Spark uses abfss:// protocol.
  • Performance: Optimized for big data analytics workloads (Spark, Hive, Presto).

🔒 ACL Permissions (Unix-style)

Each file/folder has three permission bits:

r - Read

List directory contents or read file data

w - Write

Create or modify files in the directory

x - Execute

Navigate into the directory (required to traverse path)

Permissions are applied to: Named user (specific user/service principal), Named group, or Other (everyone else).

7.4 Identity and Access
Service Principals, Managed Identities, and RBAC - securing your Azure data platform

👥 Service Principal

An app identity - like a user account for an application. Has a client_id + client_secret (or certificate). You manage and rotate the secrets yourself.

✅ Managed Identity

Azure manages credentials automatically. No secrets to store or rotate. Best practice for Databricks, ADF, and all Azure services. Two types: System-assigned and User-assigned.

🔒 RBAC

Role-Based Access Control - assign built-in or custom roles to identities at any scope (subscription, resource group, or resource).

8.1 Azure Data Factory (ADF)
Cloud ETL and orchestration - move and transform data without writing Spark code

⚙️ What is Azure Data Factory?

ADF is a serverless data integration and orchestration service. It connects 100+ data sources (databases, SaaS apps, files) and orchestrates complex workflows. You build pipelines visually or with JSON - no cluster management needed.

Best for: data movement, orchestration, scheduled pipelines, connecting heterogeneous sources. Not ideal for complex in-memory transformations (use Databricks for that).

🔄 Pipeline

A logical grouping of activities that together perform a task. The top-level unit in ADF.

⚡ Activity

A single step in a pipeline: Copy, Databricks Notebook, Stored Procedure, Web, etc.

💾 Dataset

A named view of data - describes the structure and location of data in a Linked Service.

🔗 Linked Service

Connection string / auth info for a data source or compute (like a Databricks workspace).

⌛ Trigger

Defines when a pipeline runs: Schedule (cron), Tumbling Window, Storage Event, or Manual.

🌎 Integration Runtime

The compute infrastructure: Azure IR (cloud), Self-hosted IR (on-premises), or SSIS IR.

8.2 Azure Databricks
Managed Apache Spark platform - notebooks, jobs, and production workloads

⚡ What is Azure Databricks?

Databricks is a managed Apache Spark platform deeply integrated with Azure. It provides collaborative notebooks, auto-scaling clusters, built-in MLflow for ML tracking, and Unity Catalog for governance. Best for complex data transformations, ML workloads, and production Spark jobs.

📄 Notebook

Interactive Python/Scala/SQL/R environment. Cells execute on the attached cluster.

🔨 Cluster

Apache Spark compute. Auto-scaling, auto-terminating. Two main types (see below).

⏱ Job

Scheduled or triggered run of a notebook/JAR/Python script. Production workloads run as Jobs.

💾 Repos

Git integration directly in the Databricks workspace. Sync notebooks with GitHub/Azure DevOps.

8.3 Unity Catalog
Centralized data governance for all Databricks workspaces - permissions, lineage, and audit

📚 What is Unity Catalog?

Unity Catalog is Databricks' centralized governance layer. One catalog governs all workspaces in an Azure region. It provides:

  • 3-level namespace: catalog.schema.table - eliminates ambiguity across workspaces
  • Fine-grained permissions: grant access at catalog, schema, table, or column level
  • Column-level security: mask PII fields (SSN, email) for certain groups
  • Data lineage: automatic tracking of how data flows from source to consumption
  • Audit logs: every query, every access attempt logged to Azure Monitor

USE CATALOG prod_catalog;
USE SCHEMA silver;

-- Grant table access to a group
GRANT SELECT ON TABLE silver.employees TO GROUP analysts;
GRANT SELECT, MODIFY ON SCHEMA silver TO GROUP data_engineers;

-- Column masking (hide SSN from non-privileged users)
CREATE OR REPLACE FUNCTION mask_ssn(ssn STRING)
  RETURN IF(is_account_group_member('pii_access'), ssn, '***-**-****');

ALTER TABLE silver.employees
  SET TBLPROPERTIES ('delta.columnMask.ssn' = 'mask_ssn');

-- Query lineage information
SELECT * FROM system.access.audit
WHERE table_name = 'employees'
ORDER BY event_time DESC
LIMIT 100;

8.4 Delta Live Tables (DLT)
Declarative pipeline framework - define what you want, Databricks handles the rest

🔄 What is Delta Live Tables?

DLT is a declarative ETL framework built on top of Databricks and Delta Lake. Instead of writing imperative Spark code that says HOW to process data, you declare WHAT the output should look like using Python decorators or SQL. Databricks handles:

  • Dependency resolution: automatically determines execution order
  • Incremental processing: only processes new data (via Auto Loader or APPLY CHANGES)
  • Data quality: built-in expectation framework (expect, expect_or_drop, expect_or_fail)
  • Monitoring: event log with row counts, quality metrics per pipeline run
  • Retries: automatic retry on transient failures
  • Lineage: automatic lineage tracking in Unity Catalog

Python - DLT Pipeline Definition 
import dlt
from pyspark.sql import functions as F

# Bronze: raw ingestion from ADLS using Auto Loader
@dlt.table(
    name="bronze_employees",
    comment="Raw employee data ingested from ADLS"
)
def bronze_employees():
    return (spark.readStream
        .format("cloudFiles")  # Auto Loader
        .option("cloudFiles.format", "csv")
        .load("abfss://raw@datalakestorage01.dfs.core.windows.net/employees/"))

# Silver: clean + validate
@dlt.table(name="silver_employees", comment="Cleaned employee data")
@dlt.expect("valid_email", "email IS NOT NULL AND email LIKE '%@%'")
@dlt.expect_or_drop("valid_salary", "salary > 0")
@dlt.expect_or_fail("valid_employee_id", "employee_id IS NOT NULL")
def silver_employees():
    return (dlt.read_stream("bronze_employees")
        .withColumn("full_name", F.concat(F.col("first_name"), F.lit(" "), F.col("last_name")))
        .withColumn("salary", F.col("salary").cast("double"))
        .dropDuplicates(["employee_id"]))

# Gold: aggregated for reporting
@dlt.table(name="gold_dept_headcount", comment="Department headcount summary")
def gold_dept_headcount():
    return (dlt.read("silver_employees")  # batch read for aggregation
        .groupBy("department")
        .agg(
            F.count("employee_id").alias("headcount"),
            F.avg("salary").alias("avg_salary")
        ))

🎓 Interview Questions
15 common Azure Data Engineering interview questions with detailed answers
1. When would you use ADF vs Databricks for a data pipeline? Medium ⌄

Use ADF when:

  • You need to move data between 100+ supported connectors (SAP, Salesforce, Oracle, S3, etc.)
  • Your transformations are simple (column mapping, type casting) - use Data Flow activities
  • You need to orchestrate other services (trigger Databricks jobs, run stored procedures)
  • You want a no-code/low-code visual pipeline builder
  • You need built-in scheduling and monitoring with alerting

Use Databricks when:

  • Transformations are complex (joins across huge datasets, window functions, ML feature engineering)
  • You need full PySpark/Scala control for performance tuning
  • Building ML models or running MLflow experiments
  • Working with streaming data (Structured Streaming)
  • Need notebook-based iterative development

Best practice: Use ADF for orchestration + data movement, call Databricks notebooks/jobs for heavy transformation. ADF acts as the conductor, Databricks as the musician.

2. What is the difference between ADLS Gen2 and plain Blob Storage? Medium ⌄

The key difference is Hierarchical Namespace (HNS):

  • Blob Storage: flat namespace - "directories" are just prefixes in blob names. Renaming a "folder" means copying every file then deleting originals - O(n) operation.
  • ADLS Gen2: true directory structure. Rename/delete a directory is atomic O(1) - updates a single metadata pointer, regardless of how many files are inside.
  • ACL Permissions: Blob only supports container-level RBAC. ADLS Gen2 supports Unix-style ACLs at file and folder level.
  • Analytics performance: ADLS Gen2 uses the ABFS driver optimized for Spark/Hadoop workloads. Significantly faster for large analytical queries.
  • Hadoop compatibility: ADLS Gen2 implements the Hadoop Filesystem interface - Databricks and HDInsight work natively with it.

For any data lake use case, always choose ADLS Gen2 (enable HNS when creating the storage account - it cannot be changed later).

3. What is the difference between Managed Identity and Service Principal? Medium ⌄

Service Principal:

  • An app registration in Azure Entra ID - creates a client_id + client_secret
  • You are responsible for creating, storing, and rotating secrets
  • Can be used outside Azure (from on-premises or other clouds)
  • Risk: secrets can be leaked if stored insecurely

Managed Identity:

  • Azure automatically creates and manages the identity - no secrets exist
  • System-assigned: tied to a specific Azure resource (deleted when resource is deleted)
  • User-assigned: standalone identity that can be attached to multiple resources
  • Only works within Azure services
  • Best practice: always prefer Managed Identity for Azure-to-Azure authentication

Use Service Principal when: connecting from non-Azure environments, CI/CD pipelines, or when the service doesn't support Managed Identity.

4. Explain Unity Catalog's 3-level namespace with an example. Medium ⌄

Unity Catalog introduces a catalog.schema.table hierarchy (previously Databricks only had database.table):

  • Catalog: top-level container, often mapped to environment (prod_catalog, dev_catalog, sandbox_catalog)
  • Schema: logical grouping within a catalog, often mapped to medallion layer (bronze, silver, gold) or domain (finance, hr, sales)
  • Table: the actual Delta table

Example: prod_catalog.silver.employees

  • prod_catalog = production environment catalog
  • silver = cleaned/validated layer
  • employees = the employees Delta table

Benefits: You can grant access at any level (GRANT SELECT ON CATALOG dev_catalog TO GROUP developers), have separate dev/prod catalogs in the same workspace, and all workspaces across an Azure region share the same catalog.

5. What are the three DLT expectation types and their behaviors? Easy ⌄
  • @dlt.expect("name", "condition"): If condition is violated, the row is kept in the output but a warning is logged in the DLT event log. Use when you want to monitor quality without blocking data flow.
  • @dlt.expect_or_drop("name", "condition"): If condition is violated, the row is dropped (filtered out). Pipeline continues. Dropped rows are counted in quality metrics. Use when bad rows should be excluded but pipeline should proceed.
  • @dlt.expect_or_fail("name", "condition"): If any row violates the condition, the entire pipeline run fails immediately. Use for critical constraints (e.g., primary key must never be null) where bad data flowing downstream would be catastrophic.

All three types log metrics to the DLT event log so you can monitor data quality trends over time.

6. What are the three layers of the Medallion architecture? Easy ⌄
  • Bronze (Raw): Data ingested as-is from source systems. No transformations. Append-only historical record. Format: Delta table (or raw files). Contains all original data including bad records.
  • Silver (Cleaned): Validated, deduplicated, type-cast, and lightly transformed data. Bad rows filtered or flagged. Joined with reference data. This is the "single source of truth" layer that analysts and data scientists typically query.
  • Gold (Aggregated/Business-Ready): Business-level aggregations optimized for reporting and dashboards. Department KPIs, daily summaries, denormalized tables for specific use cases. Directly feeds BI tools (Power BI, Tableau).

Key principle: Each layer adds value. Bronze = keep everything. Silver = make it reliable. Gold = make it fast and business-friendly.

7. How do you implement incremental load in ADF? Hard ⌄

Three main approaches:

  • Watermark pattern: Store the last successful load timestamp in a control table (Azure SQL). Each pipeline run: (1) read watermark from control table, (2) query source WHERE updated_at > watermark, (3) load new rows, (4) update watermark to current timestamp. Use Lookup activity to read watermark, Filter in Copy activity source query.
  • Change Data Capture (CDC): Enable CDC on SQL Server or use Azure SQL's built-in change tracking. ADF copies only changed rows. Much more efficient for high-update tables.
  • Partition-based: If source data is naturally partitioned by date (e.g., files named employees_2024_01_15.csv), use ADF's tumbling window trigger to process one partition per run. Combine with @trigger().outputs.windowStartTime dynamic expression.

Example watermark pipeline:

  • Activity 1: Lookup → SELECT max_watermark FROM dbo.watermark_table WHERE table_name = 'employees'
  • Activity 2: Copy → SELECT * FROM employees WHERE updated_at > '@{activity('Lookup').output.firstRow.max_watermark}'
  • Activity 3: Stored Procedure → UPDATE watermark_table SET max_watermark = GETDATE()
8. What is a Self-Hosted Integration Runtime and when do you need it? Hard ⌄

The Self-Hosted Integration Runtime (SHIR) is an agent you install on a machine inside your corporate network. It acts as a bridge between ADF (in Azure) and data sources that are not publicly accessible.

When you need SHIR:

  • Connecting to on-premises databases (SQL Server, Oracle, SAP) behind a firewall
  • Accessing databases in a private VNet without public endpoints
  • Connecting to file servers on-premises
  • When data cannot leave the corporate network due to compliance requirements

How it works:

  • SHIR makes outbound HTTPS connections to Azure Service Bus - no inbound firewall rules needed
  • ADF sends instructions via Service Bus; SHIR executes queries locally and sends results back
  • Data flows: On-Premises DB → SHIR agent → Azure Service Bus → ADF → ADLS
  • For HA: install SHIR on 2+ machines and configure as a cluster

Alternatives: Azure Managed VNet IR (Microsoft-managed, no agent install needed, connects to private endpoints in your VNet).

9. What are the different ADF trigger types? Medium ⌄
  • Schedule Trigger: Run at specific times using cron expressions (e.g., "0 2 * * *" = 2 AM daily). Simple and most common. Does not backfill missed runs.
  • Tumbling Window Trigger: Fixed-size, non-overlapping time windows starting from a defined start time. Supports backfill (re-processes historical windows). Use for: hourly/daily batches where you need to process one specific time window per run.
  • Storage Event Trigger: Fires when a blob is created or deleted in ADLS/Blob Storage. Event-driven ingestion - process files as they land. Subscribe to specific container/folder patterns.
  • Custom Event Trigger: Fires based on events published to Azure Event Grid. Use for integration with external systems.
  • Manual / API Trigger: Triggered programmatically via REST API or ADF UI. Use for on-demand runs and testing.
10. How do you monitor pipeline failures in ADF? Easy ⌄
  • ADF Monitor tab: Built-in UI showing all pipeline runs, activity runs, trigger runs with status, duration, and error messages. Filter by status (Failed), date range, pipeline name.
  • Azure Monitor Alerts: Create metric alerts on "Failed pipeline runs" - triggers email/SMS/webhook/Teams notification when count > 0.
  • Diagnostic Logs to Log Analytics: Enable diagnostic settings to send ADF logs to Azure Log Analytics. Then query with KQL: ADFPipelineRun | where Status == 'Failed'
  • Web Activity for custom notifications: Add a Web activity in the pipeline's Failure path to call a Teams webhook, PagerDuty, or Slack API with failure details.
  • Azure Data Factory failure email: Configure built-in failure notifications in ADF > Manage > Alerts & Metrics.
11. What RBAC roles would you assign to a data engineer vs a data analyst? Medium ⌄

Data Engineer:

  • Storage Blob Data Contributor on ADLS Gen2 (read + write + delete blobs)
  • Databricks Workspace Contributor (create clusters, run jobs, manage notebooks)
  • Data Factory Contributor (create and manage ADF pipelines)
  • Key Vault Secrets User (read secrets for connection strings)
  • Unity Catalog: GRANT CREATE, SELECT, MODIFY ON SCHEMA silver, bronze, gold

Data Analyst:

  • Storage Blob Data Reader on specific containers (read-only)
  • Databricks workspace access (but no cluster creation - use SQL Warehouse)
  • Unity Catalog: GRANT SELECT ON SCHEMA gold (read-only access to gold layer)
  • No ADF access needed (they don't build pipelines)

Principle of least privilege: Always grant the minimum permissions needed. Analysts should never have write access to production storage.

12. What are the advantages of Delta Live Tables over regular Databricks notebooks? Hard ⌄
  • Automatic dependency management: In notebooks, you manually order cells/notebooks. In DLT, you reference tables by name and DLT builds the DAG automatically.
  • Built-in data quality: @dlt.expect annotations enforce quality rules - no need to manually write quality check code.
  • Automatic incremental processing: DLT tracks what has been processed. On re-run, only new data is processed. With notebooks, you must manage this yourself (watermarks, checkpoints).
  • Auto-retry and error recovery: DLT automatically retries failed tasks. Notebooks fail and require manual re-run.
  • Monitoring UI: DLT provides a visual pipeline graph with row counts, quality metrics, and data quality trends per run. Notebooks have no built-in pipeline monitoring.
  • Automatic lineage: DLT registers all tables in Unity Catalog with full lineage. Notebooks must manually manage Unity Catalog registration.
  • Pipeline modes: Development mode (doesn't write to production) vs Production mode (full optimization).
13. How does Unity Catalog data lineage work technically? Hard ⌄

Unity Catalog lineage is captured at the query execution layer:

  • Capture mechanism: When a Spark/SQL query runs, the Databricks runtime intercepts the query plan and records which input tables were read and which output tables were written.
  • Lineage store: Stored in the Unity Catalog metastore (managed by Databricks, backed by cloud object storage).
  • What is captured: Table-level lineage (table A → table B) and column-level lineage (which source columns contributed to each output column).
  • Sources: DLT pipelines (rich automatic lineage), Databricks notebooks (captured automatically for Unity Catalog tables), ADF via Databricks activities (captured at Databricks side).
  • Access: Via Unity Catalog UI (Data tab → table → Lineage tab), REST API, or system tables: SELECT * FROM system.access.table_lineage
  • Limitations: Lineage is only captured for Unity Catalog tables (not legacy Hive metastore tables). External queries (e.g., direct JDBC reads) are not captured.
14. What is the difference between All-purpose and Job clusters in Databricks? Medium ⌄

All-purpose Cluster:

  • Started manually or via auto-start when attaching a notebook
  • Persists between runs - stays alive until manually stopped or auto-termination kicks in
  • Multiple users and notebooks can attach simultaneously
  • Billed at "all-purpose compute" DBU rate (more expensive)
  • Best for: interactive development, exploratory analysis, debugging

Job Cluster:

  • Created automatically when a job run starts, destroyed when the run completes
  • Dedicated to exactly one job run - no sharing
  • Billed at "jobs compute" DBU rate (cheaper, typically 30-50% less than all-purpose)
  • Completely isolated - no interference from other users/notebooks
  • Best for: production ETL jobs, scheduled pipelines, CI/CD

Rule of thumb: Develop on all-purpose, ship to production on job clusters. Never run production workloads on all-purpose clusters.

15. How would you implement Change Data Capture (CDC) using ADF? Hard ⌄

CDC captures INSERT, UPDATE, DELETE events from a source database. Three approaches in ADF:

1. SQL Server Change Tracking (lightweight CDC):

  • Enable: ALTER DATABASE mydb SET CHANGE_TRACKING = ON
  • ADF Native CDC source: In Copy activity source, select "Change Tracking" mode
  • ADF stores the last sync version and only copies changed rows on each run
  • Limitation: Only tracks which rows changed, not the old values

2. SQL Server CDC (full CDC with before/after values):

  • Enable CDC on the database and tables in SQL Server
  • ADF reads from CDC system tables: cdc.fn_cdc_get_all_changes_dbo_employees
  • Each row has an __$operation column: 1=delete, 2=insert, 3=before update, 4=after update
  • Apply changes to Delta Lake using MERGE statement in Databricks

3. Delta Lake APPLY CHANGES (with DLT):

  • ADF ingests CDC events to Bronze Delta table
  • DLT uses APPLY CHANGES INTO to automatically handle inserts/updates/deletes using sequence keys
  • Most elegant solution - DLT manages the merge logic automatically
