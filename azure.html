Delta Lake Fundamentals

ACID transactions, time travel and schema evolution on cloud storage

⏱ 45 min 🔎 Intermediate

💡What is Delta Lake?

Delta Lake is an open-source storage layer that brings ACID transactions to Apache Spark and big data workloads. It sits on top of your existing data lake (like ADLS Gen2 or S3) and adds reliability features that traditional data lakes lack.

The core component is the _delta_log folder -- a transaction log stored as JSON files that records every operation performed on the table. This log is the foundation of all Delta Lake features: ACID compliance, time travel, schema enforcement, and audit history.

❓Why Delta Lake Exists

Without Delta Lake, data lakes suffered from serious problems:

  • No ACID guarantees -- partial writes could leave corrupted data in the lake
  • No schema enforcement -- anyone could write any schema, breaking downstream consumers
  • No time travel -- no audit trail, no way to recover from bad data loads
  • Difficult upserts -- implementing CDC required rewriting entire partitions
📚 Real World Analogy

Think of Delta Lake like Git for your data. Just as Git tracks every commit and lets you checkout any historical version of your code, Delta Lake tracks every write operation and lets you query any historical version of your data using TIME TRAVEL. The _delta_log is the .git folder for your table.

🎓Interview Questions
What is the _delta_log and what does it contain?
The _delta_log is a directory of JSON files (000000000000000000000.json, 000000000000000000001.json, etc.) stored alongside your data files. Each JSON file represents one transaction (commit) and contains: the operation type (WRITE, UPDATE, DELETE, MERGE), statistics about affected files (number of records, min/max values), schema information, and configuration. Every 10 transactions, a checkpoint Parquet file is created for faster reading of table history.
Explain time travel in Delta Lake. When would you use it?
Time travel allows you to query historical versions of a Delta table using VERSION AS OF or TIMESTAMP AS OF. Use cases: debugging (what did the data look like before a bad load?), reproducibility (ML experiments need the same training data), auditing (what did customer X see on date Y?), and recovery (restore a table to a previous state with RESTORE TABLE).
What is the difference between mergeSchema and overwriteSchema?
mergeSchema allows you to add new columns to an existing schema without error -- it merges the incoming schema with the table schema. overwriteSchema completely replaces the existing schema, which is destructive. Use mergeSchema for additive changes (new columns). Never use overwriteSchema on a production table without careful planning as it breaks downstream consumers expecting the old schema.

Delta Merge Operations

Upsert, CDC and SCD patterns with MERGE INTO

⏱ 50 min 🔎 Intermediate
MERGE INTOUpsertCDCSCD Type 2PySpark
💡What is MERGE INTO?

MERGE INTO (also called UPSERT) is one of the most powerful Delta Lake operations. It allows you to atomically update, insert, and delete records based on a matching condition -- all in a single operation. This is essential for Change Data Capture (CDC) patterns where you need to apply database change events to your data lake.

The operation reads both source and target, computes matches using the ON condition, then applies WHEN MATCHED, WHEN NOT MATCHED, and WHEN NOT MATCHED BY SOURCE clauses atomically. The entire operation is recorded as a single transaction in the _delta_log.

❓Why MERGE Exists

Before Delta Lake MERGE, implementing upserts in Spark required: reading the entire target table, performing a left join, using CASE WHEN logic to determine inserts vs updates, then rewriting everything. This was non-atomic, slow, and error-prone. MERGE INTO makes this atomic, efficient (using statistics to skip files that cannot match), and readable.

📚 Real World Analogy

MERGE INTO is like a postal sorting facility. When mail arrives (source), workers check if the recipient already has a mailbox (WHEN MATCHED). If yes, they update it (UPDATE). If no mailbox exists, they create a new one (WHEN NOT MATCHED INSERT). If someone no longer receives mail (WHEN NOT MATCHED BY SOURCE), their mailbox is closed (DELETE).

📄MERGE INTO Syntax
sql
MERGE INTO target_table AS t
USING source_table AS s
ON t.id = s.id
WHEN MATCHED AND s.operation = 'UPDATE' THEN
  UPDATE SET t.name = s.name, t.salary = s.salary, t.updated_at = s.updated_at
WHEN MATCHED AND s.operation = 'DELETE' THEN
  DELETE
WHEN NOT MATCHED AND s.operation != 'DELETE' THEN
  INSERT (id, name, salary, updated_at)
  VALUES (s.id, s.name, s.salary, s.updated_at)
📄 Code Example 1 -- Basic MERGE (SCD Type 1)
python
from delta.tables import DeltaTable
from pyspark.sql.functions import current_timestamp

# Load the target Delta table
target = DeltaTable.forPath(spark, "/mnt/datalake/customers")

# Source: incoming CDC changes
source = spark.createDataFrame([
    (101, "Alice Smith", "alice@new.com", "2024-01-20"),  # Update
    (102, "Bob Jones", "bob@example.com", "2024-01-20"),   # Insert
], ["customer_id", "name", "email", "updated_date"])

# Execute MERGE
target.alias("t").merge(
    source.alias("s"),
    "t.customer_id = s.customer_id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()

print("Merge complete. Table updated.")

📄 Code Example 3 -- SCD Type 2
python
from pyspark.sql.functions import lit, current_date, col

# SCD Type 2: keep history, close old record, open new record
scd_source = spark.createDataFrame([
    (101, "Alice Smith", "Engineering", "New York")
], ["id", "name", "department", "location"])

target = DeltaTable.forPath(spark, "/mnt/datalake/dim_employees")

# Step 1: Expire current active records that changed
target.alias("t").merge(
    scd_source.alias("s"),
    "t.id = s.id AND t.is_current = true AND (t.department != s.department OR t.location != s.location)"
).whenMatchedUpdate(set={
    "is_current": lit(False),
    "end_date": current_date()
}).execute()

# Step 2: Insert new records for changed rows + net new
from pyspark.sql import functions as F
new_rows = scd_source.withColumn("is_current", lit(True)) \
                     .withColumn("start_date", current_date()) \
                     .withColumn("end_date", lit(None).cast("date"))

new_rows.write.format("delta").mode("append") \
    .save("/mnt/datalake/dim_employees")
⚠️Common Mistakes
1

Missing cardinality check on source -- If source has duplicate keys, MERGE fails with "DELTA_MERGE_HINT_AMBIGUOUS_MATCH". Always deduplicate source with df.dropDuplicates(["key"]) before merging.

2

Forgetting WHEN NOT MATCHED BY SOURCE -- Without it, rows deleted in source persist in target indefinitely. Use this clause to handle soft-deletes or hard-deletes.

3

Using MERGE for high-volume streaming -- Each micro-batch should deduplicate within itself before merging. Accumulate changes first, then merge in foreachBatch.

4

Using .whenMatchedUpdateAll() carelessly -- updateAll() maps ALL matching column names automatically, which can accidentally overwrite system columns like _ingested_at.

🎓Interview Questions
What happens if MERGE source has duplicate keys?
Delta Lake raises an AnalysisException: "DELTA_MERGE_HINT_AMBIGUOUS_MATCH". The merge operation requires each source row to match at most one target row. To fix: deduplicate source before merge using df.dropDuplicates(["key_column"]) or use a subquery with ROW_NUMBER() to pick the latest record per key.
How do you implement SCD Type 2 in Delta Lake?
Two-step approach: First, MERGE to expire current records (set is_current=false, end_date=today) for rows whose tracked attributes changed. Second, INSERT new rows with is_current=true, start_date=today, end_date=null for all changed and new records. Delta Lake MERGE supports WHEN NOT MATCHED BY SOURCE for source-driven deletes. Alternatively, merge with condition on all slowly changing attributes in a single pass.
Explain WHEN NOT MATCHED BY SOURCE.
Added in Delta 2.0, WHEN NOT MATCHED BY SOURCE handles rows in the target that have no matching row in the source. This is useful for soft-deletes: you can set deleted_flag=true or hard-delete records that have been removed from the source system. Without this clause, you must do a separate DELETE after the MERGE, which is two operations and not atomic.

Delta Optimization

OPTIMIZE, VACUUM, Z-ORDER and file compaction for performance

⏱ 40 min 🔎 Intermediate

💡The Small Files Problem

Delta Lake tables accumulate many small files over time, especially with streaming workloads or frequent small appends. This is called the "small files problem" and it severely impacts query performance. Delta provides three main tools: OPTIMIZE for file compaction, Z-ORDER for data clustering, and VACUUM for cleanup of old files no longer needed for time travel.

Parquet query performance is directly related to file sizes and data layout. Reading 10,000 small files of 1MB each is far slower than reading 100 files of 100MB each -- each file requires filesystem metadata lookups, open/close operations, and Parquet footer reads.

📚 Real World Analogy

Think of your Delta table like a physical library. OPTIMIZE is like a librarian reorganizing scattered books into full shelves (file compaction). Z-ORDER is like arranging books by author AND genre together so finding "Stephen King thrillers" skips all non-thriller shelves. VACUUM is like discarding old editions that nobody reads anymore to free shelf space.

⚠️Common Mistakes
1

VACUUM with 0 hours retention -- Immediately destroys all time travel. Always keep at minimum 168 hours (7 days) in production.

2

Z-ORDERing on high-cardinality columns -- Z-ORDER works best on low-to-medium cardinality columns (date, country, product_category). Avoid UUID or email.

3

Running OPTIMIZE too frequently -- It rewrites files and creates new versions in the transaction log. Run on a schedule (nightly/weekly), not after every write.

4

Not knowing about Liquid Clustering -- Delta 3.0+ introduces liquid clustering which replaces Z-ORDER and supports online re-clustering without full rewrites.

🎓Interview Questions
What is the small files problem and how does OPTIMIZE solve it?
Spark writes many small files (one per task/partition) which causes high read overhead -- each file requires filesystem metadata lookups, open/close operations, and Parquet footer reads. OPTIMIZE compacts these small files into target-size files (default 128MB) by reading all data and rewriting in fewer, larger files. The old files are retained for time travel but marked for removal. Performance improvement can be 10-100x for query-heavy workloads.
How does Z-ORDER improve query performance?
Z-ORDER physically co-locates related data using a space-filling curve algorithm. When you Z-ORDER by (date, user_id), records with the same date AND user_id end up in the same Parquet files. The query engine reads file-level statistics (min/max per column per file). For a query filtering WHERE date='2024-01' AND user_id=123, the engine can skip entire files whose min/max range does not overlap -- this is called data skipping. Z-ORDER works best when your query patterns frequently filter on the same columns.
When should you use partitioning vs Z-ORDER?
Partitioning creates physical directory separation (date=2024-01-15/). Use it for high-cardinality time columns where you always filter by exact partition. Z-ORDER works within partitions using data skipping statistics. Use Z-ORDER for medium-cardinality columns (user_id, product_id, country) within partitions. Never partition by high-cardinality columns (UUID, email) -- you would get millions of tiny partition directories. Rule of thumb: partition by date (1 partition per day), Z-ORDER by other filter columns.

💡The Azure Hierarchy

Azure organizes everything in a strict hierarchy: Management Groups contain Subscriptions, which contain Resource Groups, which contain individual Resources. Understanding this hierarchy is essential for designing proper governance, access control, and cost management in enterprise data platforms.

This hierarchy exists for enterprise governance: Management Groups enforce policies across multiple subscriptions, Subscriptions provide billing boundaries and access isolation, Resource Groups provide lifecycle management (delete an RG to delete everything in it), Resources are the actual services (storage accounts, VMs, databases).

📚 Real World Analogy

Think of Azure like a university campus. The Management Group is the university itself. Subscriptions are different colleges (Business School, Engineering School). Resource Groups are departments within each college. Resources are individual classrooms, labs, and offices. A professor (service principal) can have different roles in different departments.

⚠️Common Mistakes
1

Putting all resources in one Resource Group -- Makes access control and lifecycle management difficult. Separate by workload and environment (rg-dataplatform-prod, rg-dataplatform-dev).

2

Choosing the wrong region -- Pick the region closest to your data consumers AND where the Azure services you need are available. Not all services exist in all regions.

3

Confusing RG region with resource region -- The RG region only stores metadata. Your resources can be in different regions than the Resource Group itself.

4

Not using tags from day one -- Without tags, cost allocation and governance becomes impossible. Define a tagging strategy (Environment, Team, CostCenter) before deploying anything.

🎓Interview Questions
What is the difference between a Subscription and a Resource Group?
A Subscription is a billing unit and trust boundary -- it is linked to an Azure AD tenant and has a spending limit. All resources in a subscription are billed together. A Resource Group is a logical container within a subscription for organizing related resources. Resource Groups share a lifecycle (deleting an RG deletes all its resources), enable role assignments at the group level, and help organize by workload/environment. You can have multiple RGs per subscription but a resource belongs to exactly one RG.
What are Azure Availability Zones?
Availability Zones are physically separate datacenters within an Azure region, each with independent power, cooling, and networking. Deploying resources across zones protects against datacenter-level failures. Zone-redundant services automatically replicate across 3 zones. Use AZs for production workloads requiring 99.99% SLA. Note: not all Azure regions have Availability Zones yet.
How does Azure RBAC work?
Azure Role-Based Access Control (RBAC) controls who can do what to which resources. A role assignment has three parts: Security Principal (user/group/service principal/managed identity), Role Definition (collection of permissions like "Storage Blob Data Contributor"), and Scope (subscription/resource group/resource). Assignments are inherited down the hierarchy -- a role at subscription scope applies to all RGs and resources within. Use least-privilege principle: assign the narrowest scope and minimal permissions needed.

💡What is an Azure Storage Account?

An Azure Storage Account is a unique namespace in Azure for your storage objects. It supports multiple storage types: Blob Storage (unstructured data/files), Azure Files (SMB/NFS shares), Queue Storage (messages), and Table Storage (NoSQL). For data engineering, Blob Storage is the primary concern.

The Storage Account is the foundation of every Azure data platform. ADF reads from it, Databricks mounts it, Power BI connects to it. Understanding storage account configuration (access tiers, redundancy, network security) is fundamental.

📚 Real World Analogy

A Storage Account is like a post office branch. Containers are the mailboxes (PO boxes). Blobs are the individual pieces of mail. Access tiers are like different classes of mail storage: Hot is on the front desk (quick access, expensive), Cool is in the back room (slower, cheaper), Archive is in the warehouse offsite (cheapest but takes hours to retrieve).

🎓Interview Questions
What are the blob access tiers and when do you use each?
Hot tier is for frequently accessed data (default, highest storage cost, lowest access cost). Cool tier is for infrequently accessed data stored at least 30 days (lower storage cost, higher access cost, 30-day early deletion penalty). Archive tier is for rarely accessed data stored at least 180 days (lowest storage cost, highest access cost, rehydration takes up to 15 hours). Use Hot for recent data (last 30 days), Cool for historical data you occasionally query, Archive for compliance and backup retention.
What is a SAS token and when should you use it?
A Shared Access Signature (SAS) token is a URI that grants restricted access to Azure Storage resources for a specified time period. It encodes permissions (read/write/delete/list), resource scope (service/container/blob), validity period, and IP restrictions. Use SAS when you need to give external systems or users temporary, scoped access without sharing account keys. Types: Account SAS (any service), Service SAS (specific service), User Delegation SAS (uses Azure AD -- preferred, more secure).
What is the difference between LRS, ZRS, GRS and GZRS?
LRS (Locally Redundant Storage) keeps 3 copies in one datacenter -- cheapest, no zone or geo protection. ZRS (Zone Redundant Storage) keeps copies across 3 Availability Zones in one region -- protects against datacenter failure. GRS (Geo-Redundant Storage) keeps 6 copies: 3 in primary region (LRS), 3 in a paired secondary region -- protects against regional disaster. GZRS combines ZRS in primary and LRS in secondary -- highest availability. Use GRS/GZRS for production data platforms where data loss is unacceptable.

💡What is ADLS Gen2?

Azure Data Lake Storage Gen2 (ADLS Gen2) is built on top of Azure Blob Storage with the Hierarchical Namespace (HNS) feature enabled. This transforms the flat blob structure into a true directory tree, enabling efficient rename operations (O(1) instead of O(n)), fine-grained POSIX ACLs, and Hadoop-compatible access for Spark and Databricks.

The key insight: without HNS, renaming a "folder" in blob storage requires copying every file and deleting originals -- extremely slow for large datasets. With HNS, rename is atomic and instant. This matters because Spark renames _temporary staging directories on job completion.

📚 Real World Analogy

Regular Azure Blob Storage is like a filing cabinet with one big drawer containing folders simulated by naming files "folder/subfolder/file.txt". ADLS Gen2 with HNS is like a real computer filesystem -- actual directories, atomic renames, permissions per folder, just like Linux. This is what Hadoop was built for, and why Spark performs significantly better with HNS enabled.

1

Not enabling Hierarchical Namespace at creation -- You cannot enable HNS after storage account creation. It must be set at creation time. Migrating requires a full data copy.

🎓Interview Questions
What is Hierarchical Namespace and why does it matter for Spark?
HNS transforms Azure Blob Storage into a true hierarchical filesystem. Without it, "folders" are just filename prefixes and renaming requires copying all files and deleting originals (O(n) operation). With HNS, rename is an O(1) atomic operation. For Spark, this means: faster job staging (Spark renames _temporary folders on job completion), support for POSIX ACLs (security at directory level), and Hadoop FileSystem API compatibility enabling all Hadoop/Spark ecosystem tools.
What is the difference between RBAC and ACLs in ADLS Gen2?
RBAC (Role-Based Access Control) is Azure-level and operates at the storage account or container scope. Roles like "Storage Blob Data Reader" grant read access to ALL blobs in a container. ACLs (Access Control Lists) are POSIX-style and operate at the individual file and directory level. ACLs support owner/group/other and named user/group permissions. ACLs are evaluated AFTER RBAC: if RBAC grants access, ACLs are bypassed. If RBAC denies, ACLs can still grant access. Use RBAC for broad access, ACLs for fine-grained directory/file security.
What is the ABFSS URL format for ADLS Gen2?
abfss://[filesystem]@[account-name].dfs.core.windows.net/[path]. Example: abfss://raw@mystorageaccount.dfs.core.windows.net/bronze/sales/2024/01/. The "s" in abfss means secure (TLS). "dfs" indicates DFS (Data Lake) endpoint as opposed to "blob" for regular Blob Storage. Unity Catalog external locations use this format. Always use the DFS endpoint (not blob endpoint) for Spark with ADLS Gen2 for full HNS and ACL support.

💡Azure Identity for Data Engineering

🎓Interview Questions
What is the difference between a Service Principal and a Managed Identity?
A Service Principal is an application identity in Entra ID with explicit credentials (client ID + client secret or certificate). You manage the secret rotation, which is a security and operational overhead. A Managed Identity is also an Entra ID identity but Azure automatically manages the credentials -- no secret to rotate, store, or leak. Managed Identities come in two types: System-assigned (tied to an Azure resource lifecycle) and User-assigned (standalone, can be shared across resources). Always prefer Managed Identity over Service Principal when the workload runs on Azure infrastructure.
How does Databricks authenticate to ADLS Gen2 using Managed Identity?
Two approaches. Unity Catalog Storage Credential: create a User-assigned Managed Identity, assign it "Storage Blob Data Contributor" on the ADLS storage account, then reference it in a Unity Catalog storage credential. Databricks handles token acquisition transparently. Cluster-level: assign the Databricks workspace a System-Assigned Managed Identity, then grant RBAC on ADLS. The cluster automatically uses this identity for all ADLS access without any configuration in notebooks.
What is OAuth 2.0 client credentials flow?
This is the flow used by service principals for machine-to-machine authentication (no user involved). Steps: (1) Application sends client_id + client_secret to Entra ID token endpoint. (2) Entra ID validates credentials and issues a JWT access token with an expiry (typically 1 hour). (3) Application attaches token as Bearer header to API requests. (4) Azure resource validates the token's signature, expiry, and claims. (5) Access granted based on the role assignments for that service principal. Tokens are cached and auto-refreshed by Azure SDKs.

🎓Interview Questions
What is the difference between a Linked Service, Dataset, and Activity in ADF?
A Linked Service defines the CONNECTION to a data store (the "where to connect" -- server name, authentication). Think of it as a database driver/connection string. A Dataset defines the DATA STRUCTURE within that connection (the "what to access" -- table name, file path, schema, format). An Activity is the OPERATION performed (Copy: move data, Execute Notebook: run Spark code, Web: call REST API). You can reuse a linked service in multiple datasets, and reuse datasets in multiple activities.
What is Integration Runtime in ADF?
Integration Runtime (IR) is the compute infrastructure ADF uses to execute activities. Azure IR: auto-scaling, managed by Microsoft, for cloud-to-cloud connections. Self-Hosted IR: installed on your on-premises server or VM, enables ADF to access private network resources (on-prem SQL Server, Oracle, SAP). Azure-SSIS IR: lifts and shifts SSIS packages to run in Azure. For data engineering, you typically use Azure IR for cloud workloads and SHIR for on-prem source systems.
How do you implement incremental loads in ADF?
Three common patterns: (1) Watermark pattern -- store last load timestamp in a control table, query source WHERE modified_date > watermark, update watermark after successful load. (2) Tumbling Window trigger -- ADF maintains window state, each run processes one time slice. (3) Change Data Capture (CDC) -- use SQL Server CDC or Change Tracking, ADF reads the change log. Use Azure SQL Database or Cosmos DB to store watermarks. Always implement idempotency so re-runs do not duplicate data.

What is the difference between all-purpose and job clusters?
All-purpose clusters are persistent, multi-user clusters designed for interactive development in notebooks. They stay running until manually terminated and are expensive when idle. Job clusters are ephemeral -- created for a specific job run and automatically terminated when the job completes. How do you optimize Databricks cluster performance?

(5) Partition tuning: repartition DataFrames to match cluster parallelism (2-4x number of cores). (6) Cluster policies: enforce memory-optimized instances for ML, compute-optimized for Spark SQL.

🎓Interview Questions
What is the 3-level namespace in Unity Catalog?
Unity Catalog uses catalog.schema.table format. Catalog is the top-level container (typically mapped to an environment like prod, dev, or a business domain). Schema (database) groups related tables and views. Table is the actual data object. Example: catalog_prod.gold.sales_daily. Benefits: cross-workspace sharing (multiple workspaces share one Unity Catalog metastore), consistent access control at each level, clear separation of domains.
How does Unity Catalog lineage work?
Unity Catalog automatically captures lineage for all queries executed in Databricks that read/write Unity Catalog tables. No code changes needed. It tracks: table-to-table lineage (which tables were read to create this table), column-to-column lineage (which source column feeds this column), and job/notebook provenance. Lineage is visible in the Catalog Explorer UI. You can also query it via REST API or system tables (system.access.table_lineage). Lineage enables impact analysis: "if I change this column, what downstream tables break?"
What is the difference between column masking and row filtering?
Column masking (Dynamic Data Masking) applies a masking function to a specific column -- non-privileged users see masked values (e.g., XXX-XX-1234 instead of full SSN) while privileged users see real values. The masking is applied transparently at query time with no table rewrites. Row filtering applies a boolean filter function that restricts which rows users can see -- a regional analyst only sees rows WHERE region = 'Europe'. Both use SQL functions that check group membership (IS_ACCOUNT_GROUP_MEMBER) for conditional logic. Both are enforced whether users query via SQL, PySpark, or BI tools.

💡What is Delta Live Tables?

Delta Live Tables (DLT) is Databricks' declarative framework for building reliable, maintainable ETL pipelines. Instead of writing imperative Spark code with explicit DataFrame operations, you declare WHAT your tables should contain using the @dlt.table decorator. 🎓Interview Questions
What are the three DLT expectation decorators and when do you use each?
@dlt.expect("name", "condition"): Records failing the condition are allowed into the table but failures are counted in metrics. Use for monitoring/alerting without dropping data. @dlt.expect_or_drop("name", "condition"): Records failing the condition are silently excluded from the table. Use when bad records should be filtered but should not fail the pipeline (e.g., null optional fields). @dlt.expect_or_fail("name", "condition"): Any record failing the condition causes the entire pipeline run to fail. Use for critical data quality gates where downstream consumers cannot tolerate bad data (e.g., missing primary key).
How does DLT handle the Medallion architecture differently than regular Spark?
In regular Spark, you write Bronze/Silver/Gold as separate jobs with explicit dependencies managed by ADF or Airflow. In DLT, you define all three layers in one or more Python files in a pipeline. DLT automatically builds the dependency DAG, determines execution order, handles incremental processing (only new Bronze records flow through Silver to Gold), manages checkpoints for streaming tables, and provides end-to-end lineage visible in Unity Catalog -- all without explicit orchestration code.
What is Auto Loader and how does it work with DLT?
Auto Loader (cloudFiles) is Databricks' file discovery mechanism for streaming from cloud storage. It uses Azure Event Grid and Queue Storage to detect new files as they arrive -- no polling required. As files land in ADLS, Event Grid fires an event, Auto Loader processes it, and new data flows through the DLT pipeline within seconds. It also handles schema evolution (cloudFiles.inferColumnTypes), schema rescue (unknown columns saved to _rescued_data column), and exactly-once processing via checkpointing. Much better than spark.readStream.format("parquet").load() which polls the directory.

💡ETL vs ELT Fundamentals

ETL (Extract, Transform, Load) is the foundational pattern for moving data from source systems to analytics platforms. Modern data engineering largely uses ELT (Extract, Load, Transform) -- load raw data first, transform in-place on the data lake using Spark. Understanding extraction patterns (batch vs streaming, push vs pull, CDC) and load strategies (full load vs incremental) is essential for building reliable pipelines.

Every data platform is only as good as its ingestion layer. Wrong extraction patterns cause: duplicate data (missing idempotency), late data (no watermarking), data drift (no CDC), and slow pipelines (full loads when incremental is possible).

Transform is the filtration and purification stages (cleansing, validation, enrichment). Load is filling the reservoirs (data warehouse/data lake). CDC is like a flow meter that only measures NEW water since the last reading, instead of re-measuring the entire river every time.

🎓Interview Questions
What is CDC and how does it differ from batch incremental loads?
Change Data Capture (CDC) captures row-level changes (INSERT/UPDATE/DELETE) from a source database's transaction log in real-time, without querying the table. Methods: SQL Server CDC (reads transaction log), Debezium (Kafka connector reading database logs), Azure SQL Change Tracking. Batch incremental loads poll the source table for rows WHERE modified_date > watermark -- requires a reliable updated_at column, misses DELETEs, and adds load on the source. CDC is event-driven, captures deletes, has lower source impact, and enables near-real-time pipelines. Prefer CDC for operational source systems; use watermark batch for simpler scenarios.
What is idempotency in data pipelines and how do you achieve it?
Idempotency means running a pipeline multiple times produces the same result as running it once -- re-runs do not create duplicates. Achieving it: (1) Use MERGE/UPSERT instead of INSERT -- if a record already exists, update rather than insert. (2) Use Delta Lake replaceWhere for partition-level overwrites. (3) Store pipeline run IDs and skip already-processed batches. (4) For streaming, use checkpointing (exactly-once semantics). (5) For ADF Copy activity, use staging + final swap pattern. Critical for production pipelines where retries are common due to transient failures.
What is ELT vs ETL and when do you use each?
ETL (Extract, Transform, Load): transform data BEFORE loading into the target. Traditional data warehouse approach -- data arrives clean and structured. Limited by the transformation server's compute. ELT (Extract, Load, Transform): load raw data first, then transform using the target system's compute (Spark/Databricks/Synapse). Modern cloud data lake approach -- preserves raw data for reprocessing, uses massively parallel compute for transformation. Use ETL when target system has limited compute and storage. Use ELT for modern cloud data lakes where storage is cheap and compute scales horizontally -- the standard approach for Lakehouse architectures.

💡What is the Medallion Architecture?

The Medallion Architecture (also called Multi-hop Architecture) organizes data into three progressively refined layers: Bronze (raw, as-is from source), Silver (cleansed and conformed), and Gold (aggregated, business-ready). Each layer adds data quality and business value while the previous layer is preserved for reprocessing.

Without layering: no raw data preservation (if transformation has a bug, raw data is gone), mixing raw and processed data (unclear which to trust), monolithic pipelines (change in one step breaks everything). Medallion solves this with clear contracts between layers.

📚 Real World Analogy

Medallion is like an oil refinery. Bronze is crude oil straight from the well -- raw, unrefined, high volume, not directly usable. Silver is refined petroleum -- processed, impurities removed, standardized to specifications. Gold is the final products (gasoline, jet fuel, plastics) -- purpose-built for specific uses. You never blend crude oil directly into a car (mix raw data into dashboards).

🎓Interview Questions
Explain the Medallion Architecture and the purpose of each layer.
Bronze (raw layer): append-only ingestion of raw source data with zero transformation. Preserves exact source format for full reprocessing. Adds metadata (_ingested_at, _source_file). Silver (cleansed layer): applies data quality rules (deduplication, null handling, type casting, standardization). Enforces business rules and conforms to enterprise data standards. Reusable by multiple Gold tables. Gold (serving layer): aggregated, denormalized tables optimized for specific BI/analytics use cases. One Gold table per business domain/report (sales_daily, customer_360, inventory_snapshot). Consumed directly by Power BI, dashboards, or ML features.
How do you handle late-arriving data in Medallion?
Bronze: use MERGE with source key so late-arriving records upsert correctly. Silver: use Delta MERGE to upsert, re-derive all computed columns. Gold: use Delta replaceWhere to recompute affected date partitions. For streaming: use Spark watermarking with a late data tolerance (e.g., 24 hours for daily batches) -- records arriving within the window are included, beyond the window are sent to a late_arrivals table. Track SLA compliance in a monitoring table.
What are the advantages of Medallion over a traditional data warehouse?
Traditional DWH (Kimball/Inmon): schema-on-write (must define schema upfront), difficult schema changes, poor at semi-structured data, expensive compute+storage coupling, limited historical data. Medallion on Lakehouse: schema-on-read (Bronze accepts any schema), Delta schema evolution, handles JSON/Parquet/CSV natively, separates compute (Spark/Databricks) from storage (ADLS), time travel enables full history at low cost, one platform for both BI and ML, open formats avoid vendor lock-in.

💡The Three Pillars of Observability

Observability means knowing what is happening in your data platform at all times. The three pillars are: Logs (what happened -- pipeline runs, errors, data volumes), Metrics (how it performed -- duration, CPU, data rows/second), and Alerts (notifications when things go wrong). For data engineering, add a fourth: Data Quality (did the data CONTENT change in unexpected ways).

A pipeline that runs silently and produces wrong data is more dangerous than one that fails loudly. Without monitoring: SLA breaches discovered by angry business users, data quality issues propagate for days before detection, no capacity planning data, debugging takes hours asking "what did the job do at 3am?"

📚 Real World Analogy

Monitoring a data platform is like a hospital ICU. The monitoring dashboard is the patient monitors showing vital signs (metrics). Log Analytics is the nurse's detailed log of every medication and vital reading (logs). Azure Monitor alerts are the alarms that sound when vitals go out of range. Data quality checks are the lab tests that detect subtle internal issues the vital signs alone cannot reveal.

⚠️Common Mistakes
1

Only monitoring job success/failure -- A job that succeeds but produces 0 rows is a silent failure. Monitor data volumes, not just job status codes.

2

Not setting up SLA alerts -- Agree on SLA with business (e.g., "Gold sales data by 7am"). Set Azure Monitor alerts if the pipeline has not completed by 6:45am.

3

Logging without structure -- Plain text logs are hard to query. Always use structured/JSON logging for Azure Log Analytics kusto queries.

4

Not monitoring data freshness -- Track the MAX(updated_at) in key Gold tables and alert if it is stale beyond the expected refresh interval.

🎓Interview Questions
What is the difference between monitoring and observability?
Monitoring is watching pre-defined metrics and alerting on known failure modes ("CPU > 80%", "job failed"). It answers: "Is the thing I'm watching broken?" Observability is the ability to understand the internal state of a system from its external outputs (logs, metrics, traces). It answers: "WHY is it broken, even for novel failure modes I did not anticipate?" For data engineering: monitoring catches pipeline failures and metric threshold breaches. Observability lets you trace a data anomaly back to its root cause by correlating logs, lineage, and data quality metrics.
How do you implement SLA monitoring for data pipelines?
(1) Define SLA: "Gold sales_daily table populated with current day's data by 07:00 UTC". (2) Track completion: after each pipeline run, write completion timestamp and row count to a monitoring table. (3) Freshness check: run a scheduled query every 15 minutes after expected completion time checking if MAX(run_date) = CURRENT_DATE AND rows_loaded > 1000. (4) Alert: configure Azure Monitor metric alert or Logic App to send Teams/email notification on SLA_BREACH. (5) Dashboard: Power BI report showing SLA compliance percentage per pipeline per day.
What data quality metrics should you track?
Volume metrics: row counts per pipeline run, compare to historical average (alert on +/- 20% deviation). Completeness: null rates for key columns (customer_id, amount, date). Uniqueness: duplicate key counts for tables expecting unique keys. Validity: percentage of records passing business rules (valid email format, positive amounts). Timeliness: MAX(source_timestamp) vs ingestion_time to detect lagging sources. Consistency: cross-table reconciliation (total_orders count in Bronze = count in Silver = sum in Gold aggregates). Store all metrics in a Delta table, visualize in Power BI, alert on anomalies.
